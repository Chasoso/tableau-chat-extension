# v0.7.0 Plan

## 1. Purpose

This document defines the v0.7.0 plan before any transport or metadata implementation begins.

Its purpose is to:

- lock the v0.7.0 scope
- make the Hosted Tableau MCP migration assumptions explicit
- clarify how the child issues fit together
- fix the safety and governance boundaries before implementation work starts
- treat v0.7.0 as a foundation milestone, not an implementation sprint
- prepare a clean handoff to v0.8.0 Hosted Tableau MCP Metadata Execution

## 2. Background

The project has moved through a sequence of architecture milestones:

- v0.2.0 established the backend and agent foundation
- v0.3.0 introduced the Tableau context-first UI
- v0.4.0 introduced Structured Orchestration
- v0.5.0 introduced the Tool Layer
- v0.6.0 introduced the AgentRunner boundary, `LambdaAgentRunner`, and the deferred AgentCore decision

That sequence left the project with a stable deterministic baseline, but it also left Tableau MCP centered on stdio integration.

v0.7.0 is where the roadmap shifts to the next transport assumption: Hosted / remote Tableau MCP.

## 3. Current state after v0.6.0

After v0.6.0, the current architecture looks like this:

- `selected_mark_explanation` runs through `LambdaAgentRunner`
- deterministic `ResponseComposer` remains the baseline
- the Tool Layer can already execute context-derived pseudo tools
- Tableau MCP is still assumed to be stdio-based
- free-form chat remains on the legacy path
- `ChatService` and `runLightweightAgentLoop` still carry the main legacy flow
- AgentCore is still in deferred partial adoption
- Hosted MCP and remote MCP are not implemented yet

This means the repo has a good comparison baseline, but it has not yet created the transport boundary needed for hosted Tableau MCP readiness.

## 4. v0.7.0 milestone goal

v0.7.0 prepares tableau-chat-extension for a future migration from current stdio Tableau MCP usage to Hosted / remote Tableau MCP by defining readiness assumptions, transport boundaries, read-only metadata tool contracts, fake/no-network execution paths, and governance boundaries.

v0.7.0 is not the Hosted MCP production migration itself.

It is also not the point where arbitrary query execution, free-form chat migration, or AgentCore implementation begins.

The milestone exists to make the hosted transition safe, observable, and testable in later versions.

## 5. Scope

The v0.7.0 scope is intentionally narrow and foundation-oriented:

- current stdio Tableau MCP usage audit
- Hosted Tableau MCP readiness audit
- Tableau MCP transport abstraction design
- stdio vs hosted transport configuration strategy
- Hosted MCP OAuth / user-context / site-settings boundary
- read-only Tableau metadata tool definitions
- metadata input/output schemas
- metadata preconditions and governance boundaries
- ToolRegistry registration with fake/no-network handlers
- transport-aware metadata execution boundary
- metadata output normalization and trace events
- migration path documentation from stdio to Hosted Tableau MCP
- v0.7.0 wrap-up

## 6. Non-goals

v0.7.0 explicitly does not include:

- full Hosted Tableau MCP production migration
- actual Hosted MCP connection as a required behavior
- arbitrary query execution
- Tableau MCP query tools
- underlying data access
- write-capable Tableau MCP tools
- raw MCP tool exposure
- LLM-generated tool execution
- free-form chat full migration
- `ChatService` replacement
- `runLightweightAgentLoop` replacement
- LLM `ResponseComposer` implementation
- Bedrock prompt generation changes
- AgentCore implementation
- AgentCoreRunner implementation
- broad frontend redesign

## 7. Hosted Tableau MCP migration assumptions

v0.7.0 assumes the following:

- Hosted / remote MCP is the future target candidate
- the current implementation is still stdio-based Tableau MCP
- the Tool Layer should not know transport details
- a transport abstraction is required to keep that boundary stable
- local development and CI must continue to work with fake or no-network transport
- stdio fallback should remain available for the near term
- Hosted MCP will make OAuth, user context, site settings, and token boundaries more important
- the relationship between Cognito and Tableau authentication needs to be understood clearly
- Hosted MCP-free test runs must still succeed
- read-only metadata should be the first migration target
- arbitrary query, underlying data, and write operations are out of scope for this milestone

Current:

```text
backend / Lambda
↓
stdio Tableau MCP server
↓
Tableau Cloud
```

Target candidate:

```text
backend / Lambda
↓
remote MCP client / Hosted Tableau MCP transport
↓
Hosted Tableau MCP on Tableau Cloud
↓
Tableau Cloud content
```

## 9. Safety and governance boundaries

v0.7.0 must preserve the following boundaries:

- read-only metadata first
- raw MCP tool exposure is forbidden
- arbitrary query execution is forbidden
- underlying data access is forbidden
- write-capable tools are forbidden
- LLM-generated tool execution is forbidden
- row and column limits must exist where relevant
- output truncation must be explicit
- datasource ambiguity must be handled as a precondition
- auth and permission preconditions must be explicit
- `safeForPreviewOnly` policy should remain part of the contract
- no secrets or tokens may appear in trace output
- no raw stack traces may appear in user-facing results
- output must remain JSON-safe
- local tests must not require Hosted MCP
- `selected_mark_explanation` regression behavior must remain protected
- free-form chat legacy behavior must remain unchanged

## 10. Decision criteria

Several decisions may come up during the v0.7.0 milestone.

### Hosted MCP connection timing

The team should decide whether v0.7.0 includes a real Hosted MCP connection or defers that to v0.8.0.

Decision criteria:

- OAuth / site settings / token boundary is sufficiently clear
- no-network tests already pass
- stdio fallback exists
- CI works without Hosted MCP

### First metadata tool choice

The first metadata tool should be chosen between:

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`

Decision criteria:

- the schema is simple
- the output is small
- datasource ambiguity handling can be exercised
- the tool has visible user value
- fake handlers are easy to test before a real Hosted MCP connection exists

### Code-bearing issue start condition

The backend implementation issues should not start until:

- the docs-only issues are complete
- the transport boundary is clear
- the safety boundary is clear
- the fake / no-network strategy is clear

### v0.8.0 readiness

v0.8.0 can start when:

- metadata tools can be registered and executed with fake / no-network behavior
- a transport-aware boundary exists
- hosted readiness open questions are reduced to a manageable list

## 11. Validation strategy

Validation in v0.7.0 is split by issue type.

### Docs-only issues

For docs-only issues:

- verify the document exists
- verify the required headings are present
- check consistency against the roadmap
- avoid local absolute paths
- ensure the text does not contradict the deferred AgentCore decision

### Code-bearing issues

For code-bearing backend issues:

- backend lint
- backend typecheck
- backend tests
- ToolRegistry lookup tests
- ToolDefinition contract tests
- ToolPrecondition tests
- ToolExecutionWrapper tests
- fake / no-network transport tests
- metadata output normalization tests
- trace event tests
- `selected_mark_explanation` regression tests
- free-form chat legacy regression tests

### Frontend

Frontend validation is not expected unless frontend code changes.

v0.7.0 should avoid broad frontend changes.

### Hosted MCP

Hosted MCP integration tests should remain optional or gated.

CI must not require Hosted MCP.

No secrets should be required in tests.

## 12. Risks and mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Hosted MCP requirements are unclear | Migration work stalls | Keep the milestone docs-first and use fake transport until the requirements are understood |
| OAuth / user-context boundary is complex | Auth bugs or security mistakes | Isolate the auth boundary in its own issue before implementation |
| Tool Layer leaks transport details | Future migration becomes harder | Define the transport abstraction before metadata execution is wired |
| Metadata output grows too large | Unsafe responses and traces | Normalize, truncate, and summarize output before surfacing it |
| Arbitrary query sneaks into the milestone | Governance risk | Keep the scope metadata-only and enforce preconditions |
| Local tests start requiring Hosted MCP | CI instability | Keep fake / no-network handlers available |
| `selected_mark_explanation` breaks | Regression in the baseline path | Keep regression tests around the legacy and deterministic path |
| Free-form chat changes accidentally | User-facing regression | Keep the legacy chat behavior unchanged in v0.7.0 |

## 13. Handoff to v0.8.0

v0.8.0 is expected to be the Hosted Tableau MCP Metadata Execution milestone.

v0.7.0 should hand off the following to v0.8.0:

- stdio usage audit
- Hosted readiness audit
- transport abstraction design
- configuration strategy
- OAuth / user-context / site-settings boundary
- read-only metadata tool definitions
- schemas
- preconditions
- fake / no-network handlers
- transport-aware execution boundary
- output normalization and trace policy
- migration path document

v0.8.0 should then decide:

- whether real Hosted MCP metadata execution should be enabled
- how stdio fallback should be preserved
- whether integration tests should stay gated
- how far metadata tools should be used for free-form discovery

- Should local development use fake, stdio, or hosted transport by default?
- Should the first metadata tool be `describeDatasource` or `listFields`?
- Should v0.7.0 include a real Hosted MCP connection?
