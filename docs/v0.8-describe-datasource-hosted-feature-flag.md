# describeDatasource Hosted Execution Behind Feature Flag

## 1. Purpose

This document locks the first v0.8.0 Hosted execution target:
`tableau.metadata.describeDatasource` behind an explicit feature flag and
configuration check.

It exists to ensure that:

- Hosted execution is only used when explicitly enabled
- fake / no-network fallback remains available
- stdio fallback remains available when configured
- auth context is passed through the safe adapter boundary
- normalized output and trace metadata stay JSON-safe
- `listFields` is not promoted to Hosted execution yet

## 2. Background

v0.7.0 established the metadata foundation:

- transport abstraction and transport configuration strategy
- auth / user-context / site-settings boundaries
- read-only metadata tool definitions
- metadata input / output schemas
- preconditions and governance checks
- fake / no-network handlers
- transport-aware execution boundary
- output normalization and trace events

v0.8.0 uses that foundation to connect the first read-only metadata tool to
Hosted Tableau MCP in a guarded way.

## 3. Feature flag boundary

Hosted execution is controlled by an explicit opt-in flag in execution context
and matching transport configuration.

Recommended shape:

- `tableauMetadataHostedExecutionEnabled: true`
- `tableauMetadataTransportKind: "hosted"`

The boundary must not auto-select Hosted mode. If the flag is absent or false,
the runtime keeps the fake / no-network or stdio fallback path.

## 4. Execution flow

The intended flow is:

```text
tableau.metadata.describeDatasource
↓
handler
↓
precondition evaluator
↓
transport config resolver
↓
Hosted MCP auth context adapter
↓
transport-aware execution boundary
↓
HostedTableauMcpTransport
↓
transport result
↓
metadata output normalization
↓
safe normalized result
```

If Hosted mode is not explicitly enabled, the same handler keeps using the
existing fake / no-network or stdio fallback path.

## 5. Auth context adapter usage

The handler should use the Hosted MCP auth context adapter to build:

- safe auth state
- tokenReference handoff
- user-context summary
- trace-safe auth summary

Raw access tokens, refresh tokens, authorization headers, and stack traces must
never enter the transport request, result, or trace surfaces.

## 6. Transport selection and fallback

Hosted transport is only selected for `tableau.metadata.describeDatasource`.

Fallback rules:

- feature flag off -> fake / no-network path
- no-network requested -> fake / no-network path
- Hosted transport unavailable -> fallback path or safe `not_configured`
- auth missing or expired -> safe auth-required result
- stdio fallback configured -> stdio fallback remains available

`tableau.metadata.listFields` stays on the non-hosted path until a separate
readiness decision is made.

## 7. Hosted transport result mapping

Hosted transport results must be normalized into the existing metadata output
shape.

Requirements:

- raw MCP output must not be exposed directly
- safe errors should map to safe metadata summaries
- timeout / unsupported / not_configured states must be preserved safely
- hosted trace metadata should remain JSON-safe
- host-specific fields should stay behind metadata summaries, not raw payloads

## 8. Normalized output behavior

The user-facing output for `describeDatasource` should remain summary-first and
should not change shape drastically between fake, stdio, and Hosted paths.

The normalizer should continue to preserve:

- datasource summary
- field summary when present
- connection summary when present
- warnings and truncation / omission signals
- safe error summaries

## 9. Trace metadata behavior

Trace metadata should capture Hosted execution without exposing secrets.

Safe trace fields include:

- tool name
- transport kind
- request / correlation / run identifiers
- duration and timeout information
- hosted feature enabled / fallback used flags
- safe auth state summary

Trace data must not include:

- raw tokens
- raw authorization headers
- raw MCP responses
- raw stack traces

## 10. No-network and gated integration behavior

Hosted execution remains gated.

Default CI should stay on the no-network path.
Hosted integration tests should remain optional and explicitly enabled.
If Hosted configuration is missing, the runtime should skip or fall back safely
instead of failing the default test path.

## 11. Safety and governance boundaries

The v0.7.0 safety boundary remains in place:

- read-only metadata first
- no arbitrary query execution
- no underlying data access
- no write-capable tools
- no broad raw MCP tool exposure
- no raw tokens or secrets in output
- no broad frontend redesign
- `selected_mark_explanation` remains unchanged
- free-form chat remains on the legacy path

## 12. Relationship with listFields readiness

`tableau.metadata.listFields` is explicitly not the first Hosted execution
target.

Its output size, truncation, omission, and permission behavior need a separate
readiness review before it can move to Hosted execution.

## 13. Relationship with #162, #163, and #164

This issue is the handoff point for:

- `#162` Hosted MCP metadata error normalization
- `#163` hosted metadata execution trace coverage
- `#164` fake / no-network and stdio fallback regression coverage

Those issues should continue to preserve the same Hosted gating and fallback
rules established here.

## 14. Open questions

- whether the feature flag should remain context-based or move into a higher
  level config object
- whether stdio fallback should remain available indefinitely
- whether `listFields` can be promoted to Hosted execution in v0.8.0
- whether Hosted integration tests need an additional runtime marker beyond the
  feature flag
