# AgentRunner Comparison Metrics

## 1. Purpose

This document defines the comparison metrics used to evaluate `LambdaAgentRunner` against a future `AgentCoreRunner`.

The comparison is meant to answer whether AgentCore is worth implementing for `selected_mark_explanation`, not whether it can produce better answers.

## 2. Comparison target

Current:

- `LambdaAgentRunner`

Candidate:

- `AgentCoreRunner`

Initial comparison path:

- `selected_mark_explanation`

Comparison intent:

- compare runtime behavior, reliability, observability, cost, and debugging characteristics
- keep the comparison deterministic and repeatable
- support `#109` decision record writing

## 3. Metric categories

1. Runtime performance
2. Reliability
3. Trace and observability
4. Debuggability
5. Cost
6. Implementation complexity
7. Local development / testability
8. Output consistency
9. Operational readiness
10. Future extensibility

The metrics below should be treated as measurement definitions, not as code to implement now.

## 4. Runtime performance metrics

### `p50LatencyMs`

- Definition: median end-to-end runtime duration for a single run
- Measurement method: record total duration from runner start to runner completion
- Possible source: runner timing, trace metadata, or future benchmark logs
- LambdaAgentRunner: compute from `startedAt` / `completedAt` in `AgentRunResult.observability`
- AgentCoreRunner: compute from AgentCore timing plus runner timing
- Expected for selected_mark_explanation: low and stable
- Rule of thumb: should stay well below the API response budget

### `p95LatencyMs`

- Definition: 95th percentile end-to-end duration
- Measurement method: aggregate multiple runs of the same scenario
- Possible source: comparison log / benchmark sheet
- LambdaAgentRunner: local or test harness aggregation
- AgentCoreRunner: same aggregation, plus AgentCore timing
- Expected for selected_mark_explanation: close to p50 if the path stays deterministic
- Rule of thumb: large gaps between p50 and p95 suggest unstable runtime behavior

### `maxLatencyMs`

- Definition: slowest observed end-to-end run in the comparison set
- Measurement method: max of scenario runs
- Expected for selected_mark_explanation: should not approach the API limit in normal cases

### `minLatencyMs`

- Definition: fastest observed end-to-end run
- Measurement method: min of scenario runs

### `averageLatencyMs`

- Definition: arithmetic mean of the measured durations
- Measurement method: average across identical scenario runs

### `runnerOverheadMs`

- Definition: time spent in runner boundary code only, excluding tool execution and response composition where possible
- Measurement method: subtract inner orchestration timing from total runner timing when those sub-durations are available
- LambdaAgentRunner: derived from runner wrapper timing
- AgentCoreRunner: compare wrapper overhead plus managed runtime overhead
- Expected for selected_mark_explanation: thin runner overhead should stay small

### `orchestrationDurationMs`

- Definition: time spent in intent resolution, planning, execution engine, tool layer, and composition
- Measurement method: sum of step timings or a single orchestration window
- Expected: mostly stable for selected_mark_explanation

### `toolExecutionDurationMs`

- Definition: time spent executing tools in the selected_mark_explanation path
- Measurement method: sum or window covering tool execution steps
- Expected: small, because the path is context-derived and deterministic

### `responseCompositionDurationMs`

- Definition: time spent turning response material into the final message
- Measurement method: measure the composer step separately if available

### `coldStartSuspectedCount`

- Definition: number of runs where startup behavior looks like a cold start
- Measurement method: compare first-run latency to warmed-up runs and note startup spikes
- AgentCoreRunner: may be more sensitive to managed runtime startup effects

### `durationWithinApiGatewayLimit`

- Definition: whether the run completed within the API Gateway budget used by the current frontend-triggered path
- Measurement method: `durationMs <= API budget`
- Expected for selected_mark_explanation: yes in normal cases

## 5. Timeout metrics

### `timeoutRate`

- Definition: timed-out runs divided by total runs

### `timeoutCount`

- Definition: total number of timed-out runs

### `timedOutRuns`

- Definition: list or count of runs whose final status is `timed_out`

### `runsExceeding15s`

- Definition: runs whose duration exceeded 15 seconds

### `runsExceeding25s`

- Definition: runs whose duration exceeded 25 seconds

### `runsExceedingApiGatewayLimit`

- Definition: runs that would likely exceed the current API response budget

### `toolTimeoutCount`

- Definition: timeouts that occurred during tool execution

### `modelTimeoutCount`

- Definition: timeouts that occurred during model calls

### `runnerTimeoutCount`

- Definition: timeouts attributed to the runner boundary itself

Selected_mark_explanation is expected to have low timeout rates.

For comparison:

- LambdaAgentRunner should be near zero in normal deterministic runs
- AgentCoreRunner should be measured for overhead-induced timeout risk

## 6. Failure metrics

### `failureRate`

- Definition: failed runs divided by total runs

### `fallbackRate`

- Definition: runs that completed via expected fallback behavior divided by total runs

### `partialRate`

- Definition: runs that completed partially with warnings divided by total runs

### `unexpectedErrorRate`

- Definition: runs that failed due to unexpected exceptions or unclassified errors

### `preconditionFailureRate`

- Definition: runs blocked by required precondition failures

### `selectedMarksMissingRate`

- Definition: runs that fell back because selected marks were missing

### `optionalContextMissingRate`

- Definition: runs that proceeded with warnings because optional context was missing

### `toolExecutionFailureRate`

- Definition: runs that failed during tool execution

### `responseCompositionFailureRate`

- Definition: runs that failed while building the final deterministic response

### `unsupportedModeRate`

- Definition: runs rejected because the runner did not support the requested mode

For `selected_mark_explanation`, distinguish:

- expected fallback: no selected marks
- warning-only continuation: optional context missing
- unexpected failure: code error or unclassified exception
- timeout: runtime exceeded budget
- unsupported request: wrong mode or misrouted request

## 7. Trace completeness metrics

### `traceEventCount`

- Definition: number of trace events emitted for the run

### `hasRunnerStartedEvent`

- Definition: whether the trace includes a runner start event

### `hasRunnerCompletedEvent`

- Definition: whether the trace includes a runner completion event

### `hasIntentResolvedEvent`

- Definition: whether the trace includes intent resolution

### `hasPlanSelectedEvent`

- Definition: whether the trace includes plan selection

### `hasToolRegistryLookupEvent`

- Definition: whether the trace includes registry lookup

### `hasToolExecutionStartedEvent`

- Definition: whether the trace includes tool execution start

### `hasToolExecutionCompletedEvent`

- Definition: whether the trace includes tool execution completion

### `hasResponseComposedEvent`

- Definition: whether the trace includes response composition completion

### `hasFallbackReason`

- Definition: whether the trace or result contains a fallback reason

### `hasErrorSummary`

- Definition: whether the trace or result contains a safe error summary

### `traceCompletenessScore`

- Definition: required events present divided by required events total
- Suggested required events for selected_mark_explanation:
  - runner started
  - intent resolved
  - plan selected
  - tool lookup
  - tool execution started
  - tool execution completed or fallback recorded
  - response composed
  - runner completed

Score can be normalized to 0-1 or 0-100. Use one form consistently in the comparison notes.

## 8. Observability usefulness metrics

### `canInspectIntermediateOutputs`

- Definition: whether the run exposes safe summaries of intermediate outputs

### `canCorrelateWithCloudLogs`

- Definition: whether the run can be correlated with platform logs or cloud traces

### `canIdentifyFailureStep`

- Definition: whether the failing step can be located quickly

### `canIdentifyTimeoutSource`

- Definition: whether timeout origin is visible

### `canInspectToolExecution`

- Definition: whether tool lookup and tool execution details are visible

### `canCompareRuns`

- Definition: whether repeated runs can be compared using the same fields

### `hasRunLevelMetadata`

- Definition: whether run-level metadata is present

### `hasSessionOrMemoryReadiness`

- Definition: whether the runtime exposes session or future memory related identifiers

### `observabilityUsefulnessScore`

- Definition: qualitative score
- Suggested scale:
  - 0 = cannot observe
  - 1 = basic logs only
  - 2 = trace summary available
  - 3 = step-level trace available
  - 4 = trace + cloud correlation
  - 5 = trace + cloud correlation + intermediate output inspection

LambdaAgentRunner should already score at least at trace summary / step-level visibility for the selected-mark path.
AgentCoreRunner should be evaluated on whether it raises the score meaningfully.

## 9. Implementation complexity metrics

### `filesChangedEstimate`

- Definition: estimated number of files required for a first implementation

### `newAwsResourcesRequired`

- Definition: count of new AWS resources or managed services needed

### `newEnvironmentVariablesRequired`

- Definition: number of new environment variables or feature flags

### `newIamPoliciesRequired`

- Definition: number of new IAM permissions or policy groups

### `newDependenciesRequired`

- Definition: number of new runtime dependencies

### `runtimeCouplingRisk`

- Definition: risk of coupling the runner too tightly to a cloud runtime

### `migrationRisk`

- Definition: risk of breaking existing selected_mark_explanation behavior during migration

### `rollbackComplexity`

- Definition: how hard it is to revert the change

### `implementationComplexityScore`

- Definition: normalized score
- Suggested scale:
  - 1 = docs/config only
  - 2 = isolated runner stub
  - 3 = isolated runner with fake client
  - 4 = real integration behind feature flag
  - 5 = broad runtime migration

## 11. Cost estimate metrics

### `estimatedCostPerRunUsd`

- Definition: rough total cost per single run

### `estimatedCostPer100RunsUsd`

- Definition: same estimate scaled to 100 runs

### `modelCostUsd`

- Definition: cost of model calls, if any

### `agentCoreRuntimeCostUsd`

- Definition: cost of the managed runtime itself, if any

### `toolExecutionCostUsd`

- Definition: cost associated with tool execution or related compute

### `observabilityCostUsd`

- Definition: cost of logs, traces, or extra observability features

### `cloudWatchLogCostUsd`

- Definition: log storage and ingestion cost estimate

### `additionalAwsResourceCost`

- Definition: cost of any extra AWS resources introduced by the candidate design

### `costUncertaintyLevel`

- Definition: how uncertain the estimate is

Cost guidance:

- keep estimates coarse
- include only the costs needed for a decision
- do not pretend precision that the spike does not have

## 12. Debugging effort metrics

### `timeToIdentifyFailureStep`

- Definition: estimated time to identify the failing step

### `timeToFindRelevantLogs`

- Definition: estimated time to find the useful logs

### `timeToReproduceLocally`

- Definition: estimated time to reproduce the issue locally

### `timeToInspectInputContext`

- Definition: estimated time to inspect the input context safely

### `timeToInspectToolResult`

- Definition: estimated time to inspect tool output safely

### `timeToCompareRuns`

- Definition: estimated time to compare two runs

### `debuggingEffortScore`

- Definition: normalized score
- Suggested scale:
  - 1 = very easy
  - 2 = easy
  - 3 = moderate
  - 4 = difficult
  - 5 = very difficult

AgentCore may improve cloud-side debugging, but can also make local diagnosis harder.

## 13. selected_mark_explanation consistency metrics

### `statusConsistency`

- Definition: whether the high-level status behaves consistently across repeated runs

### `messageConsistency`

- Definition: whether the user-facing message stays stable for the same input class

### `responseMaterialConsistency`

- Definition: whether the structured response material stays comparable

### `traceConsistency`

- Definition: whether trace structure stays comparable

### `fallbackConsistency`

- Definition: whether fallback behavior is stable

### `warningConsistency`

- Definition: whether warning behavior is stable for optional context missing

### `noLlmGenerationConfirmed`

- Definition: whether the run avoided LLM generation

### `noTableauMcpConfirmed`

- Definition: whether the run avoided Tableau MCP

### `rawDataSuppressionConfirmed`

- Definition: whether raw selected marks and full rows were suppressed

### `selectedMarkCountConsistency`

- Definition: whether the selected mark count is preserved in the summary

### `optionalContextHandlingConsistency`

- Definition: whether optional context is handled in the same policy shape

### `consistencyScore`

- Definition: normalized consistency score across the required conditions

For initial comparison, exact runtime internals do not need to be identical.
The comparison should insist on:

- missing selected marks lead to expected fallback
- selected mark count remains consistent
- optional context missing remains warning-only
- no LLM calls
- no Tableau MCP calls
- no raw selected marks or full rows
- deterministic / placeholder response remains the same kind of result

## 14. Comparison scenarios

### Scenario 1: Happy path

- selected marks: present
- summaryDataPreview: present
- filters: present
- parameters: present

Compare:

- latency
- trace completeness
- output consistency
- observability usefulness
- cost estimate

### Scenario 2: Minimal context path

- selected marks: present
- summaryDataPreview: absent
- filters: absent
- parameters: absent

Compare:

- fallback avoidance
- warning behavior
- consistency

### Scenario 3: Missing selected marks

- selected marks: absent
- expected fallback

Compare:

- fallback rate
- fallback consistency
- error summaries
- trace completeness

### Scenario 4: Optional context missing

- selected marks: present
- one or more optional context fields missing
- warning-only continuation

Compare:

- warning consistency
- partial rate
- message consistency

### Scenario 5: Large context summary

- high mark count or high column count
- truncation behavior

Compare:

- responseMaterialConsistency
- raw data suppression
- trace completeness

### Scenario 6: Runner failure simulation

- internal runner failure
- safe failed or fallback result

Compare:

- failure rate
- unexpectedErrorRate
- debugging effort

### Scenario 7: Timeout simulation

- timeout-like result
- trace, fallback, and status

Compare:

- timeout rate
- runnerTimeoutCount
- failure handling

## 16. Suggested decision thresholds

AgentCoreRunner should be considered further if:

- trace completeness is equal or better than LambdaAgentRunner
- observability usefulness improves by at least 1 point
- selected_mark_explanation consistency remains acceptable
- p95 latency remains within the API response budget
- local testability remains acceptable through fake clients
- implementation complexity remains at or below the agreed threshold
- estimated cost is acceptable for demo / PoC usage

AgentCoreRunner should be deferred if:

- it adds significant latency without observability benefits
- it requires broad runtime migration
- it reduces local testability
- it makes selected_mark_explanation less deterministic
- it requires production-like AWS setup for basic tests
