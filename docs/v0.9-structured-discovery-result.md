# v0.9.0 Structured Discovery Result

## Summary

v0.9.0 delivered a narrow structured metadata discovery surface rather than a broad chat replacement.

Eligible metadata discovery cases moved into a structured path, while legacy free-form chat remained available for unsupported, ambiguous, and non-discovery requests.

The milestone introduced a structured `metadata_discovery` intent, an ambiguity model, clarification response contracts, a narrow plan shape, a safe routing boundary to the hosted `describeDatasource` path, a controlled `listFields` contract, truncation and permission-aware response shaping, and regression protection for `selected_mark_explanation` and the free-form chat fallback path.

## Relationship to v0.8.0

v0.8.0 validated the first hosted metadata boundary for `describeDatasource`.

v0.9.0 reused that validated boundary instead of introducing a new raw Hosted MCP surface.

The v0.8.0 safety posture carried forward unchanged:

- hosted execution stayed explicit and opt-in
- default CI stayed no-network
- fake fallback remained the default safety posture
- stdio fallback remained preserved
- raw MCP tool exposure stayed out of scope
- arbitrary query execution stayed out of scope
- underlying data access stayed out of scope
- field values and row data stayed out of scope
- write-capable tools stayed out of scope
- tokens, secrets, and stack traces stayed out of user-facing output and traces

## What moved into structured discovery

The following moved into the structured path:

- the `metadata_discovery` intent
- ambiguity evaluation for datasource / workbook / view targets
- clarification responses for missing, ambiguous, or unsafe requests
- the narrow structured metadata discovery plan shape
- safe executable datasource metadata discovery plans
- the hosted `describeDatasource` boundary for eligible datasource cases
- truncation and permission-aware response shaping
- a limited structured-vs-legacy routing gate

Only discovery-oriented requests that were safe to structure were moved.
The milestone did not expand into general-purpose planning or broad Hosted MCP exposure.

## What stayed on the legacy path

The following intentionally stayed on the legacy path:

- non-discovery free-form chat
- broad analysis requests
- unsupported metadata-ish requests
- ambiguous or missing-identifier requests that could not be safely resolved
- workbook or view targets that could not be safely resolved to a datasource
- `selected_mark_explanation`
- arbitrary query requests
- underlying data, field values, and row data requests
- write or update requests
- raw MCP tool requests
- the legacy free-form chat fallback path

## metadata_discovery intent result

The `metadata_discovery` intent was defined as a structured orchestration intent for safe metadata-first discovery cases.

Its result established:

- a stable intent name
- structured input and output envelopes
- routing conditions that avoid raw execution
- explicit fallback and unsupported handling
- a guarantee that the intent does not directly expose raw MCP payloads or transport details

## Datasource / workbook / view ambiguity result

The ambiguity model distinguishes:

- datasource
- workbook
- view
- unknown
- ambiguous
- unsupported, where needed

It also captures:

- missing or insufficient identifiers
- multiple candidates
- cases that cannot be safely resolved from current context
- cases that should return clarification instead of speculative execution

The outcome is intentionally conservative: if the target cannot be resolved safely, the runtime prefers clarification or fallback over guessing.

## Clarification response result

Clarification responses were defined as safe, JSON-friendly contracts for incomplete or ambiguous requests.

The result established:

- clarification-required responses
- unsupported and unsafe responses
- stable reason codes
- safe response fields
- a caller resume contract

Clarification responses do not contain raw MCP payloads, raw transport output, tokens, secrets, stack traces, underlying data, field values, or row data.

## Structured metadata discovery plan result

The plan shape defined in #186 is a narrow metadata-discovery-only envelope.

It separates:

- `clarification_required`
- `executable`
- `unsupported`
- `fallback`

The result confirmed that the plan is:

- ToolLayer-friendly
- JSON-safe
- narrow enough to avoid a general planning DSL
- suitable for later routing without exposing raw tool surfaces

## describeDatasource boundary result

Eligible executable datasource plans were connected to the existing `describeDatasource` boundary.

The result confirmed that:

- only safe datasource cases are eligible
- the existing hosted boundary was reused
- feature-flag and configuration gating remained in place
- fake / no-network fallback remained intact
- stdio fallback remained intact
- no new raw MCP surface was introduced

`describeDatasource` remains the first validated hosted metadata boundary, not a general Hosted MCP exposure point.

## Controlled listFields result

`listFields` remained a controlled candidate rather than a new execution surface.

The result established:

- controlled limits and caps
- truncation and omission signals
- permission-aware behavior
- validation gates before any future execution wiring
- a safe field-metadata subset only

`listFields` did not gain actual execution wiring in v0.9.0.

## Truncation and permission-aware response result

Large, incomplete, or restricted discovery responses were shaped into safe summaries.

The result established:

- truncation indicators
- warning notes
- permission-aware limitation notes
- stable warning and reason codes
- trace-safe summaries
- summary-first and JSON-safe output

This shaping suppressed raw MCP output, raw transport output, and stack traces.

## Legacy chat reduction result

v0.9.0 reduced legacy chat coverage only for eligible discovery cases.

The result confirmed that:

- eligible metadata discovery cases moved to structured orchestration
- unsupported, ambiguous, and non-discovery cases stayed in legacy fallback
- `ChatService` remained in place
- `runLightweightAgentLoop` remained in place

The milestone was a reduction step, not a full replacement step.

## selected_mark_explanation and fallback regression result

Regression coverage in #191 protected the deterministic selected-mark explanation path and the free-form chat fallback path.

The result confirmed that:

- `selected_mark_explanation` stayed unchanged
- selected-mark requests were not misclassified as metadata discovery
- selected marks context was not repurposed as a metadata discovery target
- non-discovery free-form chat remained on legacy fallback
- unsupported and ambiguous cases did not leak into structured execution

## Safety boundaries preserved

v0.9.0 preserved the following safety boundaries:

- read-only metadata first
- hosted execution explicit opt-in
- default CI no-network
- fake fallback preserved
- stdio fallback preserved
- Hosted integration tests optional and gated
- no raw MCP tool exposure
- no raw MCP result exposure
- no raw transport result exposure
- no arbitrary query execution
- no underlying data access
- no field values or row data retrieval
- no write-capable tools
- no tokens, secrets, or stack traces in user-facing output or traces
- output normalization
- JSON-safe output
- `selected_mark_explanation` unchanged
- legacy free-form chat fallback preserved
- AgentCore deferred
- LLM ResponseComposer not implemented
- frontend redesign not done

## Deferred work

The following work remains deferred beyond v0.9.0:

- `listFields` execution
- broader workbook / view resolution
- Tableau content crawling
- arbitrary query execution
- underlying data access
- field values / row data access
- write-capable tools
- production Hosted MCP migration
- OAuth full implementation
- token storage / refresh
- AgentCore
- LLM ResponseComposer
- exploration session / multi-turn memory
- full `ChatService` replacement
- full `runLightweightAgentLoop` replacement
- broader legacy chat migration

## Remaining risks and follow-ups

The v0.9.0 result leaves a deliberately narrow surface behind.

Remaining follow-ups include:

- workbook / view resolution remains limited and deferred
- `listFields` is still contract-first and execution-deferred
- permission-aware behavior still depends on the underlying transport and Tableau permission model
- the structured / legacy boundary will continue to need regression coverage
- optional hosted integration validation remains useful for later environment-specific confidence
- v0.10.0 still needs to decide between safer LLM ResponseComposer support and an exploration-session model

## Validation summary

This milestone result documentation is based on the completed v0.9.0 code-bearing and test-focused issues:

- #183 through #191

The relevant backend work was validated in the issue sequence with no-network defaults preserved.

Docs-only issue #192 itself did not add new runtime validation.

## Notes for v0.10.0

v0.10.0 can now make a decision from a narrow, explicit, regression-safe structured discovery surface.

If v0.10.0 moves toward LLM ResponseComposer support, it still needs deterministic baselines and evidence discipline.

If v0.10.0 moves toward an exploration-session model, it still needs no-network defaults, fallback preservation, and the same safety boundaries.

AgentCore remains deferred and should be treated as a separate decision input rather than an implied next step.
