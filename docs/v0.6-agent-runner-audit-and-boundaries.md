# v0.6.0 Agent Runner Audit and Boundaries

## 1. Purpose

This document audits the current backend runtime responsibilities and defines the minimum boundary where an `AgentRunner` wrapper should sit for v0.6.0.

The goal is not to change behavior now. The goal is to make the current runtime comparable so that a later `LambdaAgentRunner` and a future `AgentCoreRunner` can be evaluated against the same selected_mark_explanation path.

## 2. Current runtime overview

The current backend runtime is split across several layers:

- `ChatService` owns the legacy free-form chat service flow
- `runLightweightAgentLoop` owns the legacy agent planning loop
- `chatHandler` owns HTTP parsing, auth, route selection, and response serialization
- `selected_mark_explanation` is already routed through a dedicated structured orchestration path
- v0.5 Tool Layer owns tool definitions, routing, registry lookup, execution wrapping, trace-first tool events, and deterministic response composition

The most important existing split is this:

- free-form chat still uses the legacy `ChatService` + lightweight loop path
- selected_mark_explanation already uses the structured orchestration + Tool Layer path
- the explicit selected_mark_explanation action path is wrapped by `LambdaAgentRunner`, while free-form chat stays on the legacy path

That makes selected_mark_explanation the smallest and cleanest runner comparison target.

## 3. ChatService responsibility audit

Current `ChatService` responsibilities include:

- accepting the user chat request from the handler layer
- validating the request shape and required dashboard context
- loading recent session history
- looking up Tableau context through the configured context provider
- deciding whether the fast path or agent loop path should run
- invoking `runLightweightAgentLoop` for the agent path
- handling deadline-aware fallback behavior
- composing the final user-facing answer
- persisting the chat job / answer result
- attaching Notion draft data and dashboard patch data when needed
- emitting logging and progress updates around each stage

What should stay outside `AgentRunner`:

- HTTP parsing and auth
- job persistence details
- frontend-specific request shaping
- infrastructure-specific response serialization

What can move behind `AgentRunner` later:

- the decision to run a structured runner vs a legacy loop
- the execution result normalization
- trace / fallback aggregation for a specific run mode
- response composition for structured orchestration paths

## 4. runLightweightAgentLoop responsibility audit

Current `runLightweightAgentLoop` responsibilities include:

- deriving or reusing the base question interpretation
- deciding whether the chat agent should run
- building a plan through the configured `ChatAgent`
- applying request-type overrides to the plan
- iterating context retrieval and evaluation passes
- asking the context provider for more Tableau context
- merging contexts across passes
- evaluating whether the collected context is sufficient
- handling deadline guards and no-progress fallback
- collecting debug data about passes, plans, and evaluation outputs

The loop currently mixes concerns that are now also represented in structured orchestration:

- planning
- context retrieval
- retry/fallback
- deadline handling
- evidence aggregation

This is the strongest argument for wrapping the current runtime behind an `AgentRunner` abstraction rather than expanding the legacy loop further.

The likely direction is:

- keep `runLightweightAgentLoop` as the legacy free-form chat implementation for now
- let `AgentRunner` wrap it only where a runner abstraction is needed
- do not expand the legacy loop to cover structured orchestration cases

## 5. selected_mark_explanation structured path audit

Current selected_mark_explanation path:

```text
frontend action
-> /intent/resolve endpoint with runMode=resolve_and_execute_fixed_plan
-> intent resolution
-> fixed plan selection
-> ExecutionEngine
-> ToolRouter
-> ToolRegistry
-> ToolExecutionWrapper
-> context.selectedMarks required
-> context.summaryDataPreview optional
-> context.filters optional
-> context.parameters optional
-> responseMaterial
-> ResponseComposer skeleton
-> deterministic / placeholder response
```

Current backend responsibilities on this path:

- `chatHandler` parses the explicit orchestration request
- `createDefaultIntentResolver()` resolves the intent
- `runSelectedMarkExplanationOrchestration()` coordinates the structured path
- fixed plan selection determines whether the selected-mark path can run
- `ExecutionEngine` drives the plan step processing
- `ToolRouter` consults `ToolRegistry` and `ToolExecutionWrapper`
- `selectedMarkContextTools` builds the selected-mark response material
- `composeSelectedMarkExplanationResponse()` creates the deterministic placeholder response
- orchestration trace events are collected and returned with the response

What belongs inside `AgentRunner.run()` for this path:

- intent resolution
- plan selection
- execution engine invocation
- tool routing / lookup / execution
- response composition
- trace aggregation
- fallback normalization
- budget aggregation

What should stay outside `AgentRunner.run()`:

- HTTP parsing
- auth
- frontend request formatting
- API response serialization
- route dispatching
- job persistence

## 6. Tool Layer boundary review

Current Tool Layer responsibilities are already clear:

- `ToolDefinition` describes a tool without executing it
- `ToolRegistry` stores and looks up definitions
- `ToolPrecondition` describes safety and availability requirements
- `ToolRouter` decides whether a tool can be used
- `ToolExecutionWrapper` safely executes a registered tool
- tool trace helpers summarize lookup, precondition, and execution results
- `ResponseComposer` turns collected materials into a user-facing response

Boundary recommendation:

- Tool Layer should live inside the runner boundary for selected_mark_explanation
- `ToolRegistry` can be shared as a read-only contract or factory product, but the runner should own the runtime view of it
- `ToolExecutionWrapper` is part of the execution path, not the API layer
- `ResponseComposer` belongs inside the runner because it normalizes execution output into a final result
- trace event collection should be inside the runner so observability follows the execution path

AgentCore compatibility:

- the current Tool Layer should remain reusable
- `AgentCoreRunner` should be able to call the same Tool Layer contract where possible
- if AgentCore later owns tool execution, the same result shapes should still be returned to the runner contract
- the current fit assessment for `selected_mark_explanation` is conditional, so the custom runner boundary should stay intact until comparison data justifies a migration

## 7. Responsibility overlap and duplication

The main overlaps are:

- ChatService response composition vs ResponseComposer
- runLightweightAgentLoop planning vs IntentResolver / fixed plan selection
- runLightweightAgentLoop context retrieval / tool execution vs ToolRouter / ToolRegistry / ToolExecutionWrapper
- existing logging / evidence vs trace-first events
- deadline fallback in legacy flow vs structured fallback result
- selected_mark_explanation vs free-form chat path

How to classify each overlap:

- ChatService response composition vs ResponseComposer: facade now, replace by runner result later
- runLightweightAgentLoop planning vs IntentResolver / Plan: keep legacy for free-form chat, structured path should own selected_mark_explanation
- runLightweightAgentLoop tool execution vs Tool Layer: facade now, structured path already owns selected_mark_explanation
- existing logging / evidence vs trace-first events: facade or parallel for now, but runner should own the comparable trace view
- timeout handling vs RunBudget / ToolExecutionWrapper: keep current behavior, but move comparison-ready timeout data into runner output
- fallback handling vs structured fallback result: keep legacy path fallback for free-form chat, normalize structured fallback in runner results
- selected_mark_explanation vs free-form chat path: keep separate for now

## 8. AgentRunner insertion point options

Options considered:

### Option A

```text
API route / ChatService
-> AgentRunner.run()
-> IntentResolver / Plan / ExecutionEngine / ToolLayer / ResponseComposer
```

This is the cleanest wrapper boundary for structured orchestration, but it still leaves free-form chat outside.

### Option B

```text
ChatService
-> AgentRunner.run()
-> existing ChatService internal flow or structured orchestration
```

This keeps the runner close to the legacy service, but it can blur the boundary if it is applied too broadly.

### Option C

```text
selected_mark_explanation endpoint/mode only
-> AgentRunner.run()
-> structured orchestration path
```

This is the safest first step and best matches the current roadmap.

### Option D

```text
free-form chat and selected_mark_explanation both
-> AgentRunner.run()
-> runtime decides legacy vs structured path
```

This is a later-stage option. It is too broad for the first comparison phase.

Recommended direction:

- use Option C first
- keep free-form chat on the existing path
- let `AgentRunner` compare selected_mark_explanation only
- route the explicit selected-mark action path through `LambdaAgentRunner` without moving free-form chat

## 9. Recommended minimum wrapper boundary

Minimum boundary for v0.6.0:

### Outside AgentRunner

- HTTP / API event parsing
- auth / session validation
- frontend-specific request shape
- existing createChatJob behavior
- free-form chat legacy flow
- UI rendering
- infrastructure-specific Lambda handler logic

### Inside AgentRunner

- agentRunId propagation
- intent resolution
- plan selection
- execution engine invocation
- tool routing / registry / execution
- response composition
- trace event collection
- budget usage aggregation
- fallback normalization
- runner metadata

### Not inside AgentRunner

- frontend-specific UI logic
- raw API response formatting details
- auth internals
- infrastructure-specific Lambda event handling
- persistence implementation details
- Cloud provider-specific AgentCore client details

## 10. What stays outside AgentRunner

- API gateway event parsing and validation
- auth and user/session validation
- route selection
- job creation and persistence
- frontend request shape and UI behavior
- legacy free-form chat path
- infrastructure-specific deployment concerns

## 11. What belongs inside AgentRunner

- propagation of `agentRunId`
- `selected_mark_explanation` intent resolution
- fixed plan selection
- `ExecutionEngine` invocation
- `ToolRouter` / `ToolRegistry` / `ToolExecutionWrapper` orchestration
- `ResponseComposer` invocation
- trace-first result aggregation
- budget / timeout summary
- fallback reason normalization
- runner metadata and observability summary

## 12. What remains legacy for now

- free-form chat stays on `ChatService` + `runLightweightAgentLoop`
- `createChatJob` keeps its existing behavior
- `ChatService` remains the entry point for the legacy chat path
- selected_mark_explanation is the only path to consider for runner comparison first

## 13. AgentCore considerations

There is already a thin `AgentRunner` abstraction in the codebase:

- `backend/src/agent/runner.ts`
- `backend/src/agent/lambdaRunner.ts`

The current `LambdaAgentRunner` is now a thin wrapper around the existing selected_mark_explanation structured orchestration path. It is still intentionally small and does not broaden free-form chat behavior.

That means:

- the existing runner abstraction is useful
- the wrapper is thin enough to compare against a future AgentCoreRunner
- v0.6.0 should refine the runner boundary around selected_mark_explanation, not broaden the legacy chat path further

The future `AgentCoreRunner` should remain a comparison target, not a migration target for the first spike.

## 15. Recommendations for Issue #101

For the next issue, the runner contract should focus on:

- a small and stable `AgentRunner` interface
- `AgentRunInput` with enough information to run selected_mark_explanation and compare runtimes
- `AgentRunResult` with comparable observability, budget, trace, warning, and fallback data
- compatibility with both Lambda and future AgentCore implementations
- preserve the existing thin wrapper behavior while widening the contract for comparison fields only

Recommended contract direction:

- keep selected_mark_explanation as the first runnerized path
- keep free-form chat outside the runner for now
- preserve the current Tool Layer as the execution core
- make `LambdaAgentRunner` the default comparison wrapper before introducing `AgentCoreRunner`

## 16. Open questions

- Should the current `AgentRunner` contract be widened in place, or should v0.6.0 add a runner-specific contract on top of it?
- Should `LambdaAgentRunner` become the selected_mark_explanation wrapper while the legacy chat path stays separate?
- Should trace comparison happen in the runner result, or as a separate observability artifact?
- Should `ToolRegistry` be shared across runner types or instantiated per runner factory?
- Should `AgentCoreRunner` eventually own planning, or only execution and observability?
