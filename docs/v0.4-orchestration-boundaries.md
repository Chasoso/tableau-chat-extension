# v0.4 Orchestration Audit and Boundaries

## 1. Purpose

This document clarifies the current orchestration boundaries before v0.4.0 "Structured Orchestration" is implemented.

The goal is not to replace the current chat flow yet. The goal is to document:

- what the existing backend and frontend orchestration layers currently do
- where responsibilities overlap
- which boundaries should be introduced for `Intent -> Plan -> Tools -> Response`
- how to keep the existing chat flow working during migration
- how to unlock a safe first path from selected-mark actions to `selected_mark_explanation`

## 2. Current user question flow

The current user-facing question flow is still centered on the existing chat job pipeline:

1. The user types a question in `ChatPanel`.
2. `frontend/src/api/chatApi.ts` sends the request through `createChatJob()`.
3. `backend/src/handlers/chatHandler.ts` receives the `/chat-jobs` request.
4. `backend/src/services/chatJobService.ts` persists the job and later executes it.
5. `ChatJobService` creates a `ChatService` instance and calls `generateAnswer()`.
6. `backend/src/services/chatService.ts` interprets the question, checks fast paths, and runs `runLightweightAgentLoop()` when needed.
7. `backend/src/services/chatAgent.ts` performs the current Bedrock-guided planning/evaluation loop.
8. `backend/src/tableau/tableauMcpContextProvider.ts` gathers additional Tableau context and MCP observations.
9. `backend/src/services/answerGenerator.ts` produces the final response text.
10. The chat job result is saved and polled back to the frontend.

For the preview-driven action path:

1. The user selects marks in Tableau.
2. `AIContextPreviewPanel` shows a read-only preview of the current Tableau context.
3. `buildContextActionSuggestions()` offers safe actions such as "Explain this selection".
4. The action currently pre-fills `ChatPanel` input only.
5. The user still has to submit the message manually.
6. The existing chat job flow then handles the request.

This means the current system still has one shared execution path for free-form chat and preview-prefilled chat.

## 3. Current backend orchestration responsibilities

### `ChatService`

Current responsibilities in `backend/src/services/chatService.ts` include:

- request interpretation
- fast-path response decisions
- dashboard context lookup and patching
- progress reporting
- invoking `runLightweightAgentLoop()`
- metadata-path fallback and deadline-aware fallback
- response composition and persistence
- Notion memo side effects

`ChatService` currently owns too many orchestration concerns at once.

### `runLightweightAgentLoop()`

Current responsibilities in `backend/src/services/chatAgent.ts` include:

- deciding whether the lightweight agent should run
- creating a plan with Bedrock or heuristic fallback
- applying request-type overrides
- executing multiple context passes
- asking for additional Tableau context
- evaluating whether the current context is sufficient
- deciding whether to replan or ask a follow-up question
- stopping on deadline or repeated follow-up conditions

This function currently mixes:

- intent-like decision making
- plan selection
- execution control
- retry/fallback logic
- trace-like debug data

### `LambdaAgentRunner`

`backend/src/agent/lambdaRunner.ts` already provides a runner abstraction, but it is still a thin wrapper around the existing `ChatService` path and is not yet the active orchestration entry.

### `TableauMcpContextProvider`

`backend/src/tableau/tableauMcpContextProvider.ts` still holds a large portion of the Tableau/MCP work:

- dashboard context retrieval
- MCP lookup and fallback behavior
- MCP tool planning support
- context normalization and warnings

This is appropriate for v0.3 context collection, but it is too much to remain the long-term orchestration boundary.

### Trace and logging

Trace-like information exists, but it is still fragmented across:

- service logs
- agent loop debug output
- MCP context debug structures
- chat job persistence

The current trace story is useful for debugging, but not yet a single orchestration trace model.

## 4. Current frontend action responsibilities

### `AIContextPreviewPanel`

The preview panel is currently read-only and shows:

- overview
- worksheets
- filters
- parameters
- selected marks
- summary data preview
- warnings and availability
- suggested actions

### Action suggestion handling

`buildContextActionSuggestions()` currently generates safe UI suggestions from the preview model.

At the moment, the suggestion click handler only pre-fills `ChatPanel` input.

That is intentionally safe, but it means the frontend action is not yet a first-class orchestration request.

### `ChatPanel` and app state

`ChatPanel` still owns the manual send path to `createChatJob()`.

`App` keeps:

- the current dashboard context
- the preview model
- the prefill text
- the preview update loop for `MarkSelectionChanged`

The frontend is now capable of expressing a user intent, but it does not yet send that intent as a structured orchestration request.

## 5. Responsibility overlap and risks

The following responsibilities are currently overlapping:

- intent-like judgment appears in `ChatService`, `runLightweightAgentLoop()`, `questionInterpretation`, and the frontend suggestion UI
- plan selection appears in `chatAgent.ts`, `ChatService`, and the MCP planning path
- tool selection appears in the agent loop and `TableauMcpContextProvider`
- fallback behavior appears in multiple places
- response composition is split between `answerGenerator`, `ChatService`, and job persistence
- trace/debug data is still spread across logs, response metadata, and debug structures

Risks caused by this overlap:

- it is hard to add a new intent without touching multiple layers
- it is hard to cap model/tool calls in a single place
- it is hard to see which plan produced which result
- it is hard to keep free-form chat behavior stable while adding action-driven orchestration

## 6. Proposed v0.4 boundaries

### 6.1 IntentResolver boundary

Current closest responsibility:

- `questionInterpretation`
- `chatAgent.shouldRun()`
- the preview action suggestion UI

v0.4 boundary:

- resolve a stable orchestration intent from the request envelope
- prefer deterministic action metadata over free-form inference
- emit a fallback when the intent is unsupported or ambiguous

First intent:

- `selected_mark_explanation`

### 6.2 PlanBuilder boundary

Current closest responsibility:

- `chatAgent.createPlan()`
- fixed plan definitions in `backend/src/agent/fixedPlans.ts`

v0.4 boundary:

- map a resolved intent to a fixed or unsupported plan
- keep required context packs, allowed tools, and budget in one place
- avoid embedding execution logic in the plan builder

First intent:

- `selected_mark_explanation`

### 6.3 ToolRouter boundary

Current closest responsibility:

- `TableauMcpContextProvider`
- MCP tool planning and execution logic

v0.4 boundary:

- decide which tools are allowed for the current plan
- record skipped or blocked tools
- keep tool choice separate from execution control

First intent:

- `selected_mark_explanation`

### 6.4 ExecutionEngine boundary

Current closest responsibility:

- `runLightweightAgentLoop()`
- parts of `ChatService.generateAnswer()`

v0.4 boundary:

- execute a selected plan step by step
- track budget usage
- stop on timeout, unsupported tools, or failures
- emit orchestration trace events

First intent:

- `selected_mark_explanation`

### 6.5 ResponseComposer boundary

Current closest responsibility:

- `answerGenerator.generate()`
- the final response assembly logic in `ChatService`

v0.4 boundary:

- turn orchestration results into the final answer payload
- keep answer shaping separate from plan execution
- make the response composer easy to swap later

### 6.6 OrchestrationTrace boundary

Current closest responsibility:

- service logs
- agent-loop debug payloads
- MCP debug structures

v0.4 boundary:

- record resolved intent
- record selected plan
- record context used
- record tool calls and skipped tools
- record budget usage
- record fallback and errors
- keep the trace sink independent of storage backend

### 6.7 RunBudget boundary

Current closest responsibility:

- implicit deadline checks in `runLightweightAgentLoop()`
- agent loop pass limits
- MCP configuration limits

v0.4 boundary:

- define `maxModelCalls`
- define `maxToolCalls`
- define `timeoutMs`
- make the budget explicit at the plan/execution boundary

## 7. selected_mark_explanation minimal path

The first safe orchestration path should be:

1. `AIContextPreviewPanel` shows the user a selected-mark action suggestion.
2. The user chooses "Explain this selection".
3. The frontend sends an explicit action request instead of relying on free-form text alone.
4. The backend resolves `selected_mark_explanation` deterministically.
5. `PlanBuilder` selects the fixed plan for that intent.
6. `ExecutionEngine` runs the plan with bounded context and tools.
7. `ResponseComposer` returns the answer.
8. The trace records the resolved intent, selected plan, budget usage, and any fallback.

The minimal path should not require replacing the free-form chat route.

## 8. Migration strategy

The migration should be incremental:

1. Keep the current chat job pipeline as the default path.
2. Add a separate orchestration entry for action-driven requests.
3. Route only `selected_mark_explanation` through the new path first.
4. Keep free-form chat on the legacy path until the new orchestration is stable.
5. Move additional intents only after traces, budgets, and tool routing are proven.

This reduces risk and allows the existing user experience to remain stable.

## 9. What stays in the existing flow for now

- manual free-form chat submission
- `ChatService` as the main chat job processor
- `runLightweightAgentLoop()` as the fallback agent loop
- current chat job persistence and polling
- existing Tableau MCP behavior
- existing answer generation logic for unsupported orchestration paths
- existing frontend prefill behavior for preview suggestions

## 10. What moves to the new orchestration layer

- explicit intent resolution for action-driven requests
- fixed-plan selection
- tool routing decisions
- orchestration step execution
- orchestration trace emission
- budget enforcement
- response composition for structured runs

## 13. Open questions

- Should the new orchestration entry be a new endpoint, a new request field on `/chat-jobs`, or a thin facade over the existing job API?
- Should `selected_mark_explanation` initially bypass `runLightweightAgentLoop()` entirely, or share a small adapter with the legacy flow?

- How much of the current MCP planner should be reused by the first `ToolRouter` skeleton?
- Should the frontend action suggestion only prefill chat input during the first migration step, or also send a structured request in parallel?

## 14. v0.4 completion summary

v0.4.0 landed as a narrow structured-orchestration slice rather than a full replacement of the legacy chat stack.

### Implemented

- orchestration boundaries and migration constraints
- `IntentResolver` contract and minimal deterministic resolver
- plan execution contract for fixed plans and budgets
- `ToolRouter` skeleton for allow/block/skip decisions
- `ExecutionEngine` skeleton for step-by-step orchestration
- frontend action request plumbing from selected-mark suggestions
- orchestration trace events and trace metadata helpers
- the `selected_mark_explanation` fixed-plan path

### Still on the legacy path

- free-form chat submission
- `ChatService`
- `runLightweightAgentLoop()`
- Tableau MCP execution for normal chat requests
- Bedrock-backed answer generation for the current chat route

### Explicitly deferred

- Tableau MCP tool registry work
- `ToolDefinition` / schema policy work
- broader tool precondition enforcement
- `ResponseComposer` as a first-class runtime component
- AgentCore migration
- persistent trace storage
- trace UI

### v0.5.0 boundary

The next phase should start from the current contracts and rebuild the tool layer around a registry, schema policy, and explicit tool execution semantics.
