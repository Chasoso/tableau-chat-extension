# v0.3 Frontend Tableau Context Collector Audit

## 1. Purpose

This audit documents the current frontend Tableau context collection flow and identifies the gaps that v0.3.0 should close before any orchestration work begins.

The goal is to strengthen the Tableau context available to the user and to the future agent pipeline by clarifying:

- what the frontend already collects
- what is missing or too shallow
- how the current data moves from Tableau Extension API to the backend
- what new frontend responsibilities should be split out for v0.3.0

This audit is intentionally implementation-light. It prepares the next issues rather than changing runtime behavior.

## 2. Current frontend context flow

The current flow is centered on a single `DashboardContext` object.

1. `App` mounts the dashboard experience and calls `initializeTableauExtension()`.
2. `initializeTableauExtension()` tries the Tableau Extensions API and falls back to a mock context when Tableau is unavailable or initialization fails.
3. `getDashboardContext()` builds a `DashboardContext` from the dashboard and worksheet objects.
4. `ChatPanel` receives that `DashboardContext` as a prop and keeps it as the current dashboard state for chat requests.
5. `ChatPanel` performs a one-time enrichment call to backend `/context` when `workbookName` is missing.
6. When the user sends a question, `ChatPanel` calls `createChatJob()` and sends the current `dashboardContext` together with the question.
7. The job result can patch only a small part of the dashboard context today, primarily `workbookName`.

Important details:

- The main chat flow is job-based through `/chat-jobs`.
- The direct `/chat` API still exists in `frontend/src/api/chatApi.ts`, but the primary UI path uses `createChatJob()`.
- `DashboardContextPanel` exists as a simple summary card, but it is not wired into `App` today.
- The current enrichment call is not a broad context collector; it is a narrow backend call that is used mainly to fill in missing `workbookName`.

## 4. Current collected context

The frontend already collects more than just a dashboard title.

### Dashboard / workbook / worksheet context

- `dashboardName`
- `workbookName`
- `workbookId`
- `workbookContentUrl`
- `viewName`
- `viewId`
- `worksheets`
  - worksheet name
  - sheet type
  - worksheet size
  - a short summary string

### Data context

- `filters`
  - worksheet name
  - field name
  - filter type
  - applied values
  - `isAllSelected`
- `parameters`
  - parameter name
  - current value
  - data type
  - allowable values
- `selectedMarks`
  - worksheet name
  - selected columns
  - row count
  - availability status
- `dataSources`
  - worksheet name
  - datasource name
  - datasource id
  - extracted fields when available

### Metadata and availability

- `availability`
  - workbookId availability
  - viewId availability
  - datasourceFields availability
- `contextSource`
- `contextWarning`
- `capturedAt`

### Current mock context coverage

`createMockDashboardContext()` already includes representative examples for:

- worksheets
- filters
- parameters
- selected marks
- data sources
- availability

This is useful for UI development, but it is still a mock-first snapshot rather than a richer live context preview.

## 5. Missing or insufficient context

The current frontend context is useful, but still not enough for a context-first v0.3 experience.

### Missing or shallow context

- `summary data preview`
  - no bounded preview rows or preview table structure
- `last changed worksheet`
  - no explicit field for the worksheet that most recently changed
- `mark selection event`
  - no event-driven state update path yet
- `context preview` shaped data
  - the current `DashboardContext` is useful, but it is not yet explicitly designed as a preview model
- richer selection details
  - selected marks are represented only as worksheet-level summaries
  - the selected mark payload is not normalized into a preview-friendly structure
- collector lifecycle
  - there is no dedicated `DashboardStateCollector` abstraction yet

### Frontend/backend alignment gaps

- `ChatPanel` currently sends the whole `dashboardContext` directly to chat jobs.
- backend `/context` enrichment is only used as a narrow patch path.
- the frontend and backend share similar nouns, but not yet a fully intentional shared preview model.

## 6. Recommended v0.3 frontend architecture

For v0.3.0, split the frontend responsibility into a context-first pipeline.

### Recommended layers

- `DashboardStateCollector`
  - owns the initial Tableau Extension API snapshot
- `filters collector`
  - extracts dashboard/worksheet filter state
- `parameters collector`
  - extracts parameter state
- `selected marks collector`
  - extracts current selected mark state with preview bounds
- `summary data preview collector`
  - extracts a small preview slice, not full underlying data
- `MarkSelectionChanged listener`
  - refreshes the context state when selection changes
- `Context Preview model`
  - frontend-facing preview structure aligned with backend `ContextPack`
- `AI Context Preview panel`
  - read-only UI for the current preview model

### Design intent

- Keep Tableau state collection separate from AI execution.
- Make the preview model visible to the user before any LLM call.
- Keep the preview model close to the backend `ContextPack` naming so the future handoff is obvious.
- Avoid introducing orchestration logic in v0.3.0.

## 7. Proposed data flow

Recommended v0.3 flow:

1. `App` initializes Tableau extension access.
2. `DashboardStateCollector` builds the base preview state.
3. The filters / parameters / selected marks / summary preview collectors populate the preview model.
4. `MarkSelectionChanged` refreshes the preview state when worksheet selection changes.
5. `AI Context Preview panel` renders the current state in the UI.
6. `ChatPanel` still sends the current `DashboardContext` to the backend for now.
7. backend `/context` remains the existing enrichment path until v0.4 changes it.

This keeps v0.3 focused on the Tableau context surface, not on execution orchestration.

## 9. Risks and constraints

- Tableau Extension API availability can differ between Tableau Desktop, Tableau Server, and Tableau Cloud.
- Some worksheet APIs may not be present for every dashboard or workbook state.
- Selected marks can grow quickly, so the preview must be bounded.
- Summary data preview can become large if it is not capped early.
- Selection and worksheet-change events may be noisy and should be treated as refresh triggers, not as automatic AI execution triggers.
- Initial extension startup can fail, so the preview path needs a safe fallback state.
- The current frontend flow already has a usable job-based chat path, so v0.3 must avoid accidental changes to that behavior.

## 10. Suggested implementation order

1. Define the frontend `Context Preview` model.
2. Audit the current Tableau context collector behavior and gaps.
3. Add filters and parameters to the preview model.
4. Add selected marks collection with bounded preview data.
5. Add the `MarkSelectionChanged` listener.
6. Add summary data preview collection with strict limits.
7. Add the AI Context Preview panel.
8. Add selected-mark-aware action suggestions.

This order keeps the work incremental and makes the preview panel easier to validate as the collector surface grows.

## 11. Open questions

- Should the preview model be a new type hierarchy or a stricter projection of the existing `DashboardContext`?
- Should selection changes update only the preview state, or also cache a richer collector snapshot for later use?
- What is the correct upper bound for selected marks per worksheet in the preview?
- What is the correct upper bound for summary data preview rows and columns?
- Should `DashboardContextPanel` evolve into the new preview panel, or remain a separate compact summary card?
- Should the MarkSelectionChanged listener watch every worksheet or only the active/selected worksheet?
- Should `/context` remain a narrow workbook-name enrichment endpoint until v0.4, or should it eventually accept the richer preview model?
