# v0.3 Frontend Tableau Context Collector Audit

## 1. Purpose

This document records the completed v0.3.0 "Tableau Context First" work on the frontend side.

The goal of v0.3.0 was to make Tableau context visible before any orchestration work:

- capture the current Tableau state
- normalize it into a preview-friendly model
- keep that preview in sync when worksheet selection changes
- show the preview to the user in a read-only panel
- expose safe action suggestions without triggering AI execution

## 2. Current frontend context flow

The current frontend flow is now:

1. `App` initializes the Tableau extension.
2. `initializeTableauExtension()` builds the initial dashboard snapshot.
3. `getDashboardContext()` collects the live Tableau state.
4. `buildContextPreviewModel()` converts that snapshot into a frontend preview model.
5. `registerMarkSelectionChangedListeners()` listens for worksheet selection changes.
6. On selection changes, the frontend recollects dashboard context and rebuilds the preview model.
7. `AIContextPreviewPanel` renders the preview.
8. `ChatPanel` still sends the original `DashboardContext` to the existing job-based chat flow.
9. `buildContextActionSuggestions()` derives safe UI suggestions from the preview model.
10. Clicking a suggestion only pre-fills the chat input. It does not send a job or call the LLM.

The important boundary is still intact:

- preview state updates happen in the frontend
- chat execution still happens through the existing `/chat-jobs` flow
- AI execution is not triggered by preview updates or suggestions

## 4. Current collected context

The frontend now captures and normalizes the following Tableau context.

### Dashboard / workbook / view

- dashboard name
- workbook name, id, and content URL
- view name and id
- worksheet list and summaries

### Dashboard state

- filters
- parameters
- selected marks
- summary data preview
- data sources
- availability metadata
- warnings
- timestamps for the captured context and preview generation

### Context preview model

The frontend preview model now provides a separate display-oriented structure:

- overview
- worksheets
- filters
- parameters
- selected marks
- summary data preview
- last changed worksheet
- availability and warnings
- metadata
- action suggestions

### Current preview safety controls

- selected marks are bounded with preview limits and truncation metadata
- summary data preview is bounded by max rows and max columns
- missing or unavailable values are represented explicitly
- preview builders avoid mutating the source `DashboardContext`

## 5. Missing or insufficient context

v0.3.0 intentionally does not complete orchestration. The following remain out of scope:

- full Intent Resolver logic
- full Plan Builder logic
- Tool Router execution
- ExecutionEngine implementation
- automatic AI execution on selection changes
- automatic chat job creation from the preview UI
- full AgentCore integration

These are not gaps in v0.3.0. They are the boundary for v0.4.0.

## 6. Recommended v0.3 frontend architecture

v0.3.0 now follows a context-first frontend architecture:

- `initializeTableauExtension()`
  - captures the first live Tableau snapshot
- `getDashboardContext()`
  - collects dashboard state from the Tableau Extension API
- `buildContextPreviewModel()`
  - normalizes the snapshot into a display-oriented preview model
- `registerMarkSelectionChangedListeners()`
  - refreshes the context when worksheet selection changes
- `AIContextPreviewPanel`
  - shows the preview model to the user
- `buildContextActionSuggestions()`
  - derives safe action suggestions from the preview model
- `ChatPanel`
  - keeps the existing chat flow intact

This is the main design lesson from v0.3.0:

- Tableau context comes first
- preview and suggestion UI are separate from execution
- user-driven chat remains the only path that reaches the job pipeline

## 7. Proposed data flow

Current flow:

1. `App` initializes the extension.
2. The frontend collects the current Tableau dashboard state.
3. The preview model is built from that state.
4. `MarkSelectionChanged` updates the collected state and rebuilds the preview.
5. `AIContextPreviewPanel` renders the preview.
6. `buildContextActionSuggestions()` exposes safe UI actions.
7. Clicking a suggestion only pre-fills the chat input.
8. The user still has to send the message manually.
9. `ChatPanel` creates chat jobs only when the user submits a question.

This keeps preview updates and AI execution separate.

## 9. Risks and constraints

- Tableau Extension API availability can differ across Tableau Desktop, Tableau Server, and Tableau Cloud.
- Selected marks can be noisy and large, so preview data must remain bounded.
- Summary data preview can grow quickly if limits are not enforced.
- Selection and worksheet-change events must remain refresh triggers, not execution triggers.
- Extension startup can still fail, so the preview path needs a safe fallback state.
- The chat job pipeline is intentionally unchanged in v0.3.0.

## 10. Suggested implementation order

Completed v0.3 order:

1. Define the frontend context preview model.
2. Collect filters and parameters.
3. Collect selected marks.
4. Add the `MarkSelectionChanged` listener.
5. Add summary data preview collection.
6. Add the AI Context Preview panel.
7. Add selected-mark-aware action suggestions.

The remaining work now belongs to v0.4.0.

## 11. Open questions

- How should preview actions map to structured orchestration intents?
- What is the minimal Intent Resolver contract?
- What is the fixed-plan execution contract?
- How should `agentRunId`, trace, and execution logs be surfaced in the UI?
- Should the selected-mark suggestion eventually invoke a fixed plan instead of pre-filling chat input?
