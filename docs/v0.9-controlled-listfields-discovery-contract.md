# Controlled `listFields` Discovery Contract

## Summary

This document defines the v0.9.0 contract for `tableau.metadata.listFields`
as a controlled, truncated, permission-aware discovery candidate.

The contract is intentionally narrower than `describeDatasource`.
It is designed to protect the app from broad field enumeration, raw MCP output,
underlying-data leakage, and permission-sensitive disclosure.

## Background

v0.8.0 established `describeDatasource` as the first validated hosted metadata
boundary.

`listFields` was evaluated in v0.8.0, but it was deferred because it carries a
higher risk profile:

- larger output volume
- more truncation pressure
- hidden-field and permission-edge risk
- greater chance of exposing field-level details that should stay summarized

v0.9.0 treats `listFields` as a controlled candidate, not as broad Hosted MCP
exposure.

## Relationship to v0.8.0

The v0.8.0 readiness evaluation remains the baseline:

- `listFields` execution is still deferred
- the fake / no-network posture remains the default safety baseline
- stdio fallback remains preserved
- Hosted execution stays optional and gated
- raw MCP payloads, raw transport output, tokens, secrets, and stack traces
  remain out of user-facing output and traces

This contract does not promote `listFields` into a broad metadata surface.
It only defines the safe shape a later execution path must obey.

## Candidate contract

`listFields` may be represented as a candidate only when all of the following
are true:

- the intent is `metadata_discovery`
- the plan state is executable
- the target type is `datasource`
- the datasource identifier and required context are present
- ambiguity has already been resolved
- the request stays within metadata discovery and does not request values,
  row data, underlying data, or arbitrary query execution
- the candidate is handled through the app-level wrapper boundary, not as a raw
  MCP tool surface

If any of those checks fail, the request should remain in clarification,
unsupported, or fallback handling.

## Controlled limits and caps

The contract must stay bounded.

At minimum, the candidate must respect:

- a maximum field count
- a bounded serialized response shape
- safe string handling for field identifiers and names
- a truncation threshold
- an omission threshold
- a conservative default behavior in fake / no-network mode

The exact numeric values may remain implementation-defined if they already exist
elsewhere in the codebase, but the contract must always stay conservative and
must never permit unbounded enumeration.

If a response would exceed the cap, the contract must mark it as truncated and
must not silently expand the output.

## Safe field metadata subset

The only field metadata that may appear in the normalized contract is a safe
summary subset.

Allowed examples:

- field id
- field name
- caption
- role
- data type
- default aggregation
- semantic role
- description, when already present in a safe normalized summary
- hidden flag, when already safe and normalized
- calculated flag, when already safe and normalized
- safe source summary

Prohibited output:

- field values
- row values
- domain values
- sample values
- underlying data
- raw SQL
- arbitrary query text
- raw MCP response
- raw transport response
- tokens
- secrets
- stack traces

## Truncation and omission signals

The contract should be able to express the following safe signals:

- `truncated`
- `returnedFieldCount`
- `totalFieldCountKnown`
- `omittedFieldCount`
- `omissionReasons`
- `truncationReason`
- `limitApplied`
- `safeMessage`

If the total count is unknown, the contract must not invent one.
If omitted counts are unknown, they should remain unknown instead of being
fabricated.

Hidden or restricted fields must not be guessed into the response.

## Permission-aware behavior

The contract must be explicit that metadata visibility may vary with user
permissions and environment.

The normalized output should distinguish:

- fields returned because they are safe and available
- fields omitted because permissions are not verified or are insufficient
- fields omitted because the safe cap was reached
- fake / stdio / hosted transport differences, which must not leak as raw
  transport details in user-facing output

Permission notes should remain generic and safe.

The contract must not expose restricted details simply because a lower-level
transport returned them.

## Validation gates before execution wiring

Before any future execution wiring is considered, the following gates must be
true:

- datasource target is explicit
- required identifier and context are present
- ambiguity is resolved
- a bounded field cap is defined
- truncation behavior is defined
- omission behavior is defined
- permission-aware behavior is defined
- normalized safe output schema is defined
- fake / no-network behavior is preserved
- hosted integration remains optional and gated
- the output does not contain raw MCP data, row data, field values, or write
  operations
- `selected_mark_explanation` behavior remains unchanged

## Non-goals

- broad `listFields` exposure
- arbitrary field enumeration without limits
- raw MCP result exposure
- raw transport result exposure
- field values or row data retrieval
- underlying data access
- write-capable tools
- broad search or Tableau content crawling
- any change to `selected_mark_explanation`
- a general-purpose planning DSL

## Validation strategy

This issue is contract-first.

If only docs and type-level contract work are added, the repository should still
remain no-network by default.

If later code wiring is added, backend lint, typecheck, and tests should be run
with the existing no-network defaults preserved.
