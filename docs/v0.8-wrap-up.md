# v0.8.0 Wrap-up

## 1. Purpose

This document closes out the v0.8.0 Hosted Tableau MCP Metadata Execution milestone.

Its goals are to:

- summarize what v0.8.0 delivered
- make the hosted execution outcome explicit
- record the final status of `describeDatasource` and `listFields`
- capture the preserved fallback and regression posture
- document the remaining limitations and open questions
- prepare the handoff to v0.9.0 Structured Data Discovery and Legacy Chat Reduction

## 2. Milestone summary

v0.8.0 turned the Hosted Tableau MCP plan into a gated, documented, and testable metadata execution path.

The milestone remained deliberately narrow:

- `tableau.metadata.describeDatasource` was promoted as the first hosted metadata tool
- `tableau.metadata.listFields` was evaluated but deferred
- Hosted execution stayed explicit and opt-in
- fake / no-network execution remained first-class
- stdio fallback remained available where configured
- auth, site-settings, transport, error, and trace boundaries stayed safe and summary-first

In short, v0.8.0 delivered the first hosted metadata slice without broadening Tableau MCP access beyond the read-only wrapper boundary.

## 4. Architecture outcome

The milestone preserves the legacy free-form chat path:

```text
free-form chat
-> ChatService / runLightweightAgentLoop
-> legacy Tableau MCP usage
-> stdio Tableau MCP
```

The hosted metadata architecture is now available as a gated execution path:

```text
ToolRegistry
-> tableau.metadata.describeDatasource
-> metadata precondition evaluator
-> transport config resolver
-> Hosted MCP auth context adapter
-> HostedTableauMcpTransport
-> Hosted MCP metadata error normalization
-> metadata output normalization
-> hosted metadata trace events
-> safe normalized result
```

The safe fallback architecture remains intact:

```text
ToolRegistry
-> metadata handler
-> transport-aware execution boundary
-> fake / no-network transport or stdio fallback
-> normalized metadata result
```

v0.8.0 therefore ends with a layered architecture:

- legacy chat remains unchanged
- metadata execution is transport-aware
- hosted execution is explicit and gated
- fallback paths remain deterministic and first-class

## 5. Hosted execution status

Hosted execution is enabled only when the explicit configuration and safety checks allow it.

Current status:

- `describeDatasource` is the only hosted metadata execution target in v0.8.0
- hosted execution is feature-flagged and not selected implicitly
- no-network and CI defaults still prefer fake execution
- hosted integration remains optional and gated
- live Hosted MCP behavior is documented, not broadly exposed

This keeps the milestone narrow and prevents unplanned scope expansion.

## 6. describeDatasource status

`tableau.metadata.describeDatasource` is the v0.8.0 hosted success path.

What it demonstrates:

- the tool can flow through the hosted transport boundary when gated on
- the auth adapter can provide safe summaries without exposing raw tokens
- the execution boundary can preserve normalization and trace behavior
- fallback behavior remains available when hosted execution is not configured

What it does not do:

- it does not expose raw MCP tools
- it does not enable arbitrary query execution
- it does not access underlying data
- it does not promote `listFields` to hosted execution

## 7. listFields readiness status

`tableau.metadata.listFields` remains deferred.

The readiness review concluded that the hosted promotion risk is still higher than
for `describeDatasource`, mainly because of:

- output-size pressure
- truncation and omission risk
- hidden-field and permission variance
- schema-mapping sensitivity
- trace payload noise risk

The practical milestone result is simple: v0.8.0 does not promote hosted
`listFields` execution.

## 8. Feature flag and configuration status

Hosted execution remains explicit opt-in.

Current posture:

- default behavior is still fake / no-network
- hosted execution does not happen automatically
- missing endpoint or missing auth can be handled safely
- CI does not require Hosted MCP by default
- hosted integration tests stay optional and gated

This is the intended posture for a safe hosted rollout.

## 9. Auth, user-context, and site-settings status

The hosted auth boundary remains safe and intentionally incomplete.

Current posture:

- the auth adapter uses summaries rather than raw tokens
- tokenReference is preserved as a boundary, not resolved into a secret value
- user-context mapping is summary-first
- missing, expired, unknown, and not-configured auth states remain safe
- site-settings and admin assumptions are documented, not hardcoded as secrets
- no raw token, secret, or stack trace is allowed in result or trace surfaces

## 10. Hosted transport status

The Hosted Tableau MCP transport remains a safe boundary, not a production client.

Current posture:

- constructor-based configuration and dependency injection are supported
- `not_configured` and safe error placeholders exist
- unsupported and timeout placeholders exist
- no real Hosted MCP network connection is required by default
- no token logging is allowed

This gives the hosted path a contract without forcing live connectivity.

## 11. Error normalization status

Hosted MCP errors are normalized into safe summaries.

The result is that hosted failures can be reported without exposing raw MCP
payloads, raw stack traces, or credential material.

The normalized error model covers the hosted failure modes needed for v0.8.0:

- auth required / expired
- permission denied
- site settings disabled
- transport not configured
- network error
- timeout
- protocol error
- remote server error
- unknown error

## 12. Trace coverage status

Hosted metadata execution is traceable in summary form.

The trace contract records:

- hosted vs fake vs stdio distinction
- request and correlation identifiers
- duration information
- normalized status and error code
- warning count
- fallback information when applicable

The trace contract does not expose:

- raw MCP payloads
- raw transport results
- tokens or secrets
- raw stack traces
- UI trace panel or persistence behavior

## 13. Fake / no-network and stdio fallback status

The fallback posture remains intact.

Current status:

- fake / no-network metadata tests remain first-class
- stdio fallback remains available where configured
- default CI still does not require Hosted MCP
- `selected_mark_explanation` remains unchanged
- the free-form chat legacy path remains unchanged

This is one of the most important v0.8.0 outcomes because it keeps the repo
safe for local development and CI.

## 15. Risks and open questions

The main remaining unknowns are still the same ones documented in the earlier
milestone docs:

- official Hosted MCP endpoint and transport details
- Hosted site-settings and admin-enablement specifics
- OAuth / DCR / static registration details
- tokenReference resolution ownership
- real Hosted `describeDatasource` behavior in all environments
- `listFields` hosted readiness and promotion timing
- stdio fallback retirement timing
- v0.9 metadata discovery scope

These are now explicit follow-up questions rather than hidden assumptions.

## 16. Intentionally not done

v0.8.0 intentionally did not do the following:

- raw MCP tool exposure
- arbitrary query execution
- underlying data access
- write-capable tools
- `listFields` Hosted execution
- free-form chat Hosted migration
- ChatService replacement
- runLightweightAgentLoop replacement
- OAuth full implementation
- token storage or refresh implementation
- production Hosted MCP migration
- UI trace panel
- trace persistence
- AgentCore implementation
- LLM ResponseComposer implementation

## 17. v0.9.0 handoff

v0.9.0 is `Structured Data Discovery and Legacy Chat Reduction`.

The handoff should carry forward:

- the hosted `describeDatasource` result
- hosted error normalization behavior
- hosted trace coverage behavior
- fake / stdio fallback strategy
- the `listFields` readiness recommendation
- remaining auth / site-settings / permission issues

v0.9.0 should not begin by broadly exposing Hosted MCP tools. It should start
from structured metadata discovery intent design and treat the v0.8.0
`describeDatasource` path as the first validated metadata execution boundary.
