# Hosted MCP Integration Test Gating

## 1. Purpose

This document defines how Hosted MCP integration tests are gated for v0.8.0.

The goals are to:

- keep default CI no-network
- keep Hosted MCP tests explicit opt-in
- define skip and failure behavior before live Hosted work starts
- document the local manual verification flow for developers
- make sure #159 and later code-bearing issues can add hosted coverage without breaking default validation

## 2. Background

v0.7.0 established the no-network foundation that v0.8.0 builds on:

- fake / no-network metadata handlers exist
- transport configuration and transport-aware execution boundaries exist
- metadata normalization and trace event coverage exist
- `selected_mark_explanation` and the legacy free-form chat path remain in place

Issue #157 verified the hosted endpoint and site-settings prerequisites as far as official sources allowed.

v0.8.0 then moves toward Hosted MCP metadata execution, starting with `tableau.metadata.describeDatasource`, but that Hosted path still depends on endpoint, auth, site settings, secrets, and network assumptions that must not leak into default CI.

## 3. Current test baseline

Current backend and repository test baseline:

- backend lint: `npm --prefix backend run lint`
- backend typecheck: `npm --prefix backend run typecheck`
- backend tests: `npm --prefix backend run test`
- root unit test entrypoint: `npm run test`
- root CI entrypoint: `npm run ci`

Current default CI is no-network:

- lint
- typecheck
- backend and frontend unit tests / coverage
- build
- frontend e2e

No Hosted MCP-specific CI job exists yet.

Existing no-network and regression coverage includes:

- fake transport tests
- transport config resolver tests
- metadata precondition tests
- metadata output normalization tests
- trace event tests
- ToolRegistry registration tests
- `selected_mark_explanation` regression tests
- legacy free-form chat non-regression tests where present

## 4. Gating principles

The gating model for Hosted MCP tests should follow these principles:

- Hosted integration tests are opt-in
- default CI remains no-network
- missing secrets means skip in local/manual mode
- Hosted MCP unavailable means skip unless the run explicitly requested Hosted integration
- fake/no-network tests remain mandatory
- stdio fallback tests remain mandatory where applicable
- hosted tests must never print tokens or secrets
- hosted tests must never expose raw MCP output
- hosted tests should cover read-only metadata only
- `describeDatasource` is the first hosted candidate
- no arbitrary query execution
- no underlying data access
- no write tools

## 5. Default no-network test path

The default validation path should continue to run:

- unit tests
- contract tests
- fake transport tests
- transport config resolver tests
- metadata precondition tests
- metadata output normalization tests
- trace event tests
- ToolRegistry registration tests
- `selected_mark_explanation` regression tests
- free-form chat legacy non-regression tests, if present

The default path should not require:

- Hosted MCP access
- OAuth login
- token refresh
- Tableau Cloud access
- secrets-dependent tests
- live endpoint checks

## 6. Hosted integration test classification

Hosted integration tests should be classified as optional, gated integration tests.

| Test class | Runs by default | Requires secrets | Requires network | Purpose |
| --- | --- | --- | --- | --- |
| unit / contract / no-network | yes | no | no | keep the hosted contract safe without live dependency |
| hosted integration, local manual | no | maybe | yes | allow a developer to verify a live environment explicitly |
| hosted integration, CI opt-in | no | yes | yes | allow a separate hosted job when secrets and environment are ready |

## 7. Environment flags and markers

Recommended opt-in flag:

- `TABLEAU_MCP_RUN_HOSTED_INTEGRATION_TESTS=true`

Recommended mode flag:

- `TABLEAU_MCP_HOSTED_INTEGRATION_TEST_MODE=off|manual|ci`

Recommended future config candidates:

- `TABLEAU_MCP_HOSTED_ENDPOINT`
- `TABLEAU_MCP_HOSTED_AUTH_MODE`
- `TABLEAU_MCP_HOSTED_TOKEN_REFERENCE`
- `TABLEAU_MCP_HOSTED_SITE_ID`
- `TABLEAU_MCP_HOSTED_SITE_CONTENT_URL`
- `TABLEAU_MCP_HOSTED_TEST_DATASOURCE_ID`
- `TABLEAU_MCP_HOSTED_TEST_DATASOURCE_NAME`

Recommended test markers:

- filename suffix `.hosted.integration.test.ts`
- filename suffix `.hosted.int.test.ts`
- test name prefix `[hosted-mcp]`
- describe helper `describeHostedMcpIntegration`

Recommended future scripts:

- `test:hosted:mcp`
- `test:integration:hosted`

Default behavior:

- flag missing means skip
- flag false means skip
- flag true with required config means eligible to run

## 8. Secret requirements

Secrets and sensitive values should be separated from ordinary config.

| Variable | Required for default CI | Required for hosted integration | Sensitive | Notes |
| --- | --- | --- | --- | --- |
| `TABLEAU_MCP_RUN_HOSTED_INTEGRATION_TESTS` | no | yes | no | explicit opt-in switch |
| `TABLEAU_MCP_HOSTED_ENDPOINT` | no | likely | no | future required config candidate |
| `TABLEAU_MCP_HOSTED_AUTH_MODE` | no | likely | no | future required config candidate |
| `TABLEAU_MCP_HOSTED_TOKEN_REFERENCE` | no | likely | yes | preserve tokenReference boundary |
| `TABLEAU_MCP_HOSTED_SITE_ID` | no | maybe | no | candidate if site id is required |
| `TABLEAU_MCP_HOSTED_SITE_CONTENT_URL` | no | maybe | no | candidate if contentUrl is required |
| `TABLEAU_MCP_HOSTED_TEST_DATASOURCE_ID` | no | likely | no | preferred test fixture if available |
| `TABLEAU_MCP_HOSTED_TEST_DATASOURCE_NAME` | no | maybe | no | fallback candidate |
| `TABLEAU_CONNECTED_APP_CLIENT_ID` | no | maybe | no | only if real auth setup needs it |
| `TABLEAU_CONNECTED_APP_SECRET_ID` | no | maybe | yes | do not log or snapshot |
| `TABLEAU_CONNECTED_APP_SECRET_VALUE` | no | maybe | yes | do not log or snapshot |

Rules:

- default CI requires no secrets
- local `.env` files must not be committed
- docs must not contain real secret values
- logs must not include raw tokens, refresh tokens, or secrets
- snapshots must not capture secrets

## 9. Local manual verification flow

Suggested local flow:

```bash
# 1. Run the default test path first.
npm run test

# 2. Export the explicit hosted flag and required values.
export TABLEAU_MCP_RUN_HOSTED_INTEGRATION_TESTS=true
export TABLEAU_MCP_HOSTED_INTEGRATION_TEST_MODE=manual
export TABLEAU_MCP_HOSTED_ENDPOINT="..."
export TABLEAU_MCP_HOSTED_SITE_CONTENT_URL="..."

# 3. Run the future hosted integration command.
npm run test:hosted:mcp --workspace backend
```

If the future script does not exist yet, treat the command above as the recommended target command for #159 and later issues.

## 10. CI opt-in strategy

Hosted integration tests should not be added to the default CI job.

Recommended CI policy:

- keep default CI no-network
- add a separate opt-in job only when Hosted execution is intentionally being verified
- use workflow_dispatch or a dedicated protected job rather than the default PR path
- require Hosted secrets only in the opt-in job
- skip on forks
- skip when secrets are missing in a local/manual setting
- fail fast in an explicit hosted CI job if config is invalid
- sanitize logs and artifacts

## 11. Skip behavior

Recommended skip model:

| Condition | Default CI | Local manual | Hosted opt-in CI |
| --- | --- | --- | --- |
| flag missing | skip | skip | skip or not scheduled |
| flag false | skip | skip | skip |
| endpoint missing | skip | skip with reason | fail fast |
| auth missing | skip | skip with reason | fail fast |
| site settings unknown | skip | warn and skip | fail or skip based on mode |
| Hosted unavailable | not run | fail only if explicitly requested | fail as hosted failure |

## 12. Failure handling

Hosted test failures should be safe and actionable:

- config missing
- endpoint unreachable
- auth required
- auth expired
- site settings disabled
- permission denied
- timeout
- protocol error
- unexpected Hosted MCP response
- output schema mismatch
- trace event mismatch

Behavior rules:

- default CI should not hit these failures
- local manual runs should return a clear safe message
- opt-in hosted CI should fail explicitly
- tokens, secrets, raw MCP output, and raw stack traces must not be printed

## 13. Test placement and naming

Recommended placement:

- `backend/test/**/*.hosted.integration.test.ts`
- `backend/test/hosted/**/*.test.ts`

Naming rules:

- hosted integration tests should be easy to distinguish from normal tests
- if the default glob includes them, runtime skip is required
- if the default glob excludes them, a dedicated npm script is required

## 14. Relationship with fake/no-network and stdio fallback tests

The hosted gating model must not weaken the existing baseline:

- fake/no-network tests remain required
- hosted tests remain optional
- fake transport tests stay in place after Hosted work lands
- stdio fallback tests stay in place after Hosted work lands
- default CI continues to verify the contract without live Hosted dependency
- `selected_mark_explanation` regression tests remain mandatory
- free-form chat legacy tests remain mandatory where present

## 15. Relationship with #159, #160, and #161

Implications for later issues:

- #159 skeleton tests should be no-network by default
- #159 should cover `not_configured` paths without secrets
- #160 auth adapter tests should use fake inputs and no-network contracts
- #160 should not require real OAuth in default CI
- #161 hosted `describeDatasource` tests should remain gated
- #161 should keep the fake and stdio fallback paths intact

## 16. Safety and governance boundaries

Hosted test gating must preserve the same governance boundary as v0.7.0:

- no arbitrary query
- no underlying data
- no write tools
- read-only metadata only
- no raw MCP output in logs
- no tokens or secrets in logs
- no raw stack traces in test output
- no snapshots with secrets
- no default CI dependency on Hosted MCP
- `selected_mark_explanation` unchanged
- free-form chat legacy unchanged

## 17. Implementation recommendations

Recommended follow-up steps for #159 and later:

- add a small pure gating helper before live tests
- use dependency injection for config
- prefer runtime skip with a clear reason in local manual mode
- prefer fail-fast config validation in explicit hosted CI mode
- keep the hosted test command separate
- do not add Hosted secrets to the default workflow
- add log redaction before any live Hosted test
- add output schema contract tests before any live smoke test
