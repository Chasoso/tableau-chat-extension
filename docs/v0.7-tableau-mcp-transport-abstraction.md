# Tableau MCP Transport Abstraction

## 1. Purpose

This document defines the transport boundary that will let Tableau MCP execution move across `stdio`, hosted / remote, and fake / no-network paths without exposing transport internals to the Tool Layer or higher layers.

Its purpose is to:

- define a normalized transport contract before implementation begins
- keep Tool Layer concerns transport-agnostic
- make stdio, hosted / remote, and fake behaviors comparable
- define safe result, error, warning, trace, and timing shapes
- provide the contract basis for #130, #135, and #136

This is a design document only. No production transport implementation is introduced here.

## 2. Background

The repository currently uses stdio-based Tableau MCP as the active path, while the v0.7 roadmap assumes hosted / remote Tableau MCP may become the future target.

At the same time:

- the legacy chat path remains in place
- the Tool Layer already has its own definition, precondition, router, execution, and trace boundaries
- selected-mark orchestration is intentionally separate from the future metadata transport work
- hosted readiness and auth boundaries are still being documented in earlier v0.7 issues

That means transport should be treated as a first-class contract, not as an implementation detail hidden inside a tool handler.

## 3. Transport boundary goals

The transport boundary should:

- hide whether execution is stdio, hosted, remote, or fake
- normalize success, partial, timeout, cancelled, and failure outcomes
- keep tokens, secrets, process details, and network details out of user-facing results
- keep ToolDefinition and ToolExecutionWrapper transport-agnostic
- support deterministic no-network behavior for tests
- allow later hosted adoption without rewriting tool contracts

## 4. Non-goals

This document does not define:

- a production Hosted MCP implementation
- a production stdio transport implementation
- a production remote transport implementation
- OAuth flow implementation
- token persistence or refresh implementation
- ToolRegistry registration
- metadata tool execution behavior
- broad legacy client refactoring
- real network integration

## 5. Transport contract

The contract should model a Tableau MCP transport as a small interface with a normalized request and result shape.

Recommended shape:

```ts
export interface TableauMcpTransport {
  readonly kind: TableauMcpTransportKind;
  readonly name?: string;
  call(request: TableauMcpTransportRequest): Promise<TableauMcpTransportResult>;
}
```

The contract should also remain compatible with an `execute` or `invoke` naming style if the repository later prefers that naming convention.

Contract rules:

- callers should not know whether the transport is stdio, hosted, remote, or fake
- transports should not expose raw process state or network details to higher layers
- tokens and secrets should not be required in the top-level request type
- results must be JSON-safe
- raw exceptions should be normalized before leaving the transport boundary

## 6. Normalized request shape

The request shape should contain the minimum tool-call context needed for transport execution without embedding secrets.

Recommended shape:

```ts
export type TableauMcpTransportKind =
  | "stdio"
  | "hosted"
  | "remote"
  | "fake"
  | "unknown";

export type TableauMcpUserContextSummary = {
  userId?: string;
  tableauUserId?: string;
  email?: string;
  siteId?: string;
  siteName?: string;
  source?: "cognito" | "tableau" | "unknown";
};

export type TableauMcpAuthContextSummary = {
  mode: "none" | "direct_trust" | "oauth_delegated" | "token_reference" | "unknown";
  tokenReference?: string;
  scopes?: string[];
  expiresAt?: string;
  metadata?: Record<string, unknown>;
};

export type TableauMcpTransportRequest = {
  requestId: string;
  toolName: string;
  input: Record<string, unknown>;
  timeoutMs?: number;
  correlationId?: string;
  agentRunId?: string;
  userContext?: TableauMcpUserContextSummary;
  authContext?: TableauMcpAuthContextSummary;
  trace?: TableauMcpTransportTraceOptions;
  metadata?: Record<string, unknown>;
};
```

Request rules:

- do not require raw access tokens
- do not require raw refresh tokens
- do not require raw headers or secrets
- do not pass raw Tableau Extension context directly
- do not assume arbitrary query execution
- keep the shape usable for read-only metadata tools first
- keep the shape JSON-safe

## 7. Normalized result shape

The result shape should normalize success, partial success, and failure across all transports.

Recommended shape:

```ts
export type TableauMcpTransportStatus =
  | "success"
  | "partial"
  | "failed"
  | "timeout"
  | "cancelled"
  | "unsupported"
  | "not_configured";

export type TableauMcpTransportWarning = {
  code?: string;
  message: string;
  source?: string;
  metadata?: Record<string, unknown>;
};

export type TableauMcpTransportTiming = {
  startedAt?: string;
  completedAt?: string;
  durationMs?: number;
  timeoutMs?: number;
  timedOut?: boolean;
};

export type TableauMcpTransportResult = {
  requestId: string;
  transportKind: TableauMcpTransportKind;
  status: TableauMcpTransportStatus;
  toolName: string;
  data?: unknown;
  warnings?: TableauMcpTransportWarning[];
  error?: TableauMcpTransportError;
  trace?: TableauMcpTransportTraceMetadata;
  timing?: TableauMcpTransportTiming;
  metadata?: Record<string, unknown>;
};
```

Result rules:

- `data` should be bounded and normalization-friendly
- `data` should not imply raw MCP payload exposure to higher layers
- `error` should be normalized, not a raw exception object
- stack traces should not be included
- tokens and secrets should not be included
- the result should be serializable with `JSON.stringify`

## 8. Normalized error and warning shape

The error shape should be specific enough for later routing, fallback, and UX decisions without leaking implementation details.

Recommended shape:

```ts
export type TableauMcpTransportErrorCode =
  | "TRANSPORT_NOT_CONFIGURED"
  | "UNSUPPORTED_TRANSPORT"
  | "AUTH_REQUIRED"
  | "AUTH_EXPIRED"
  | "PERMISSION_DENIED"
  | "SITE_SETTINGS_DISABLED"
  | "NETWORK_ERROR"
  | "TIMEOUT"
  | "MCP_PROTOCOL_ERROR"
  | "TOOL_NOT_FOUND"
  | "INVALID_TOOL_INPUT"
  | "REMOTE_SERVER_ERROR"
  | "STDIO_PROCESS_ERROR"
  | "UNKNOWN_ERROR";

export type TableauMcpTransportError = {
  code: TableauMcpTransportErrorCode;
  message: string;
  retryable?: boolean;
  userActionRequired?: boolean;
  source?: string;
  metadata?: Record<string, unknown>;
};
```

Recommended warnings:

- fake transport used
- hosted transport not configured
- stdio fallback used
- output truncated
- auth context missing
- site settings unknown
- metadata omitted

Warning rules:

- warnings should be safe to surface in user-facing summaries
- warnings should not contain secrets or raw stack traces
- warnings should be suitable for no-network and partial-result paths

## 9. Trace metadata boundary

Trace metadata should be enough for debugging and correlation without exposing raw transport internals.

Recommended shape:

```ts
export type TableauMcpTransportTraceMetadata = {
  correlationId?: string;
  agentRunId?: string;
  transportEventId?: string;
  startedAt?: string;
  completedAt?: string;
  durationMs?: number;
  transportKind?: TableauMcpTransportKind;
  toolName?: string;
  attemptCount?: number;
  remoteTraceId?: string;
  stdioProcessIdAvailable?: boolean;
  hostedSessionId?: string;
  metadata?: Record<string, unknown>;
};

export type TableauMcpTransportTraceOptions = {
  includeTiming?: boolean;
  includeMetadata?: boolean;
};
```

Trace rules:

- do not place raw MCP payloads in trace metadata
- do not place tokens or secrets in trace metadata
- keep process IDs and hosted session hints optional and minimal
- keep the trace shape compatible with ToolExecutionWrapper / orchestration trace correlation

## 10. Timing and timeout boundary

Transport timing should be normalized independently from the calling layer.

Recommended shape:

```ts
export type TableauMcpTransportTiming = {
  startedAt?: string;
  completedAt?: string;
  durationMs?: number;
  timeoutMs?: number;
  timedOut?: boolean;
};
```

Timing rules:

- timeout should be explicit on the normalized result
- transport timing should not be confused with overall request timing
- cancellation should be distinguishable from timeout
- any transport-specific duration breakdown should remain optional

## 11. stdio transport candidate

The stdio candidate represents the current local / self-hosted process-backed Tableau MCP path.

Documented characteristics:

- local process startup / shutdown are part of the transport behavior
- stderr-based diagnostics may exist
- command / args / env handling are implementation details, not contract fields
- it can remain the default fallback while hosted readiness is uncertain

Transport implications:

- should normalize process startup failures into `STDIO_PROCESS_ERROR`
- should normalize unsupported / misconfigured paths into `UNSUPPORTED_TRANSPORT` or `TRANSPORT_NOT_CONFIGURED`
- should not leak the raw child process model to higher layers

## 12. Hosted / remote transport candidate

The hosted / remote candidate represents a future Tableau Cloud transport backed by a hosted MCP endpoint.

Documented characteristics:

- remote connectivity will likely depend on auth and site settings
- session lifecycle is not yet defined in this repository
- timeout and retry behavior may differ from stdio
- token boundary and user-context boundary are likely more important than in the current stdio path

Transport implications:

- should normalize remote connectivity failures into `NETWORK_ERROR`, `TIMEOUT`, `AUTH_REQUIRED`, `AUTH_EXPIRED`, or `REMOTE_SERVER_ERROR`
- should not expose network internals or raw tokens
- should allow later user-context and auth summaries without changing the contract

## 13. Fake / no-network transport candidate

The fake transport candidate exists so tests and local development can run without network access or hosted dependencies.

Documented characteristics:

- it never opens a network connection
- it does not require tokens or secrets
- it returns deterministic results
- it can simulate success, failure, timeout, unsupported, and not-configured outcomes
- it should be clearly marked as fake in the result

Transport implications:

- should always set `transportKind: "fake"`
- should be safe for unit and contract tests
- should not return real Tableau data
- should not be treated as a production execution mode

## 14. Relationship with Tool Layer

ToolDefinition / ToolRegistry / ToolExecutionWrapper should not know transport details.
Tableau MCP metadata tool handlers should construct transport requests.
Transport results should be normalized before they become user-facing output.

Recommended division of responsibilities:

| Layer | Responsibility |
| --- | --- |
| ToolDefinition | tool name / schema / safety / capability |
| ToolPrecondition | auth / datasource / ambiguity / policy checks |
| ToolExecutionWrapper | timeout / execution trace / handler call |
| Tableau MCP metadata handler | tool input → transport request |
| TableauMcpTransport | stdio / hosted / fake execution |
| Output normalization | transport result → safe metadata result |
| ResponseComposer | safe metadata result → user-facing response |

This keeps transport concerns out of the wider orchestration model and preserves the existing selected-mark and legacy chat boundaries.

## 15. Relationship with auth and user context

The transport contract should support auth and user-context summaries without becoming the auth implementation.

Rules:

- transport requests can carry user-context summaries
- transport requests can carry auth-context summaries
- raw token values should not be required at the contract boundary
- token refresh and token storage belong in later auth work, not in this issue
- hosted OAuth implementation belongs to #131
- current direct-trust / stdio assumptions belong to #127 and #128
- permission enforcement remains relevant both in ToolPrecondition and in Tableau / hosted-side authorization

## 16. Migration implications

Defining the transport boundary first makes the later migration safer:

- #130 can choose a safe default and explicit feature-flag behavior
- #135 can attach fake handlers without leaking transport internals into Tool Layer
- #136 can execute metadata tools through the transport boundary without changing tool definitions
- stdio fallback can remain available while hosted readiness is still uncertain
- hosted adoption can happen without rewriting higher-level contracts

## 17. Recommendations for #130, #135, and #136

### For #130 Define stdio vs hosted transport configuration strategy

- make the default safe, with stdio or fake fallback as the baseline
- gate hosted mode behind an explicit config switch
- keep CI independent from hosted availability
- fail closed on misconfiguration
- avoid exposing missing-secret details to users

### For #135 Register metadata tools with fake no-network handlers

- use the normalized transport contract so the fake path can stand in for hosted / stdio later
- keep fake results deterministic and JSON-safe
- make sure the fake path never requires network access

### For #136 Add transport-aware metadata tool execution boundary

- call transport through a thin boundary from metadata tool handlers
- normalize results and errors before output composition
- preserve Tool Layer transport-agnostic behavior
- keep timing and trace correlation explicit but minimal

## 18. Open questions

- Should `hosted` and `remote` remain distinct kinds in later code, or be merged at implementation time?
- Which layer should own default timeout policy selection?
- Should fake transport support scripted scenarios or only fixed fixtures?
- Should the contract include a bounded raw-result field for internal debugging, or keep raw payloads fully out of the shape?
- What correlation fields will be most useful once hosted trace data exists?
- Should transport-level retry be explicit in the contract, or remain a policy at a higher layer?
