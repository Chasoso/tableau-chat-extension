# LLM ResponseComposer Fallback Regressions

## Purpose

Issue #213 adds regression coverage for the guarded LLM ResponseComposer prototype introduced in #212.

The goal is to verify that unsafe, failed, timed out, too-long, or malformed composition always falls back to the deterministic composer.

## Covered regressions

- disabled and not-configured paths
- timeout and provider error paths
- invalid and unsafe output paths
- missing required limitation notes
- output-too-long paths
- metadata_discovery safety and fallback behavior
- selected_mark_explanation protection
- legacy free-form chat fallback preservation
- trace-safe fallback reasons

## Safety boundaries

- no raw MCP tool exposure
- no raw MCP output exposure
- no raw transport output exposure
- no arbitrary query execution
- no underlying data access
- no field values / row data retrieval
- no write-capable tools
- no tokens / secrets / stack traces in user-facing output or traces
- default tests remain no-network
- hosted / real LLM integration remains optional or gated
- `selected_mark_explanation` remains protected
- legacy free-form chat fallback remains available

## Relation to earlier issues

- #210 defined the safety boundary.
- #211 defined the response material contract.
- #212 added the guarded prototype.
- #213 protects the fallback behavior with regression tests.

## Out of scope

- real LLM integration
- production LLM provider setup
- new execution features
- Exploration Session
- AgentCore
