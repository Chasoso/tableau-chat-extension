# AgentCoreRunner Design

## 1. Purpose

This document defines a safe design for a future `AgentCoreRunner` so it can be compared against `LambdaAgentRunner` without changing the current runtime.

The goal is not to migrate `selected_mark_explanation` yet. The goal is to prepare a runner design that can later be implemented, measured, and rolled back safely.

## 2. Background

v0.6.0 is the AgentCore spike phase.

The current state is:

- `AgentRunner` is defined
- `LambdaAgentRunner` already wraps the current `selected_mark_explanation` path
- `selected_mark_explanation` is routed through the wrapper
- the path is deterministic, lightweight, and summary-focused
- AgentCore fit was assessed as conditional rather than clearly strong

That makes the next step a design exercise, not an implementation exercise.

## 3. Design goals

- keep `AgentCoreRunner` comparable with `LambdaAgentRunner`
- preserve the same `AgentRunInput` and `AgentRunResult` contract
- keep `selected_mark_explanation` deterministic
- make observability and runtime behavior comparable
- keep AgentCore-specific fields optional
- avoid coupling the design to free-form chat

## 4. Non-goals

- AgentCore implementation
- AgentCoreRunner implementation
- runtime migration
- selected_mark_explanation route changes
- LambdaAgentRunner behavior changes
- free-form chat migration
- Tableau MCP migration
- LLM ResponseComposer implementation

## 5. AgentCoreRunner concept

`AgentCoreRunner` is a future implementation of `AgentRunner` that accepts `AgentRunInput` and returns `AgentRunResult`.

It is a comparison target for:

- observability
- runtime behavior
- timeout handling
- failure handling
- cost visibility
- session readiness

It is not a promise to replace the current custom orchestration.

## 6. Runtime boundary options

### Option A: Thin AgentCoreRunner

```text
selected mark action
-> AgentCoreRunner
-> AgentCore runtime
-> custom structured orchestration remains in backend code
-> AgentRunResult normalization
```

Pros:

- preserves current Tool Layer and response material model
- easiest to compare with LambdaAgentRunner
- lower migration risk
- keeps deterministic behavior mostly unchanged

Cons:

- may underuse AgentCore capabilities
- still leaves most business logic custom
- may not simplify orchestration much

### Option B: AgentCore-managed execution

```text
selected mark action
-> AgentCoreRunner
-> AgentCore runtime
-> AgentCore tools / gateway
-> AgentRunResult normalization
```

Pros:

- stronger use of managed runtime features
- could improve platform-native observability

Cons:

- higher migration risk
- more overlap with current Tool Layer
- more difficult to debug locally
- probably too broad for the first spike

### Option C: Deferred AgentCoreRunner

```text
selected mark action
-> LambdaAgentRunner only for now
-> AgentCoreRunner remains design/stub candidate
```

Pros:

- safest for the current deterministic path
- gives more time to build comparison metrics

Cons:

- delays AgentCore evaluation
- postpones the runtime comparison work

### Recommendation

Recommend Option A: Thin AgentCoreRunner.

Reason:

- `selected_mark_explanation` is deterministic and lightweight
- the point of the spike is observability and runtime comparison, not answer quality
- the current Tool Layer should remain intact
- a thin boundary keeps the design comparable to `LambdaAgentRunner`

## 7. Recommended design

`AgentCoreRunner` should be a thin runtime wrapper that:

- accepts `AgentRunInput`
- normalizes it into an AgentCore-compatible runtime request
- delegates execution to AgentCore runtime primitives
- maps runtime output back into `AgentRunResult`
- keeps the current structured orchestration concepts visible in the output

This design intentionally leaves the current deterministic orchestration logic custom unless later issues decide otherwise.

## 8. Input normalization strategy

`AgentRunInput` should not be forwarded to AgentCore verbatim.

It should be normalized into an AgentCore runtime request that includes:

- `agentRunId`
- `runMode`
- `requestedIntent`
- `actionId`
- selected marks summary
- summaryDataPreview summary
- filters summary
- parameters summary
- tool policy summary
- model policy summary
- budget summary
- trace / correlation identifiers
- locale
- metadata that is safe to expose

It should not include:

- raw selected marks
- full summary rows
- secrets
- auth headers
- large raw payloads

If AgentCore needs session or run identifiers, they should be derived from `agentRunId` and stored as optional metadata rather than replacing the current contract.

## 9. Output normalization strategy

AgentCore output should be mapped back to `AgentRunResult` using the same comparison shape as `LambdaAgentRunner`.

Mapping should include:

- AgentCore run/session identifiers in `observability`
- runtime status mapped to `AgentRunStatus`
- final user-facing message mapped to `finalMessage` or `response`
- warnings mapped to `warnings`
- errors mapped to safe structured errors
- fallback or unsupported states mapped to `fallbackReason`
- trace summaries mapped to `trace`
- budget and duration summaries mapped to `budgetUsage`

The result should remain JSON-safe and should not expose raw tool payloads or raw AgentCore trace blobs.

## 10. Observability mapping

| AgentRunResult observability field | LambdaAgentRunner | AgentCoreRunner candidate |
| --- | --- | --- |
| startedAt | from the current runner clock | from AgentCore run start time or runner clock |
| completedAt | from the current runner clock | from AgentCore run completion time or runner clock |
| durationMs | computed locally | computed locally and/or confirmed by AgentCore |
| traceId | local trace correlation | AgentCore trace or correlation identifier |
| correlationId | request / runner correlation | AgentCore correlation identifier |
| cloudProviderRunId | not usually present | optional AgentCore run identifier |
| agentCoreSessionId | not present | optional session identifier |
| agentCoreTraceId | not present | optional trace identifier |
| logGroupName | optional local deployment value | optional cloud runtime value |
| logStreamName | optional local deployment value | optional cloud runtime value |
| metrics.latencyMs | local duration summary | AgentCore runtime metric summary |
| metrics.modelCalls | current runner summary | AgentCore runtime metric summary |
| metrics.toolCalls | current runner summary | AgentCore runtime metric summary |
| metrics.timeoutCount | current runner summary | AgentCore runtime metric summary |
| metrics.retryCount | current runner summary | AgentCore runtime metric summary |

Comparison should focus on:

- latency
- failure rate
- timeout count
- trace completeness
- tool execution visibility
- debugging effort
- cost estimate
- local reproduction difficulty

## 11. Trace comparison strategy

Trace comparison should keep the existing internal trace schema intact.

AgentCore-specific trace details should be summarized in:

- `observability`
- `metadata`
- trace summary fields

Do not expose raw AgentCore trace payloads to user-facing responses.

The comparison baseline should include:

- event count
- first event type
- last event type
- hasErrors
- hasToolExecution
- fallback reason
- correlationId
- agentRunId

`#106` should define the comparison metrics; `#105` only defines the mapping strategy.

## 12. selected_mark_explanation comparison fit

`selected_mark_explanation` is a strong comparison candidate because:

- it is deterministic
- it is lightweight
- it already uses a fixed plan
- it already has a stable custom Tool Layer
- it already has clear trace and fallback behavior

For the initial comparison, the recommendation is:

- no LLM
- no external tool migration
- no Tableau MCP migration
- keep the current deterministic path intact

That makes the comparison about runtime wrapper behavior, not about answer quality.

## 13. Configuration and safety

`AgentCoreRunner` should be behind safe configuration controls:

- feature flag
- environment variable
- disabled / not configured fallback
- local development fallback
- test fake runner
- explicit timeout budget
- optional rollback to `LambdaAgentRunner`

Safety rules:

- do not pass secrets through the runner boundary
- do not pass raw selected marks by default
- keep unsupported / not configured states explicit
- preserve JSON-safe results

## 14. Testing strategy

Future tests should cover:

- input normalization
- output normalization
- unsupported / not configured handling
- comparison with `LambdaAgentRunner`
- trace and observability mapping
- timeout and failure simulation
- selected_mark_explanation golden case

Tests should be local and no-network by default.

## 15. Implementation stages

Suggested stages:

1. design only
2. AgentCoreRunner stub / not configured result
3. input and output normalization helpers
4. fake AgentCore client contract tests
5. optional real integration behind a feature flag
6. selected_mark_explanation comparison run
7. decision record

## 17. Open questions

- Should `AgentCoreRunner` remain thin, or should the first implementation include more runtime responsibilities?
- Should session identifiers be derived from `agentRunId` or generated independently?
- Should trace comparison live in the runner result only, or also in a separate debug artifact later?
- When should AgentCore be allowed to handle more than `selected_mark_explanation`?
