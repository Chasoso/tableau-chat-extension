# Structured Metadata Discovery Plan Shape

This document defines the narrow plan envelope introduced by #186.

## Purpose

`metadata_discovery` needs a structured plan shape before it can reach the ToolLayer.
The plan must separate four outcomes:

- `clarification_required`
- `executable`
- `unsupported`
- `fallback`

The envelope is intentionally small and metadata-discovery-only.
It is not a general-purpose planning DSL.

## Plan contract

The plan shape carries only safe, JSON-friendly contract data:

- intent and target type
- ambiguity state
- missing preconditions
- clarification gate
- execution gate
- unsupported gate
- fallback gate
- narrow app-level wrapper candidate metadata
- trace-safe summaries

It must not carry:

- raw MCP payloads
- raw transport payloads
- arbitrary query text
- underlying data
- field values or row data
- tokens, secrets, or stack traces

## Gates

- `clarification_required` means the request must be restated or clarified before execution.
- `executable` means the request may later be connected to the hosted `describeDatasource` wrapper boundary in #187.
- `unsupported` means the request is unsafe or out of scope for structured metadata discovery.
- `fallback` means the legacy free-form chat path should remain in control.

## ToolLayer boundary

The only executable datasource metadata candidate in this milestone is the app-specific `describeDatasource` wrapper.
`listFields` remains deferred as a controlled / truncated / permission-aware candidate.

## Relation to earlier contracts

- #183 defines the `metadata_discovery` intent.
- #184 defines target ambiguity and preconditions.
- #185 defines clarification response contracts.
- #186 combines those into a narrow structured plan shape.
- #187 will connect only safe executable datasource cases to `describeDatasource`.
