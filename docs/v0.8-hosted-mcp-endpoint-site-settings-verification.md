# Hosted Tableau MCP Endpoint and Site Settings Verification

## 1. Purpose

This document verifies official information before any live Hosted MCP execution work begins.

Its purpose is to:

- confirm whether a Hosted Tableau MCP endpoint shape is documented by official sources
- confirm whether site settings and admin enablement requirements are documented by official sources
- compare Tableau Cloud and Tableau Server availability where possible
- define the v0.8.0 implementation gate for Hosted execution
- provide the evidence needed before #159 through #161 can move beyond guarded planning

## 2. Background

v0.7.0 completed the Hosted Tableau MCP Migration Foundation.

That foundation established:

- transport abstraction
- transport configuration strategy
- auth / user-context / site-settings boundary
- read-only metadata tools
- input/output schemas
- preconditions and governance boundaries
- fake / no-network handlers
- transport-aware execution boundary
- output normalization and trace events
- stdio to Hosted migration path

v0.8.0 is expected to connect the first read-only metadata tool, `tableau.metadata.describeDatasource`, to a gated Hosted / remote Tableau MCP execution path.

Hosted execution must remain explicit opt-in, and default CI must not depend on Hosted MCP.

This issue is the verification step before #159 and later implementation issues.
No real connection, token exchange, or transport implementation is performed here.

## 3. Verification method

The review followed these rules:

- official Tableau and Salesforce / Tableau help pages were checked first
- the MCP specification was checked for transport-level context
- non-official sources were not used as evidence
- if an item could not be confirmed from official sources, it is marked `Unknown / needs verification`
- each reviewed source records URL, title, and checked date
- Tableau Cloud and Tableau Server were checked separately where the docs differ
- endpoint shape, site settings, admin enablement, and REST API availability were checked separately
- no live Hosted MCP connection, token acquisition, or endpoint probing was attempted

Repository context was also reviewed for the current migration assumptions in `docs/v0.8-plan.md` and the v0.7 docs, but the findings in this document are grounded in the official sources listed below.

## 5. Hosted MCP availability

No official Tableau document reviewed here explicitly describes a Hosted Tableau MCP endpoint or a Tableau-specific hosted MCP availability model.

What the reviewed official docs do confirm:

- Tableau Cloud and Tableau Server both support Connected Apps for REST API and Metadata API authorization
- direct trust and OAuth 2.0 trust are both available for Connected Apps
- connected apps are site-configured and admin-enabled

What remains unconfirmed:

- whether Hosted Tableau MCP is GA, preview, beta, or otherwise gated
- whether Hosted MCP is available on Tableau Cloud, Tableau Server, or both
- whether Hosted MCP requires a specific product edition, license, region, pod, or site version
- whether Hosted MCP is tenant-scoped, site-scoped, or user-scoped
- whether a Tableau-specific hosted MCP endpoint is documented at all

Status summary:

| Environment | Hosted MCP availability | Notes |
| --- | --- | --- |
| Tableau Cloud | Unknown / needs verification | Connected Apps prerequisites are documented, but Hosted MCP itself is not documented in the reviewed sources |
| Tableau Server | Unknown / needs verification | Connected Apps prerequisites are documented, but Hosted MCP itself is not documented in the reviewed sources |

## 6. Endpoint shape findings

The official Tableau docs reviewed here do not document a Hosted Tableau MCP endpoint URL, path, or transport shape.

What is confirmed from the MCP specification:

- MCP defines stdio and HTTP with Server-Sent Events as standard transports
- MCP also allows custom transports if they preserve the protocol lifecycle and JSON-RPC message format
- MCP is transport-agnostic at the specification level

What remains unconfirmed for Tableau Hosted MCP:

- endpoint URL format
- whether the endpoint is per site, per tenant, or global
- whether the transport is stdio, SSE, a custom HTTP transport, or another pattern
- required headers
- auth header shape
- site or tenant identifier passing
- user-context passing
- versioning or endpoint discovery rules
- timeout and streaming behavior
- localhost or callback handling

Important consequence:

- the endpoint should be treated as `Unknown / needs verification`
- `HostedTableauMcpTransport` should support a `not_configured` first path if endpoint configuration is missing
- no endpoint should be invented by inference

## 7. Site settings findings

Connected Apps are clearly configured from the site settings area in both Tableau Cloud and Tableau Server.

Confirmed findings:

- Tableau Cloud uses `Settings > Connected Apps`
- Tableau Server uses `Settings > Connected Apps`
- a site admin enables Connected Apps in Tableau Cloud
- a server admin enables Connected Apps in Tableau Server
- connected apps are disabled by default when created and must be enabled manually
- REST API authorization workflows are supported for connected apps

What remains unconfirmed:

- whether a Hosted MCP-specific site setting exists
- whether Hosted MCP is controlled by a separate feature flag, connected app setting, or site setting
- whether the hosted enablement setting can be inspected or changed via API
- whether the setting differs between dev, staging, and production sites

## 8. Admin enablement requirements

The official docs establish that Connected Apps are admin-enabled, but they do not document a Hosted MCP-specific admin model.

Confirmed administration facts:

- Tableau Cloud Connected Apps are configured by a site admin
- Tableau Server Connected Apps are configured by a server admin
- new connected apps are disabled by default and must be enabled after creation

What is still unknown:

- whether Hosted MCP requires site admin, server admin, org admin, or another role
- whether enablement is site-level, tenant-level, user-level, or app-level
- whether an app admin has any role in Hosted MCP enablement

Repository implication:

- admin enablement should be treated as a precondition, not as an assumption
- the user-facing failure mode for disabled hosted access should be safe and explicit
- this verification feeds the precondition boundary work associated with #134 and the later auth boundary work in #160

## 10. REST API settings availability

Confirmed:

- Connected Apps can authorize REST API usage
- Connected Apps can authorize Metadata API usage
- for direct trust, Tableau docs explicitly mention REST API authorization workflows and Metadata API authorization via JWT
- for OAuth 2.0 trust, Tableau docs also mention Metadata API authorization workflows

Unconfirmed:

- whether there is a REST API for checking Hosted MCP enablement
- whether there is a REST API for toggling a Hosted MCP-specific site setting
- whether GraphQL, Metadata API, or admin APIs expose Hosted MCP readiness directly

Recommended interpretation:

- REST API settings for connected apps are real and documented
- REST API settings for Hosted MCP itself remain `Unknown / needs verification`
- if no Hosted MCP settings API exists, #158 and #159 should rely on docs, config, and precondition checks rather than attempting to discover state at runtime

## 11. Environment and network constraints

Hosted execution would require backend reachability to the hosted endpoint, but the concrete networking model is not documented in the sources reviewed here.

Constraints and implications:

- backend Lambda likely needs outbound HTTPS access to the hosted endpoint, but this is not yet confirmed
- VPC attachment could require NAT or equivalent egress
- firewall or proxy rules may block the hosted endpoint unless explicitly allowed
- Lambda and API Gateway timeouts may constrain hosted round trips
- streaming or long-lived sessions may conflict with invocation-based runtimes
- local development and CI must remain usable without Hosted MCP
- no-network tests must stay isolated from hosted verification

MCP transport context from the official spec:

- MCP standardizes stdio and HTTP with SSE
- the spec also allows custom transports
- if Tableau uses a custom hosted transport, it still needs a documented connection and message exchange pattern

## 12. Auth and user-context implications

The reviewed official docs confirm that Tableau Connected Apps support both direct trust and OAuth 2.0 trust.

Confirmed auth facts:

- direct trust uses JWT signed with a connected-app secret
- OAuth 2.0 trust uses an external authorization server
- the authenticated Tableau user is represented in the token subject
- Metadata API authorization via connected apps uses `tableau:content:read`
- Connected Apps can be enabled from site settings and then used for REST API / Metadata API authorization

Implications for Hosted MCP:

- some form of user-context propagation is likely required
- the auth boundary should stay outside the Tool Layer
- raw access tokens and refresh tokens must not appear in tool output or trace output
- `tokenReference` or an equivalent indirection should be preserved
- the exact hosted auth flow is still unknown

Unknowns that block exact implementation decisions:

- whether OAuth is mandatory for Hosted MCP
- whether DCR or static registration is required
- whether the current Connected App / Direct Trust model can be reused
- whether a backend confidential client is supported
- how user-context and site-context are propagated to a hosted MCP endpoint

## 13. Impact on v0.8.0 Hosted execution

This verification changes the v0.8.0 plan in a conservative way:

- #159 needs a transport skeleton that can fail safely when the endpoint is missing
- #160 needs an auth boundary that does not expose raw credentials
- #161 needs a feature-flagged hosted path with explicit fallback
- #158 needs gating that keeps default CI off the hosted path

What the verification does not yet support:

- a real Hosted MCP connection
- endpoint construction by inference
- a `GO` gate for full hosted execution

Practical result:

- real Hosted execution should remain blocked pending official details
- docs, skeletons, and `not_configured`-first fallback design can still proceed

## 14. Implementation gate

```text
BLOCKED_PENDING_OFFICIAL_DETAILS
```

Reason:

- Connected Apps prerequisites are confirmed
- Hosted MCP endpoint shape is not documented in the reviewed official sources
- Hosted MCP-specific site settings are not documented in the reviewed official sources
- Hosted MCP-specific admin enablement is not documented in the reviewed official sources
- therefore real Hosted execution should not proceed yet

This gate does not block docs work or guarded skeleton design. It blocks moving #159 through #161 into a real hosted connection until official Hosted MCP details are confirmed.

## 16. Recommendations for #158, #159, #160, and #161

### #158 Define Hosted MCP integration test gating

- default CI must remain no-network
- hosted tests should be optional and gated
- required secrets should be documented but not required by default
- skip behavior should be explicit
- endpoint and site settings unknowns should determine the gating mode

### #159 Implement HostedTableauMcpTransport skeleton

- the skeleton should support `not_configured` first
- endpoint or auth missing should not throw raw errors
- no real network should be used by default
- constructor should accept endpoint config via dependency injection
- the transport should normalize missing endpoint and unsupported protocol cases
- raw tokens and secrets must not be logged

### #160 Add Hosted MCP auth context adapter boundary

- preserve the `tokenReference` boundary
- do not pass raw access tokens through ToolDefinition or trace payloads
- map missing, expired, and unknown auth states safely
- keep OAuth implementation out unless later official details require it

### #161 Connect `describeDatasource` to Hosted MCP transport behind feature flag

- only proceed if the gate is `GO` or `GO_WITH_GUARDS`
- use an explicit hosted feature flag
- preserve fake / no-network fallback
- preserve stdio fallback
- connect only `describeDatasource`
- do not connect `listFields` unless a separate readiness decision allows it
