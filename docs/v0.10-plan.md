# v0.10.0 Plan: LLM ResponseComposer or Exploration Session

## Summary

v0.10.0 is the next milestone after the v0.9.0 structured discovery result.

Its purpose is to choose between two alternative directions:

- LLM ResponseComposer
- Exploration Session

Selected branch: LLM ResponseComposer.

AgentCore full migration remains out of scope.

v0.10.0 is not a broad Hosted MCP exposure milestone and not an arbitrary query execution milestone.
It must preserve the v0.9.0 safety boundaries.

## Relationship to v0.9.0

v0.9.0 completed the narrow structured metadata discovery path and documented the result.

That result left behind a safe boundary with the following properties:

- `metadata_discovery` is defined
- ambiguity handling is defined
- clarification responses are defined
- the structured metadata discovery plan shape is defined
- the safe datasource metadata path connects to `describeDatasource`
- `listFields` remains a controlled candidate
- truncation and permission-aware response shaping exist
- legacy chat reduction happened only for eligible discovery cases
- `selected_mark_explanation` remained protected
- free-form chat fallback remained protected

v0.10.0 must decide whether the next improvement should focus on response quality or workflow continuity.

## v0.10.0 decision

Branch decision:

- Selected branch: LLM ResponseComposer
- Decision date: 2026-07-05
- Decision summary:
  - v0.9.0 already established a safe, explicit structured-discovery boundary.
  - The next smallest product win is to improve user-facing wording and explanation quality without widening state or memory scope.
  - LLM composition can stay optional and guarded while preserving the deterministic baseline.

Deferred branch:

- Exploration Session
- Deferred because session state, lifecycle, privacy, and stale-context risks are larger than the current need.
- Revisit when multi-step continuity clearly becomes the primary product pain.

AgentCore:

- AgentCore remains deferred.
- Revisit only if runtime/session complexity, observability gaps, timeout constraints, or deployment/cost/IAM trade-offs justify it.

## Option A: LLM ResponseComposer

Purpose:

- keep a deterministic ResponseComposer as the baseline
- add optional and guarded LLM composition
- improve user-facing wording and explanation quality
- keep the response material evidence-bounded

The safety boundary for this option is documented in `docs/v0.10-llm-response-composer-safety-boundary.md`.

The response material contract for `#211 Define response material contract for LLM composition` is documented in `docs/v0.10-llm-response-material-contract.md`.

The guarded prototype for `#212 Add guarded LLM ResponseComposer prototype` is documented in `docs/v0.10-guarded-llm-response-composer-prototype.md`.

Best fit when:

- the main complaint is that answers are too stiff or too hard to read
- the structured discovery boundary is already good enough
- the next gain should come from response quality rather than session continuity

## Option B: Exploration Session

Purpose:

- treat selected marks, metadata discovery, and interaction history as a session
- support multi-step exploratory analysis
- improve continuity across related discovery turns
- prepare a safer path for richer runtime/session management later

Best fit when:

- the main complaint is that discovery does not carry context well across turns
- the product needs more exploration continuity than wording polish
- safe session state is more urgent than composition quality

## Recommended direction

Recommended initial direction: planning-first.

Rationale:

- v0.9.0 already solved the hard structured discovery boundary problem
- the remaining choice is a product-direction choice, not a missing technical boundary
- the repository now needs a branch decision issue to compare the two directions with current evidence
- if the decision is wrong, the first child issue can still switch the v0.10.0 branch before implementation starts

If this assumption is wrong, the first child issue should allow switching the v0.10.0 branch before implementation starts.

## Scope

v0.10.0 scope should include:

- the branch decision issue
- only the chosen branch after that decision
- guarded safety boundaries for the selected direction
- no-network validation and regression protection
- a final wrap-up that records the selected branch and the deferred branch

## Out of scope

v0.10.0 should not include:

- both branches fully implemented at once
- AgentCore full migration
- broad Hosted MCP exposure
- arbitrary query execution
- underlying data access
- field values or row data retrieval
- write-capable tools
- frontend redesign
- default hosted CI changes

## Safety boundaries

v0.10.0 must preserve the following:

- no raw MCP tool exposure
- no raw MCP output exposure
- no raw transport output exposure
- no arbitrary query execution
- no underlying data access
- no field values or row data retrieval
- no write-capable tools
- no tokens, secrets, or stack traces in user-facing output or traces
- default tests remain no-network
- hosted or real LLM or real session-store integration remains optional or gated
- `selected_mark_explanation` remains protected
- legacy free-form chat fallback remains available unless explicitly changed by a later issue
- deterministic baseline behavior remains available

## AgentCore deferred decision

AgentCore remains deferred in v0.10.0.

Revisit it only if:

- runtime complexity increases materially
- cross-run session state becomes a product requirement
- observability is no longer sufficient
- timeout constraints become a real bottleneck
- the cost / IAM / deployment trade-offs are clearer

## Validation strategy

This milestone planning is docs-only.

Validation should remain docs review only until a branch is selected and an implementation issue is opened.

If a later code-bearing issue is created, it should use the existing no-network baseline and preserve the current regression posture.

## Open questions

The milestone starts with these open questions:
