# listFields Hosted Execution Readiness

## 1. Purpose

This document evaluates whether `tableau.metadata.listFields` is ready for
Hosted execution in v0.8.0.

The goal is not to force Hosted execution for `listFields`, but to decide
whether it is safe enough to promote now or whether it should be deferred to
v0.9.0.

## 2. Background

v0.8.0 already promotes `tableau.metadata.describeDatasource` to the first
Hosted metadata execution target.

By contrast, `listFields` has a higher risk profile because it can return more
data, is more sensitive to truncation and omission behavior, and is more likely
to encounter hidden-field and permission edge cases.

That makes it a poor candidate for the first Hosted promotion unless the
field-shape, normalization, and error-handling guarantees are already stable.

## 3. Evaluation criteria

`listFields` should be evaluated against the following criteria:

- output size risk
- truncation and omission behavior
- hidden-field behavior
- permission behavior
- schema mapping complexity
- trace payload size risk
- risk of exposing field values or row-like data
- compatibility with the existing fake / no-network baseline

## 4. Risk comparison with `describeDatasource`

`describeDatasource` is the safer first Hosted target because it is
summary-first and generally smaller.

`listFields` is riskier because:

- it can surface many more fields than `describeDatasource`
- the truncation and omission story matters more
- hidden-field and permission edge cases are more likely
- schema mapping and normalization pressure is higher
- trace payloads can become noisy faster

## 5. Output shape and normalization concerns

The current v0.7.0 baseline already normalizes fake metadata output, but
`listFields` still needs extra scrutiny before Hosted execution:

- field summary shape must remain stable
- truncation must be explicit and safe
- omission must be reported safely
- warnings must remain JSON-safe
- trace payloads must stay small and sanitized

## 6. Permission and hidden-field concerns

Hosted execution for `listFields` has more opportunity to hit:

- permission denied responses
- hidden-field visibility differences
- site or workbook-specific metadata restrictions

Those edge cases are not a reason to abandon the feature, but they do make it a
better fit for a later milestone.

## 7. Hosted error behavior

`listFields` would need the same hosted error normalization that
`describeDatasource` uses, but with additional attention to truncation and
shape-related failures.

Any future Hosted execution path for `listFields` must continue to suppress:

- raw MCP payloads
- raw transport errors
- tokens and secrets
- raw stack traces

## 8. Recommendation

Recommendation: defer Hosted execution for `tableau.metadata.listFields` to
v0.9.0.

Rationale:

- `describeDatasource` already covers the first Hosted verification path
- `listFields` carries higher output-size and permission risk
- the fake / no-network baseline already provides deterministic coverage
- v0.8.0 should stay narrow and avoid broadening Hosted scope too early

## 9. v0.9.0 handoff

If `listFields` is promoted later, v0.9.0 should require:

- validated output-shape mapping
- explicit truncation and omission rules
- hosted error behavior parity
- trace payload safety
- regression coverage for fake / no-network and stdio fallback paths

## 10. Open questions

- whether `listFields` can share the same Hosted transport contract as
  `describeDatasource`
- whether `listFields` needs a stricter output cap before promotion
- whether field-level permission differences require a separate precondition
  path
