# v0.8.0 Hosted MCP Metadata Execution Result

## 1. Purpose

This document records the result of the v0.8.0 Hosted MCP metadata execution
milestone.

It exists to summarize what was implemented, what was validated, what remains
intentionally blocked, and what should be handed to v0.9.0 and #167.

## 2. Milestone context

v0.8.0 follows the v0.7.0 Hosted Tableau MCP Migration Foundation.

That foundation established the transport, auth, precondition, normalization,
and fake / no-network boundaries needed for a gated Hosted metadata path.

v0.8.0 used that foundation to connect the first hosted metadata tool:

- `tableau.metadata.describeDatasource`

`tableau.metadata.listFields` was evaluated separately and remains deferred to
v0.9.0 because its output size, truncation, omission, hidden-field, and
permission risks are higher.

Free-form chat migration was not part of this milestone. Arbitrary query,
underlying data access, and write-capable tools were also out of scope.
AgentCore remains deferred.

## 3. Executive summary

v0.8.0 established a gated Hosted MCP metadata execution path for
`tableau.metadata.describeDatasource`, while preserving fake / no-network
execution, stdio fallback strategy, safe auth context boundaries, Hosted MCP
metadata error normalization, hosted trace coverage, `selected_mark_explanation`
behavior, and the free-form chat legacy path.

`listFields` hosted execution was evaluated separately and remains deferred to
v0.9.0 unless its output-shape, truncation, hidden-field, permission, and
schema-mapping risks are resolved.

## 4. Implemented hosted execution path

The implemented path is:

```text
tableau.metadata.describeDatasource
↓
ToolRegistry handler
↓
metadata precondition evaluator
↓
transport config resolver
↓
Hosted MCP auth context adapter
↓
HostedTableauMcpTransport
↓
Hosted MCP metadata error normalization
↓
metadata output normalization
↓
hosted metadata trace events
↓
safe normalized result
```

What this means in practice:

- only `describeDatasource` is connected to Hosted execution
- the tool layer still uses the existing precondition and transport-aware
  boundaries
- fake / no-network and stdio fallback remain available
- transport details remain hidden from the tool layer

## 5. Feature flag behavior

Hosted execution is explicit opt-in. The default path is still no-network.

| Scenario | Expected behavior | Notes |
| --- | --- | --- |
| feature flag off | fake / no-network or stdio fallback | Hosted is not selected implicitly |
| feature flag on and config ready | Hosted path is used for `describeDatasource` | This is the gated hosted path |
| no-network / CI | fake / no-network | Default CI does not require Hosted MCP |
| endpoint missing | safe `not_configured` or fallback | Depends on resolver policy |
| auth missing or expired | safe auth-required / auth-expired result | No raw credential exposure |
| site settings disabled | safe blocked result | Treated as a precondition failure |
| hosted timeout / unsupported / network failure | safe normalized error | May fall back if policy allows |

## 6. Auth and site-settings findings

The v0.8.0 auth and site-settings work established a safe boundary, but the
official documentation did not fully define Hosted MCP itself.

Confirmed:

- Connected Apps are documented for Tableau Cloud and Tableau Server
- site admin / server admin enablement is documented for Connected Apps
- the Hosted auth adapter can produce safe auth states and safe user summaries
- tokenReference is preserved only as an opaque handoff

Unconfirmed:

- Hosted MCP endpoint shape
- Hosted MCP-specific site settings
- Hosted MCP-specific admin enablement model
- whether Hosted MCP is available on Cloud, Server, or both

Implementation gate:

- `BLOCKED_PENDING_OFFICIAL_DETAILS`

That gate is about full live Hosted execution only. It does not block the docs,
the skeletons, or the fallback-first design.

## 7. Hosted transport result

The Hosted transport skeleton is in place and remains no-network friendly by
default.

It supports:

- constructor-based configuration
- dependency injection for request client, clock, and logger
- `not_configured` behavior when endpoint, auth, site context, or network
  inputs are missing
- `unsupported` behavior for unsupported tools or operations
- `timeout` and safe error placeholders
- safe trace and timing metadata
- no token logging

The transport skeleton is a contract boundary, not a live production Hosted MCP
client.

## 8. describeDatasource hosted execution result

`describeDatasource` is the only hosted metadata tool wired in v0.8.0.

What works:

- feature-flagged Hosted execution path
- precondition evaluator reuse
- Hosted auth adapter usage
- transport-aware execution boundary reuse
- normalized summary-first output
- safe Hosted trace metadata
- fake / no-network fallback
- stdio fallback when configured

What remains blocked or deferred:

- `listFields` Hosted execution
- raw MCP tool exposure
- arbitrary query execution
- underlying data access

## 9. Error normalization findings

Hosted MCP errors are normalized into safe metadata summaries.

| Hosted error category | Normalized summary | User action | Retryable |
| --- | --- | --- | --- |
| `AUTH_REQUIRED` | auth required | reauthenticate | yes |
| `AUTH_EXPIRED` | auth expired | reauthenticate | yes |
| `PERMISSION_DENIED` | permission denied | check permissions | usually no |
| `SITE_SETTINGS_DISABLED` | hosted disabled by site settings | ask admin to enable | no |
| `TRANSPORT_NOT_CONFIGURED` | transport not configured | fix config | yes / depends |
| `NETWORK_ERROR` | network error | retry / check connectivity | yes |
| `TIMEOUT` | timeout | retry or investigate latency | yes |
| `MCP_PROTOCOL_ERROR` | protocol error | inspect integration | maybe |
| `TOOL_NOT_FOUND` | tool not found | fix tool mapping | no |
| `INVALID_INPUT` | invalid input | correct request | no |
| `REMOTE_SERVER_ERROR` | remote server error | retry / inspect upstream | maybe |
| `UNKNOWN_ERROR` | unknown error | inspect safely | no |

Raw MCP error bodies, raw stack traces, tokens, and secrets are not exposed.

## 10. Trace coverage findings

Hosted metadata execution is traceable without exposing raw payloads.

Trace coverage includes:

- `tableau_metadata_tool.started`
- `tableau_metadata_tool.completed`
- `tableau_metadata_tool.failed`

Safe trace fields include:

- `transportKind`
- `requestId`
- `correlationId`
- `agentRunId`
- `durationMs`
- `errorCode`
- `warningCount`
- fallback distinctions such as `fallbackUsed`, `fallbackFrom`, and
  `fallbackTo`

The trace contract remains JSON-safe and excludes raw MCP payloads, tokens,
secrets, and stack traces.

## 11. Fake / no-network behavior

The baseline fake / no-network path remains intact.

It still provides:

- deterministic `describeDatasource`
- deterministic `listFields`
- normalized fake output
- fake warnings and trace metadata
- no Hosted endpoint requirement
- no OAuth requirement
- no token or secret requirement

Default CI remains no-network.

## 12. stdio fallback behavior

stdio fallback remains available as a safe alternative when configured.

It remains useful for:

- fallback from non-hosted configuration
- fallback from Hosted not-configured paths
- preserving local development flexibility

The default test path does not need to launch a real stdio process.

## 13. listFields readiness result

`tableau.metadata.listFields` remains deferred to v0.9.0.

Reasoning:

- output size risk is higher than `describeDatasource`
- truncation and omission behavior is more complex
- permission and hidden-field risk is higher
- schema mapping is more fragile
- trace payload size pressure is higher

The practical result is that v0.8.0 does not add Hosted `listFields`
execution.

## 14. Working cases

Validated or preserved working cases include:

- feature flag off -> fake / no-network
- no-network / CI -> fake / no-network
- `describeDatasource` hosted path behind explicit flag
- missing endpoint -> safe `not_configured` or fallback
- missing auth -> safe auth-required result or fallback
- expired auth -> safe auth-expired result
- Hosted timeout -> safe timeout result
- Hosted protocol or remote server failure -> safe normalized error
- fake `listFields` remains available
- `selected_mark_explanation` remains unchanged
- free-form chat legacy path remains unchanged

Intentionally blocked or not implemented in v0.8.0:

- raw MCP tool exposure
- arbitrary query execution
- underlying data access
- write-capable tools
- `listFields` Hosted execution
- free-form chat Hosted migration
- ChatService replacement
- runLightweightAgentLoop replacement
- OAuth full implementation
- token storage / refresh
- production Hosted MCP migration
- UI trace panel
- trace persistence
- AgentCore implementation
- LLM ResponseComposer implementation

## 16. Limitations

Remaining limitations are expected and documented:

- Hosted MCP endpoint details are still not fully documented in official
  sources
- live Hosted MCP integration remains optional and gated
- `describeDatasource` is the only Hosted metadata tool promoted in v0.8.0
- `listFields` remains deferred
- OAuth implementation is not complete
- tokenReference is preserved as a boundary, not resolved into a raw token
- default CI does not validate live Hosted MCP
- stdio fallback may still be needed in some environments

## 17. Safety and governance outcome

v0.8.0 preserves the original safety posture:

- read-only metadata first
- `describeDatasource` only for Hosted execution
- no raw MCP tool exposure
- no arbitrary query
- no underlying data
- no field values or row data
- no write tools
- output normalization
- JSON-safe output
- no raw MCP result
- no raw transport result
- no tokens, secrets, or stack traces
- Hosted execution is explicit opt-in
- default CI remains no-network
- fake / stdio fallback is preserved
- `selected_mark_explanation` is unchanged
- free-form chat remains legacy
- AgentCore remains deferred

## 20. v0.9.0 handoff

v0.9.0 is `Structured Data Discovery and Legacy Chat Reduction`.

The v0.8.0 handoff should provide:

- hosted `describeDatasource` result
- Hosted error normalization result
- Hosted trace coverage result
- fake / stdio fallback strategy
- `listFields` readiness recommendation
- remaining auth / site settings / permission issues

v0.9.0 should not start by broadly exposing Hosted MCP tools. It should start
from structured metadata_discovery intent design, using the v0.8.0
`describeDatasource` hosted path as the first validated metadata execution
boundary and treating `listFields` as a controlled, truncated,
permission-aware candidate.
