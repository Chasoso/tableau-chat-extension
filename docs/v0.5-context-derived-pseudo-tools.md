# v0.5.0 Context-Derived Pseudo Tools

## 1. Purpose

v0.5.0 Tool Layer Rebuild needs a clear answer for context-derived data such as selected marks and summary previews. These values are not external tools, but they must still be addressable from orchestration plans, routing, trace, and precondition policy.

This document defines the recommended hybrid approach for representing context-derived data on the Tool layer without turning it into external tool execution.

## 2. Decision

Adopt a hybrid approach.

- Represent context-derived capabilities as pseudo tools in the ToolRegistry contract surface.
- Keep the actual data source in orchestration context, not in an external runtime or network call.
- Treat pseudo tool execution as context reference resolution plus summarization, not as an external side-effecting tool call.

This keeps the orchestration path uniform while preserving the distinction between context access and external tool execution.

## 3. Why context-derived data is tool-like

Context-derived data is tool-like because:

- it is selected by plan steps
- it can be routed, allowed, blocked, or skipped
- it can be gated by preconditions
- it should appear in trace as a first-class orchestration step
- it can be summarized and consumed by ResponseComposer-like logic

It is not tool-like in the external-service sense because:

- it does not call MCP, REST APIs, or third-party services
- it does not mutate external state
- it should not require tool registry discovery

## 4. Options considered

### A. Do not model as tools

Pros:

- simplest implementation
- no registry expansion
- no ambiguity with external tools

Cons:

- harder to route through the same plan / trace / policy flow
- weaker fit for selected_mark_explanation

### B. Register as pseudo tools only

Pros:

- clear tool-like contract
- easier ToolRouter integration
- consistent trace and policy handling

Cons:

- registry may become a mix of external tools and context capabilities
- needs strict privacy rules so raw context does not leak into trace

### C. Hybrid approach

Recommended.

- define pseudo tools in ToolDefinition / ToolRegistry terms
- resolve data from orchestration context
- keep external tool execution semantics separate
- keep trace metadata compact and privacy-safe

## 5. Adopted hybrid approach

The hybrid model is the safest fit for v0.5.0.

- `ToolDefinition` expresses the capability contract.
- `ToolRegistry` can list and look up context-derived capabilities.
- `ToolRouter` can apply allow / disallow / availability rules uniformly.
- `ToolExecutionWrapper` can resolve a context reference rather than perform a network call.
- `ExecutionEngine` can treat the result like a routed step without changing the execution model for external tools.

Recommended defaults:

- category: `context`
- safety: `read_only`
- safeForPreview: `true`
- requiresExplicitAction: `false` for preview-safe context access
- externalAccess: `false`

## 6. Pseudo tool catalog

### 6.1 context.selectedMarks

- Purpose: expose selected mark summary and references
- Category: `context`
- Safety: read_only, preview-safe
- Required precondition: selected marks exist
- Typical output: selected mark count, worksheet names, compact field/value summary, reference only
- Trace data: counts and summary only, never raw mark payloads

### 6.2 context.summaryDataPreview

- Purpose: expose compact summary data preview
- Category: `context`
- Safety: read_only, preview-safe
- Optional for selected_mark_explanation
- Typical output: row count, column names, compact preview summary, truncation flag
- Trace data: counts and truncation status only

### 6.3 context.filters

- Purpose: expose active filters summary
- Category: `context`
- Safety: read_only
- Optional for selected_mark_explanation
- Typical output: filter names, selected value counts, compact summary
- Trace data: filter count and compact summary only

### 6.4 context.parameters

- Purpose: expose parameter summary
- Category: `context`
- Safety: read_only
- Optional for selected_mark_explanation
- Typical output: parameter names and current-value summary
- Trace data: parameter count only

### 6.5 context.dashboardMetadata / context.viewMetadata

- Purpose: expose dashboard / workbook / view references
- Category: `context`
- Safety: read_only
- Typical output: dashboard name, workbook name, view name, identifiers or references when already available
- Trace data: names and references only

## 7. ToolRegistry boundary

ToolRegistry may:

- register and list pseudo tool definitions
- report category, safety, capability, and availability
- participate in allowedTools / disallowedTools filtering
- expose summaries for routing and trace

ToolRegistry must not:

- collect frontend context
- store raw selected marks or full previews
- generate summary data from Tableau directly
- call Tableau MCP
- call external APIs
- persist context bodies

ToolRouter uses this boundary to decide whether a pseudo tool is allowed, blocked, unavailable, or skipped. It does not execute the pseudo tool itself.

## 8. ToolExecutionWrapper boundary

ToolExecutionWrapper or a similar context resolver layer should:

- resolve a pseudo tool against orchestration context
- produce a compact, JSON-safe result
- report missing context as a precondition or routing failure
- keep raw context bodies out of trace

It must not:

- perform external calls for pseudo tools
- fabricate raw payload exports
- blur the line between context resolution and external tool execution

## 9. Trace and privacy rules

- keep raw selected marks out of trace
- keep full summary rows out of trace
- keep trace to counts, compact summaries, and references
- do not log sensitive filter values unless explicitly required and already summarized
- record fallback reasons and availability status instead of raw bodies

## 10. selected_mark_explanation minimal path

Recommended minimal path:

```text
selected_mark_explanation
  -> context.selectedMarks (required)
  -> context.summaryDataPreview (optional)
  -> context.filters (optional)
  -> context.parameters (optional)
  -> deterministic ResponseComposer skeleton
```

Rules:

- do not use Tableau MCP yet
- do not use external APIs yet
- do not use Bedrock or LLMs yet
- keep response deterministic and clearly marked as a placeholder or safe summary
- if `context.selectedMarks` fails, fall back safely
- if optional context tools fail, continue with warnings
