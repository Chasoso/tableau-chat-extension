# Hosted MCP Metadata Error Normalization

## 1. Purpose

This document defines how Hosted MCP transport and hosted metadata execution errors are normalized into the existing safe Tableau metadata error summary model.

The goal is to keep raw Hosted MCP protocol details, stack traces, tokens, and secrets out of user-facing output and trace payloads.

## 2. Background

v0.8.0 connects `tableau.metadata.describeDatasource` to Hosted MCP behind a feature flag.

That hosted path needs a safe error normalization layer because transport-level failures, auth failures, site-settings failures, protocol failures, and upstream server failures must not leak raw details into the metadata tool layer.

## 3. Error normalization boundary

Hosted MCP errors are normalized at the metadata boundary, not in the Tool Layer.

The normalizer accepts safe transport or execution context and returns a `TableauMetadataErrorSummary` with:

- a stable error code
- a safe user-facing message
- retry and user-action hints
- JSON-safe metadata for trace and debugging

The normalizer must not return raw transport payloads or raw exceptions.

## 4. Error category mapping

Supported Hosted MCP categories:

- `AUTH_REQUIRED`
- `AUTH_EXPIRED`
- `PERMISSION_DENIED`
- `SITE_SETTINGS_DISABLED`
- `TRANSPORT_NOT_CONFIGURED`
- `NETWORK_ERROR`
- `TIMEOUT`
- `MCP_PROTOCOL_ERROR`
- `TOOL_NOT_FOUND`
- `INVALID_INPUT`
- `REMOTE_SERVER_ERROR`
- `UNKNOWN_ERROR`

The normalizer may preserve a generic `TRANSPORT_FAILED` summary for unsupported or uncategorized transport failures when no safer classification is available.

## 5. Auth required and expired errors

Auth failures are normalized into safe summaries:

- missing auth -> `AUTH_REQUIRED`
- expired auth -> `AUTH_EXPIRED`

Both categories should be recoverable and should point the caller toward reauthentication without exposing token values.

## 6. Permission and site settings errors

Permission and site-setting failures are normalized safely:

- permission denied -> `PERMISSION_DENIED`
- hosted execution disabled by site settings -> `SITE_SETTINGS_DISABLED`

The summary should explain the failure at a high level and avoid exposing permission scopes, role details, or admin configuration internals.

## 7. Configuration and endpoint errors

Configuration problems are normalized as safe metadata errors:

- endpoint missing -> `TRANSPORT_NOT_CONFIGURED`
- hosted auth not configured -> `TRANSPORT_NOT_CONFIGURED` or `AUTH_REQUIRED`, depending on context

The summary must not echo endpoint URLs, client secrets, or raw config payloads.

## 8. Network and timeout errors

Connectivity problems are normalized into:

- `NETWORK_ERROR`
- `TIMEOUT`

These summaries should be retryable when appropriate and must not expose low-level socket, DNS, or abort-controller details.

## 9. Protocol and tool availability errors

Hosted MCP protocol and tool-availability failures are normalized into:

- `MCP_PROTOCOL_ERROR`
- `TOOL_NOT_FOUND`

The result should stay summary-first and should not expose raw MCP frames or tool listings.

## 10. Invalid input and remote server errors

Input validation and upstream failures are normalized into:

- `INVALID_INPUT`
- `REMOTE_SERVER_ERROR`

The normalizer should keep the message safe and preserve only sanitized context needed for debugging.

## 11. Unknown error handling

Unrecognized Hosted errors are normalized into `UNKNOWN_ERROR`.

This is the final fallback for thrown strings, unknown exception shapes, or values that cannot be safely classified.

## 12. Safe user-facing messages

User-facing messages must stay generic and safe.

They should mention the category, not raw URLs, tokens, stack traces, or response bodies.

## 13. Trace and debugging metadata

Safe debug metadata may include:

- tool name
- operation
- request, correlation, and run identifiers
- transport kind
- transport status
- normalized code
- retry / user-action hints

Raw request bodies, raw responses, raw exceptions, and raw stack traces must stay out of the result.

## 14. Token, secret, and stack trace suppression

The normalizer must not expose:

- access tokens
- refresh tokens
- authorization headers
- secrets
- stack traces
- raw MCP payloads

Any metadata fields that might contain sensitive values should be omitted or redacted.

## 15. Relationship with HostedTableauMcpTransport

`HostedTableauMcpTransport` should emit safe error codes and the normalizer should convert them into stable metadata summaries.

The transport skeleton remains responsible for avoiding raw network behavior by default.

## 16. Relationship with describeDatasource hosted execution

`tableau.metadata.describeDatasource` is the first hosted metadata tool and uses this normalizer when the hosted feature flag is enabled.

The output should remain summary-first and should not expose raw transport details.

## 17. Relationship with fallback behavior

When Hosted execution falls back to fake or stdio, the existing fallback summaries remain in place.

Hosted-specific error summaries should only appear when the Hosted or remote path is actually selected.

## 18. Recommendations for #163 and #164

- #163 should validate Hosted trace coverage for these normalized categories.
- #164 should keep fake and stdio regression coverage intact so Hosted normalization does not break fallback behavior.

## 19. Open questions

- whether `TRANSPORT_FAILED` should remain a generic fallback code or be reduced further
- whether Hosted-specific summaries should also surface additional sanitized server metadata
- whether `listFields` needs the same hosted error mapping in a later issue
