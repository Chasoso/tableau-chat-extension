# LLM Response Material Contract

## Purpose

This document defines the safe response material that can be passed to an optional LLM ResponseComposer.

The purpose of the material is to provide grounded, JSON-safe, evidence-bounded input for wording and explanation quality improvements without widening execution scope.

This contract is the input-side companion to `docs/v0.10-llm-response-composer-safety-boundary.md`.

## Relationship to #210

#210 defined the safety boundary for the optional LLM ResponseComposer.

This issue defines the safe material that can cross that boundary.

The contract keeps the composer in a wording role:

- it may improve phrasing
- it may preserve limitation notes
- it may restate evidence already available
- it may not invent new facts

## Response material role

Response material is a safe intermediate representation.

It is:

- not a raw execution result
- not a raw MCP result
- not a raw transport result
- not a prompt dump
- not a memory store
- not a query execution plan

The material exists so that the composer can stay evidence-bounded and no-network friendly.

## Allowed material sources

The composer may only receive material derived from safe normalized sources such as:

- normalized metadata discovery results
- `describeDatasource` normalized summaries
- controlled `listFields` contract-derived summaries, when available as safe summaries and not raw execution output
- deterministic `selected_mark_explanation` result summaries
- clarification responses
- truncation notes
- permission-aware limitation notes
- limitation notes
- safe warning codes
- trace-safe summaries
- user request summaries
- response intent and response type metadata
- safe target summaries
- safe evidence IDs or citation-style references
- safe next-step hints

## Prohibited material sources

The composer must not receive material derived from:

- raw MCP output
- raw MCP payloads
- raw MCP tool output
- raw MCP tool names or payloads intended for execution
- raw transport output
- raw Hosted MCP responses
- raw Tableau responses
- raw trace payloads
- unredacted logs
- tokens
- secrets
- OAuth credentials
- stack traces
- internal exception dumps
- underlying data
- row data
- field values
- sample values
- domain values
- arbitrary query text intended for execution
- generated query text
- write operation payloads
- session memory or Exploration Session state

## Material schema

The material schema should remain small and JSON-safe.

Suggested fields:

- `materialVersion`
- `responseType`
- `intent`
- `sourceKind`
- `userRequestSummary`
- `targetSummary`
- `evidence`
- `citations`
- `facts`
- `limitations`
- `warnings`
- `safetyNotes`
- `traceSafeSummary`
- `fallbackSummary`
- `composerInstructions`
- `prohibitedClaims`
- `requiredDisclosures`

The concrete implementation may use a narrower subset, but the schema must preserve the distinction between evidence, facts, and limitations.

## Evidence and citation rules

Evidence and citations must be safe references, not raw payloads.

Rules:

- evidence must be derived from normalized summaries only
- evidence IDs must be stable and JSON-safe
- citation-style references must be safe references, not raw URLs or payload dumps
- evidence source type should be distinguishable
- evidence types may include `metadata_discovery`, `selected_mark_explanation`, `clarification`, `trace_summary`, and `limitation`
- evidence may not include raw data or raw tool output
- evidence may not fabricate counts, permissions, or metadata
- unknowns must remain unknown

## Facts and limitations

Facts and limitations must be separated.

### Facts

Safe facts may include:

- normalized datasource metadata summaries
- safe selected-mark explanation summaries
- clarified target summaries
- response category labels
- safe count summaries only when they are already normalized and not fabricated

### Limitations

Limitations may include:

- truncated response
- omitted count unknown
- permission limited
- permission unknown
- hosted integration unavailable
- fake / no-network fallback used
- target ambiguity
- unsupported request
- `listFields` execution deferred
- data access not available

Limitations are required disclosures, not optional decorations.

## Redaction and suppression rules

Material creation must redact or suppress the following:

- raw MCP output
- raw transport output
- token
- secret
- OAuth credential
- stack trace
- internal exception dump
- underlying data
- field values
- row data
- sample values
- domain values
- raw query text
- generated query text intended for execution
- write payload
- unredacted logs

If the source cannot be safely normalized, it must be excluded from material.

## Required disclosures

The composer must be able to preserve required disclosures such as:

- truncation notes
- permission notes
- limitation notes
- fallback reasons
- clarification requirements
- unsupported / deferred status

Required disclosures must not be silently dropped during composition.

## Composer instruction boundary

The material may include safe composer instructions or a safe instruction summary.

Allowed instruction themes:

- use only the provided evidence
- do not add unsupported facts
- preserve required limitations
- do not mention hidden details
- do not create tool calls
- do not generate queries
- keep the response concise
- ask for clarification if required
- represent unknowns as unknown

Instructions must remain non-executable and must not encode prompts that would bypass the safety boundary.

## Selected_mark_explanation material handling

`selected_mark_explanation` must remain deterministic and unchanged.

For composer material, only safe summaries are allowed:

- selected mark count summary
- safe worksheet or field label summaries
- safe preview availability
- safe limitation notes

The material must not include:

- raw selected mark rows
- row values
- field values
- underlying data
- sample values

## Metadata_discovery material handling

`metadata_discovery` material must come from the normalized response path established in v0.9.0.

Allowed examples:

- safe datasource metadata summary
- controlled metadata summary
- truncation notes
- permission notes
- warning codes
- limitation notes
- clarification response summary
- unsupported / fallback reason summary

Not allowed:

- raw `describeDatasource` output
- raw `listFields` output
- raw MCP result
- raw transport result
- field values
- row data
- underlying data
- raw Tableau response

## Validation strategy

The contract should be validated with no-network tests only.

Validation should confirm:

- the material is JSON-safe
- the material is evidence-bounded
- prohibited sources are rejected or excluded
- required disclosures survive material construction
- selected_mark_explanation behavior remains unchanged
- metadata_discovery material uses normalized summaries only
- no real LLM integration is required

- real LLM integration
- prompt execution
- LLM-generated Tableau MCP query execution
- arbitrary query generation
- arbitrary query execution
- underlying data access
- field values or row data retrieval
- write-capable tools
- Exploration Session implementation
- AgentCore implementation
- frontend redesign
- ChatService full replacement
- runLightweightAgentLoop retirement

## Safety boundaries

- response material is JSON-safe
- response material is evidence-bounded
- raw MCP / raw transport / raw Tableau responses are excluded
- underlying data / field values / row data are excluded
- sample values and domain values are excluded
- write-capable tools are excluded
- arbitrary query generation is excluded
- `selected_mark_explanation` remains protected
- legacy free-form chat fallback remains protected
- deterministic ResponseComposer remains baseline
- LLM composition remains optional / guarded
- Exploration Session remains deferred
- AgentCore remains deferred
