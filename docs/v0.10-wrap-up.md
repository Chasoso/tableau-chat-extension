# v0.10.0 Wrap-up

## Summary

v0.10.0 completed Branch A: LLM ResponseComposer.

The milestone kept the deterministic ResponseComposer as the baseline, kept LLM composition optional and guarded, and preserved the no-network default posture.

Exploration Session remains deferred, and AgentCore remains deferred.

The milestone did not add raw MCP exposure, arbitrary query execution, underlying data access, or row / field-value retrieval.

## Milestone outcome

v0.10.0 began as a branch-choice milestone and ended as a completed Branch A delivery.

The path was:

- #209 selected LLM ResponseComposer over Exploration Session
- #210 defined the LLM ResponseComposer safety boundary
- #211 defined the response material contract
- #212 added the guarded LLM ResponseComposer prototype
- #213 added fallback and regression coverage
- #214 documented the Branch A result
- #220 closes out the milestone

The deferred Branch B issues remain deferred:

- #215 `Define Exploration Session safety boundary`
- #216 `Define session state schema and lifecycle`
- #217 `Add minimal Exploration Session prototype`
- #218 `Add Exploration Session fallback and regression tests`
- #219 `Document Exploration Session result`

## Selected branch

Selected branch: LLM ResponseComposer

Why this branch was selected:

- v0.9.0 had already established a safe structured-discovery boundary
- the next smallest product win was response quality improvement
- LLM composition could remain optional and guarded
- the deterministic baseline could stay intact
- the state, privacy, and lifecycle risks were smaller than taking on an Exploration Session first

## Deferred branch

Deferred branch: Exploration Session

Deferred because:

- session state and lifecycle would expand scope more than v0.10.0 needed
- privacy and stale-context risks were larger than the immediate product pain
- uncontrolled memory needed to stay out of scope

Revisit if:

- multi-step continuity becomes the primary product pain
- a safe session-state schema and lifecycle are ready to define
- AgentCore or managed-runtime questions become more relevant

## Final LLM ResponseComposer posture

The final Branch A posture is:

- LLM composition remains optional and guarded
- the deterministic composer remains the default and fallback path
- composition flows through the response material contract
- output validation is in place
- failure, timeout, invalid output, and unsafe output fall back deterministically
- real LLM / hosted dependency is not required by default tests
- the composer does not make tool-execution decisions
- the composer does not generate Tableau MCP queries
- the composer does not generate arbitrary queries

## Final deterministic baseline posture

The deterministic baseline remains the steady-state path:

- if LLM composition is disabled or not configured, the deterministic composer is used
- if LLM composition fails, times out, or produces invalid or unsafe output, the deterministic composer is used
- required limitation notes are preserved
- fallback reasons remain safe codes
- raw errors and stack traces are not surfaced to users

## Final no-network posture

The milestone keeps the following posture intact:

- default tests remain no-network
- fake composer behavior supports validation without real LLM access
- hosted / real LLM integration tests remain optional or gated
- secrets are not required for default validation
- CI does not require a real LLM or hosted dependency
- no default hosted CI change was introduced

## selected_mark_explanation posture

The `selected_mark_explanation` path remains protected:

- deterministic behavior is unchanged
- output shape is unchanged
- selected-mark raw data, row values, and field values are not passed to LLM material
- the optional composer does not force selected-mark explanations through LLM composition
- regression posture remains intact

## metadata_discovery posture

The `metadata_discovery` path remains structured and safe:

- safe response material can be used by the optional LLM composer
- raw `describeDatasource` output is not passed to the composer
- raw `listFields` output is not passed to the composer
- truncation, permission, and limitation notes are preserved
- unknown counts and permissions remain unknown
- no arbitrary query, underlying data, field values, or row data are introduced
- `listFields` execution remains deferred unless explicitly implemented later

## Legacy fallback posture

The legacy free-form chat fallback remains available:

- `ChatService` full replacement was not done
- `runLightweightAgentLoop` retirement was not done
- LLM composition does not force broad free-form chat replacement
- non-discovery requests continue to use existing fallback behavior where appropriate

## Safety boundary

v0.10.0 preserves the following:

- no raw MCP tool exposure
- no raw MCP output exposure
- no raw transport output exposure
- no raw Tableau response exposure
- no arbitrary query generation
- no arbitrary query execution
- no underlying data access
- no field values or row data retrieval
- no sample values or domain values retrieval
- no write-capable tools
- no tokens, secrets, or stack traces in user-facing output or traces
- evidence-bounded response material only
- required limitation notes preserved
- unknown remains unknown
- fallback to deterministic composer on failure, timeout, or unsafe output
- deterministic ResponseComposer remains baseline
- `selected_mark_explanation` remains protected
- legacy free-form chat fallback remains available
- Exploration Session remains deferred
- AgentCore remains deferred
- default tests remain no-network
- hosted / real LLM / real session-store integration remains optional or gated

## What remains out of scope

The following remain out of scope:

- Exploration Session
- session state / memory
- AgentCore
- broad Hosted MCP exposure
- production Hosted MCP migration
- OAuth full implementation
- token storage / refresh
- raw MCP tool exposure
- raw MCP output exposure
- arbitrary query generation and execution
- underlying data access
- field values / row data retrieval
- sample values / domain values retrieval
- write-capable tools
- broad legacy chat replacement
- frontend redesign
- default hosted / real LLM CI

## Remaining risks

The main remaining risks are:

- structural output validation may not catch every unsupported claim
- a real provider may behave differently from the no-network fake path
- prompt and material shaping may need tuning after real usage
- evidence discipline must stay enforced as composition broadens
- future application to `selected_mark_explanation` needs extra care
- future real LLM integration still needs timeout, cost, IAM, and observability decisions
- fake composer behavior may differ from a real provider
- Exploration Session remains unresolved if workflow continuity becomes more important later
- `ChatService` and `runLightweightAgentLoop` remain in place

## v0.11.0 handoff

The next milestone should begin with a planning issue that decides among the next major directions.

Candidates include:

1. Production-grade LLM provider integration
2. Broader LLM composition coverage
3. Exploration Session revisit
4. Legacy `ChatService` / `runLightweightAgentLoop` reduction
5. AgentCore revisit

Before starting implementation, v0.11.0 should choose the next direction explicitly and keep the no-network baseline and safety boundaries intact.

## AgentCore decision

AgentCore remains deferred.

v0.10.0 did not require a broader agent runtime to deliver the milestone boundary.

Revisit only if:

- runtime complexity increases materially
- cross-run session state becomes a product requirement
- observability is no longer sufficient
- timeout constraints become a real bottleneck
- cost / IAM / deployment trade-offs are clearer
- a no-network AgentCore stub can still be maintained

v0.10.0 is complete as Branch A: LLM ResponseComposer.

Branch B: Exploration Session remains deferred.

The final safety boundary is intact, and the next step is v0.11.0 planning rather than more v0.10.0 scope.
