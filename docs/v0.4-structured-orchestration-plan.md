# v0.4.0 Structured Orchestration Plan

## 1. Purpose

v0.4.0 is the phase where the Tableau context-first frontend from v0.3.0 starts feeding a structured backend orchestration path.

The main goal is to move from "LLM-guided flow control" toward a deterministic orchestration shape:

`Intent -> Plan -> Tools -> Response`

The first safe path we want to unlock is the selected-mark action suggestion flow:

`AI Context Preview Panel -> Explain this selection -> selected_mark_explanation intent -> fixed plan -> tools/context -> response`

## 2. Current state

### Backend

- `ChatService` is still the main backend entry point for chat requests.
- `runLightweightAgentLoop()` still drives the current agent loop.
- `LambdaAgentRunner` exists as a wrapper, but it is not yet connected to the active job flow.
- v0.2 agent foundations already exist:
  - agent core types
  - trace schema
  - `AgentRunner`
  - fixed plan definitions
  - `LambdaAgentRunner`
- Tableau MCP planning and execution remain embedded in the existing service flow.

### Frontend

- The frontend now has a `ContextPreviewModel`.
- Filters, parameters, selected marks, and summary data preview are already collected and shown in the UI.
- `MarkSelectionChanged` refreshes the preview state.
- `AIContextPreviewPanel` is read-only and visible in the app.
- Selected mark action suggestions are shown and currently only prefill the chat input.

### Important boundary

v0.3.0 completed the "Tableau context first" layer, but v0.4.0 orchestration is still not wired in.

## 3. Problems to solve

- Intent is still implicit inside the current chat flow.
- Plan selection is still entangled with lightweight agent loop logic.
- Tool routing is still not a distinct boundary.
- Execution and response composition are not cleanly separated.
- Trace exists, but orchestration visibility is still fragmented.
- Frontend suggestions do not yet map into a backend orchestration request.
- The current system still relies on a hand-shaped flow that will be difficult to extend safely.

## 4. Target architecture

The intended v0.4.0 shape is:

1. Frontend captures preview state and user action intent.
2. Backend resolves the request intent deterministically where possible.
3. A fixed plan is selected for that intent.
4. The execution engine follows the plan with explicit budget and trace tracking.
5. Tools are routed only when the plan allows them.
6. A response composer turns execution results into the final answer.

The architecture should remain compatible with the current `ChatService` path during migration.

## 5. Component responsibilities

### 5.1 IntentResolver

Responsibility:

- Resolve the user request into a stable backend intent.
- Prefer deterministic resolution from UI action metadata when available.
- Provide a fallback when the request cannot be resolved safely.

Input:

- user message
- frontend action metadata
- ContextPack or preview-derived context
- optional hints from the current chat request

Output:

- resolved intent
- confidence
- reason
- fallback hint when unresolved

Can depend on:

- context preview metadata
- fixed intent vocabulary
- context pack

Must not depend on:

- execution engine internals
- tool routing side effects
- UI rendering

v0.4.0 minimum:

- deterministic resolution for `selected_mark_explanation`

v0.5.0+:

- broader intent classification
- richer fallback policies
- optional LLM-backed classification

### 5.2 PlanBuilder

Responsibility:

- Convert an intent into a fixed or partially fixed execution plan.
- Describe required context packs, allowed tools, execution steps, budget, and response strategy.

Input:

- resolved intent
- context pack
- orchestration config

Output:

- selected plan
- fallback plan when unsupported
- plan metadata for trace/debug

Can depend on:

- fixed plan definitions
- context pack contracts
- orchestration config

Must not depend on:

- tool execution results
- frontend state management
- response rendering

v0.4.0 minimum:

- plan selection for `selected_mark_explanation`
- explicit unsupported fallback

v0.5.0+:

- multi-plan branching
- tool registry awareness
- plan optimization

### 5.3 ToolRouter

Responsibility:

- Decide which tools may be invoked for a plan.
- Record skipped tools and precondition failures.
- Keep tool execution decisions separate from plan selection.

Input:

- selected plan
- context pack
- available tool descriptors

Output:

- allowed tool calls
- skipped tool records
- precondition failures

Can depend on:

- tool descriptors
- plan tool policy
- context pack

Must not depend on:

- response composition
- frontend UI
- plan creation logic

v0.4.0 minimum:

- skeleton router that can represent allowed, skipped, and blocked tools

v0.5.0+:

- tool registry
- tool schemas
- precondition registry

### 5.4 ExecutionEngine

Responsibility:

- Execute a selected plan step by step.
- Track budget usage.
- Capture orchestration trace events.
- Stop safely on failure, timeout, or budget exhaustion.

Input:

- plan
- context pack
- tool router
- run budget
- trace sink

Output:

- execution result
- updated trace
- warnings
- error state

Can depend on:

- plan execution contract
- tool router
- trace sink
- budget config

Must not depend on:

- frontend UI
- current chat service fallback logic
- unrelated orchestration layers

v0.4.0 minimum:

- a skeleton engine that can run the `selected_mark_explanation` fixed plan

v0.5.0+:

- more than one intent
- richer step orchestration
- retries and recovery policies

### 5.5 ResponseComposer

Responsibility:

- Turn orchestration output into the final answer payload.
- Keep response shaping separate from execution.

Input:

- execution result
- plan metadata
- context pack

Output:

- answer text
- response metadata
- optional patch or artifact

Can depend on:

- execution result
- plan response strategy

Must not depend on:

- tool execution internals
- frontend state

v0.4.0 minimum:

- compose a simple response for the selected-mark explanation path

v0.5.0+:

- multiple response strategies
- richer artifact composition

### 5.6 OrchestrationTrace

Responsibility:

- Keep orchestration steps observable.
- Provide a backend trace that can later feed UI debug views or persistence.

Input:

- intent resolution events
- plan selection events
- tool routing events
- execution events
- response composition events

Output:

- trace events with `agentRunId`
- step and run status
- warnings and errors

Can depend on:

- trace schema from v0.2
- orchestration execution lifecycle

Must not depend on:

- UI-specific rendering
- tool implementations

v0.4.0 minimum:

- events for resolved intent, selected plan, tool decisions, budget usage, fallback, and final status

v0.5.0+:

- persistent trace storage
- detailed frontend trace views

### 5.7 RunBudget

Responsibility:

- Put a hard cap on orchestration work.

Fields:

- `maxModelCalls`
- `maxToolCalls`
- `timeoutMs`

Where it should live:

- plan definition for the default budget shape
- execution input for run-specific overrides
- execution engine for enforcement
- optional orchestration config for global defaults

v0.4.0 minimum:

- define budget on the selected fixed plan
- pass the budget through the run input
- enforce the budget in the execution engine

## 6. Minimal v0.4.0 scope

The smallest useful v0.4.0 slice is:

1. Accept a frontend action request for `selected_mark_explanation`.
2. Resolve the intent deterministically.
3. Select the fixed plan for that intent.
4. Execute the plan through a minimal engine.
5. Route only the tools the plan allows.
6. Track trace events and budget usage.
7. Return a composed response.

This should be done without replacing the full existing chat flow.

## 7. Out of scope

The following stay out of v0.4.0:

- full replacement of `runLightweightAgentLoop`
- full replacement of `ChatService`
- broad multi-intent orchestration
- full Tool Registry implementation
- AgentCore migration
- LLM-driven intent classification as the primary entry
- large frontend UX changes
- wide tool schema redesign

## 8. selected_mark_explanation path

The first orchestration path should be:

1. User clicks "Explain this selection" in the preview panel.
2. The frontend sends an explicit orchestration request.
3. Backend resolves `selected_mark_explanation`.
4. `PlanBuilder` returns the fixed plan for that intent.
5. `ExecutionEngine` runs the minimal fixed-plan steps.
6. `ToolRouter` only allows the tools required by that plan.
7. `ResponseComposer` produces the answer.
8. Trace captures the full run lifecycle.

This path should remain deterministic and small.

## 9. Frontend action to backend orchestration flow

Recommended v0.4.0 flow:

1. `AIContextPreviewPanel` emits a user-intent action.
2. `App` forwards that action through a dedicated orchestration request path.
3. Backend receives both the action metadata and the current context snapshot.
4. Backend resolves the intent without requiring a separate LLM classification step.
5. Orchestration runs in the structured pipeline.

Important rule:

- the action click should be user-driven
- it should not auto-send a normal chat job
- it should not silently invoke AI

## 10. Migration strategy from current flow

Migration should be incremental.

### Phase 1

- add orchestration contracts
- define intent, plan, trace, and budget shapes
- keep existing chat behavior intact

### Phase 2

- connect selected-mark action suggestions to the orchestration entry
- run one deterministic intent path

### Phase 3

- decide whether the current chat job flow should call the orchestrator facade
- keep `runLightweightAgentLoop` as a fallback until confidence is high

### Phase 4

- expand beyond the single fixed plan path
- move more work into structured orchestration

## 11. Trace-first design

The trace should answer these questions:

- what intent was resolved?
- what plan was selected?
- what context was used?
- what tools were allowed, used, or skipped?
- how much budget was consumed?
- did we fall back?
- did execution fail?
- how was the final response composed?

The goal in v0.4.0 is backend trace clarity first, not a new trace UI.

## 12. Risks and constraints

- The current chat flow is already working, so migration must avoid regressions.
- selected-mark suggestions are currently UI-only, so backend orchestration entry must be added carefully.
- the first orchestration path should stay narrow to reduce debug complexity.
- trace volume can grow quickly if every step is logged without structure.
- budget enforcement must be deterministic or the new boundary will be hard to trust.

2. Define orchestration contracts.
3. Add trace and budget structures.
4. Implement the minimal deterministic intent resolver.
5. Implement the minimal fixed-plan execution path.
6. Connect the frontend action to the orchestration entry.
7. Validate trace visibility and fallback behavior.
8. Wrap up and document the v0.5 boundary.

## 15. Open questions

- Should selected-mark action suggestions invoke a dedicated orchestration endpoint or reuse a generalized job endpoint?
- Should orchestration runs and chat jobs share the same persistence model?
- How should `agentRunId` relate to chat job IDs?
- Should `runLightweightAgentLoop` remain a fallback for v0.4.0, or only for unsupported paths?
- What is the smallest trace schema needed to make debugging trustworthy?
- Should plan selection be entirely deterministic for v0.4.0, or may it include a narrow heuristic fallback?
