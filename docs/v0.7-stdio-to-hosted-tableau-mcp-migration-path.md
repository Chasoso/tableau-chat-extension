# stdio to Hosted Tableau MCP Migration Path

## 1. Purpose

This document defines the staged migration path from the current stdio Tableau MCP setup toward a Hosted / remote Tableau MCP setup.

It ties together the v0.7.0 foundation work so the team can:

- move from the current stdio baseline without breaking legacy behavior
- keep the Tool Layer transport-agnostic
- preserve fake/no-network execution for local development and CI
- retain stdio fallback during the migration window
- make the v0.8.0 Hosted Tableau MCP Metadata Execution handoff explicit

This is a documentation-only milestone artifact. It does not implement Hosted MCP production migration, OAuth, transport adapters, or infrastructure changes.

## 2. Background

The project moved through these earlier milestones:

- v0.6.0 kept AgentCore as deferred partial adoption
- v0.7.0 established the Hosted Tableau MCP Migration Foundation
- v0.7.0 defined the transport, auth, metadata, governance, and trace boundaries needed for later hosted execution

Current assumptions remain:

- Tableau MCP is still stdio-based in the live path
- free-form chat still uses the legacy `ChatService` / `runLightweightAgentLoop` path
- `selected_mark_explanation` still runs through the structured Tool Layer path
- read-only metadata tools are the first migration target
- arbitrary query, underlying data access, and write tools remain out of scope
- AgentCore is not required for this migration path

## 3. Current stdio Tableau MCP path

The current path still centers on local stdio execution.

```text
Current:
free-form chat / legacy path
↓
ChatService / runLightweightAgentLoop
↓
current Tableau MCP usage
↓
stdio Tableau MCP server
↓
Tableau Cloud
```

Key current assumptions:

- the backend or Lambda layer can launch or use a local stdio MCP process
- auth and token handling are backend-owned
- process lifecycle, timeout handling, and stderr diagnostics are tied to stdio behavior
- legacy chat can rely on MCP-derived context during orchestration
- `selected_mark_explanation` must remain stable during migration work

Current limitations:

- transport details are coupled to local process assumptions
- hosted endpoint readiness is unknown
- auth / user-context / site-settings boundaries are not yet unified across transports
- the current path is not suitable as the only future hosted strategy

The v0.7 metadata path is intentionally separate:

```text
v0.7 metadata path:
ToolRegistry
↓
metadata tool handler
↓
precondition evaluator
↓
transport-aware execution boundary
↓
fake/no-network transport
↓
normalized metadata result
```

## 4. Target Hosted Tableau MCP path

The target candidate is a hosted or remote transport path that keeps the Tool Layer stable while transport becomes swappable.

```text
Target candidate:
metadata tool request
↓
ToolRegistry
↓
metadata tool handler
↓
precondition evaluator
↓
transport config resolver
↓
transport-aware execution boundary
↓
HostedTableauMcpTransport
↓
Hosted Tableau MCP on Tableau Cloud
↓
normalized metadata result
↓
ResponseComposer / user-facing response
```

The target path assumes:

- OAuth / user-context / site-settings decisions are explicit
- transport selection is a first-class boundary
- normalized result and trace summaries remain safe for downstream consumers
- stdio fallback remains available until the hosted path is trusted
- fake/no-network behavior continues to exist for CI and local development

## 6. Migration principles

The migration must follow these principles:

- Hosted MCP is explicit opt-in
- fake/no-network remains first-class
- stdio fallback remains available during migration
- the Tool Layer must not know transport internals
- raw MCP tools must not be exposed directly
- app-specific wrapper tools only
- read-only metadata first
- no arbitrary query execution
- no underlying data access
- no write-capable tools
- no raw MCP result exposure
- no tokens, secrets, or raw stack traces in results or trace payloads
- CI must not require Hosted MCP
- local development must work without Hosted MCP
- `selected_mark_explanation` must remain unchanged
- free-form chat legacy behavior must remain unchanged in v0.7

## 7. Staged migration plan

### Stage 0: Current baseline

- stdio MCP remains the current live path
- free-form chat remains on the legacy path
- metadata tools are not yet used for free-form chat
- no Hosted MCP dependency is required

### Stage 1: v0.7 foundation

- transport contract exists
- config resolver exists
- auth boundary is documented
- metadata tools are defined
- fake/no-network handlers exist
- transport-aware metadata execution exists
- normalized output and trace events exist

### Stage 2: v0.8 gated Hosted metadata execution

- implement `HostedTableauMcpTransport` or a remote transport adapter
- keep fake/no-network tests
- keep stdio fallback
- enable hosted execution only via explicit config or feature flag
- start with `describeDatasource` or `listFields`
- no arbitrary query
- no underlying data access

### Stage 3: limited hosted metadata validation

- validate auth / user context
- validate site settings
- validate timeout / error normalization
- validate trace payload shape
- validate permission-denied behavior
- compare fake / stdio / hosted outputs
- keep rollout manual and gated

### Stage 4: structured data discovery candidate

- move selected discovery use cases into a structured path
- introduce or use a `metadata_discovery` intent later
- keep legacy chat fallback
- do not replace `ChatService` yet

### Stage 5: broader migration decision

- decide whether to reduce stdio MCP usage
- decide whether Hosted MCP becomes the default
- decide whether free-form discovery migrates
- revisit AgentCore only if session or observability needs justify it

## 8. Configuration and feature flag plan

Transport selection remains the key control point.

Supported modes from v0.7 are:

- `auto`
- `stdio`
- `hosted`
- `remote`
- `fake`
- `disabled`

Recommended rules:

- hosted remains default off
- hosted must be explicitly enabled
- CI should default to fake
- local development should default to fake or stdio
- production should only use hosted when enabled and configured
- no silent hosted adoption
- no hosted execution in CI unless explicitly gated
- misconfiguration should fail safely or fall back only when allowed

| Mode | Local | CI | Production | Notes |
| --- | --- | --- | --- | --- |
| `auto` | fake or stdio | fake | hosted only when explicitly enabled and configured | safest general-purpose mode |
| `stdio` | allowed | only when explicitly tested | allowed during migration | current baseline |
| `hosted` | explicit opt-in | gated only | allowed when configured | never silent |
| `remote` | explicit opt-in | gated only | allowed when configured | generic remote candidate |
| `fake` | recommended default | recommended default | not default | no-network safe mode |
| `disabled` | allowed | allowed | allowed | no Tableau MCP transport |

Candidate configuration knobs:

- `TABLEAU_MCP_TRANSPORT_MODE`
- `TABLEAU_MCP_HOSTED_ENABLED`
- `TABLEAU_MCP_HOSTED_ENDPOINT`
- `TABLEAU_MCP_HOSTED_AUTH_MODE`
- `TABLEAU_MCP_STDIO_ENABLED`
- `TABLEAU_MCP_FAKE_ENABLED`
- `TABLEAU_MCP_NO_NETWORK`
- `TABLEAU_MCP_ALLOW_STDIO_FALLBACK`
- `TABLEAU_MCP_ALLOW_FAKE_FALLBACK`
- `TABLEAU_MCP_ALLOW_HOSTED_IN_CI`

## 9. Fallback and rollback plan

| Failure scenario | Expected behavior | Fallback / rollback | User-facing note |
| --- | --- | --- | --- |
| hosted not configured | safe not_configured result | fall back to stdio or fake if allowed | hosted is not yet ready |
| hosted auth missing | safe auth required result | fall back to stdio or fake if allowed | authentication is required |
| hosted timeout | safe timeout result | retry only if policy allows, otherwise stdio/fake | try again later |
| hosted permission denied | safe permission denied result | do not escalate privileges automatically | access is not sufficient |
| site settings disabled | blocked result | no hosted execution | feature is disabled for the site |
| stdio unavailable | safe failure result | use fake in local / CI when allowed | current stdio path is unavailable |
| fake path only | deterministic success / partial / blocked result | no network fallback needed | no-network mode active |

Rollback guidance:

- disable hosted via feature flag
- revert to stdio while migration is validated
- revert to fake in local development and CI if hosted readiness regresses
- keep selected metadata tools isolated from the legacy chat path

## 10. Local development plan

Local development should remain predictable and low-friction.

Recommended default:

- fake / no-network or stdio, depending on developer setup

Local plan:

- hosted is explicit opt-in only
- normal unit tests must not need OAuth
- fake user context can be used
- fake token reference can be used
- fake site settings can be used
- fake metadata output must remain deterministic
- stdio fallback can be used by developers who have the local server configured
- local hosted checks should be optional and manual
- no secrets should be required for unit tests

## 11. CI and no-network plan

CI should remain stable even if Hosted MCP is unavailable.

Recommended CI policy:

- default to fake / no-network
- no Hosted MCP requirement
- no OAuth requirement
- no secrets requirement
- no stdio server requirement
- no Tableau Cloud requirement
- contract tests should use fake transport or in-memory stubs
- hosted integration tests should be optional and gated
- metadata registration, execution boundary, normalization, and trace events should all be testable without network access

## 13. Metadata tool migration scope

The v0.7 foundation intentionally narrows the hosted migration to read-only metadata tools:

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`

In scope:

- app-specific wrapper tools
- ToolRegistry registration
- fake / no-network handlers
- transport-aware execution boundary
- normalized output
- trace events

Not in scope:

- arbitrary query
- underlying data
- write tools
- raw MCP tools
- raw MCP result exposure

### Recommended first hosted tool

| Candidate | Pros | Cons | Recommendation |
| --- | --- | --- | --- |
| `describeDatasource` first | small output, summary-first, easier to bound | may still need datasource resolution | Recommended first |
| `listFields` first | visible user value | larger output and truncation concerns | Second |
| both together | faster surface coverage | larger auth / output / trace surface | Avoid unless hosted path is already stable |

## 14. Safety and governance boundaries

The following boundaries should remain in force after v0.8.0 as well:

- raw MCP tool exposure is forbidden
- arbitrary query execution is forbidden
- underlying data access is forbidden
- write-capable tools are forbidden
- read-only metadata comes first
- authenticated Tableau context is required
- datasource ambiguity blocks execution
- site settings disabled blocks execution
- permission denied is handled safely
- `safeForPreviewOnly` remains part of policy
- budget and timeout preconditions remain explicit
- output truncation must remain explicit
- no secrets, tokens, or raw stack traces in results or trace
- JSON-safe output only
- fake / no-network tests remain required

## 15. Observability and trace handoff

The metadata path now has trace candidates for safe handoff:

- `tableau_metadata_tool.started`
- `tableau_metadata_tool.completed`
- `tableau_metadata_tool.failed`

Trace payload expectations:

- `requestId`
- `correlationId`
- `agentRunId`
- `toolName`
- `transportKind`
- `durationMs`
- `warningCount`
- `errorCode`
- `fakeNoNetwork`

Trace must not include:

- raw MCP result
- raw transport internals
- tokens
- secrets
- stack traces
- stdout / stderr

UI trace panels and trace persistence are out of scope for v0.7.0, but the payload shape is now ready for hosted metadata execution observability in v0.8.0.

## 16. v0.8.0 handoff

v0.8.0 should be the first milestone that implements Hosted Tableau MCP metadata execution behind a gate.

Recommended v0.8.0 theme:

```text
v0.8.0 Hosted Tableau MCP Metadata Execution
```

Recommended v0.8.0 goal:

```text
Implement a gated Hosted / remote Tableau MCP metadata execution path for the first read-only metadata tool, while preserving fake/no-network tests and stdio fallback.
```

Recommended v0.8.0 scope:

- implement `HostedTableauMcpTransport` or `RemoteTableauMcpTransport`
- connect `describeDatasource` first
- keep `listFields` fake or add later
- use explicit hosted feature flags
- verify OAuth / user-context / site-settings boundaries
- keep CI fake / no-network
- normalize hosted errors into existing metadata error summaries
- preserve selected_mark_explanation and free-form chat legacy behavior

Recommended v0.8.0 out of scope:

- arbitrary query execution
- underlying data access
- write tools
- raw MCP exposure
- free-form chat full migration
- AgentCore implementation
- default LLM ResponseComposer enablement

## 17. Risks and mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Hosted MCP auth details remain unclear | v0.8.0 blocked | keep fake / no-network and defer hosted execution |
| OAuth / user-context mismatch | security or permission risk | require #131 checkpoints before real execution |
| Hosted endpoint unavailable | runtime failure | feature flag and rollback to stdio / fake |
| Site settings disabled | metadata tools fail | precondition block with safe user-facing message |
| Output too large | unsafe response or trace | normalization, truncation, omission signals |
| Raw MCP output leaks | governance risk | normalizer allowlist and tests |
| CI depends on Hosted MCP | instability | fake / no-network remains required |
| Free-form chat changes accidentally | user regression | keep legacy path untouched |
| `selected_mark_explanation` changes | regression | maintain regression tests |
