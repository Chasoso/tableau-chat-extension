# Hosted MCP Auth Context Adapter Boundary

## 1. Purpose

This document fixes the safe auth / user-context boundary for v0.8.0 Hosted Tableau MCP execution.

Its purpose is to:

- define a safe auth context shape for Hosted execution
- preserve the tokenReference handoff boundary without exposing raw tokens
- map app auth and Tableau user context into a Hosted-safe summary
- define how missing, expired, unknown, and not_configured auth states are represented
- keep trace and result payloads free of raw credentials
- provide the boundary needed before `#161` connects `describeDatasource` to Hosted transport

## 2. Background

v0.7.0 established the Hosted migration foundation:

- auth / user-context / site-settings boundary
- transport abstraction and transport config strategy
- read-only metadata tool definitions
- metadata preconditions and governance
- fake / no-network handlers
- transport-aware execution boundary
- output normalization and trace events

v0.8.0 adds the first Hosted transport skeleton in `#159`, and this issue defines the auth boundary that can be safely passed into that skeleton.

OAuth flow, token storage, and Hosted MCP network connection remain out of scope.

## 3. Adapter responsibilities

The adapter must:

- accept app auth and Tableau user context summaries
- produce a safe auth context for Hosted transport
- preserve tokenReference as an opaque handoff only
- map missing, expired, unknown, and not_configured auth states into safe values
- produce a userContext summary that is safe to serialize
- strip or redact raw credential fields from metadata
- generate trace-friendly metadata without raw tokens or secrets

## 4. Non-goals

This boundary does not:

- start an OAuth flow
- obtain or refresh tokens
- store or persist tokens
- resolve tokenReference into a raw token
- connect to Hosted MCP
- call Tableau REST APIs
- read secrets from env, files, DBs, or secret managers
- replace the existing metadata precondition system
- change legacy chat behavior

## 5. Safe auth context shape

The safe auth context is intentionally minimal.

```ts
type HostedMcpSafeAuthContext = {
  state: "ready" | "missing" | "expired" | "unknown" | "not_configured";
  mode: "oauth_delegated" | "token_reference" | "direct_trust" | "fake" | "unknown";
  tokenReference?: {
    kind: "token_reference";
    referenceId: string;
    expiresAt?: string;
    scopes?: readonly string[];
    source?: "oauth" | "connected_app" | "fake" | "unknown";
  };
  userActionRequired?: boolean;
  retryable?: boolean;
  reasonCode?:
    | "AUTH_REQUIRED"
    | "AUTH_EXPIRED"
    | "AUTH_STATE_UNKNOWN"
    | "HOSTED_AUTH_NOT_CONFIGURED"
    | "SITE_SETTINGS_DISABLED"
    | "TOKEN_REFERENCE_MISSING";
  message?: string;
  metadata?: Record<string, unknown>;
};
```

Raw access tokens, refresh tokens, authorization headers, and client secrets must never appear in this structure.

## 6. User context summary mapping

The user context summary should keep only safe identity and site fields.

```ts
type HostedMcpUserContextSummary = {
  userId?: string;
  tableauUserId?: string;
  email?: string;
  siteId?: string;
  siteName?: string;
  siteContentUrl?: string;
  locale?: string;
  authMode?: "oauth_delegated" | "token_reference" | "direct_trust" | "fake" | "unknown";
  metadata?: Record<string, unknown>;
};
```

This summary is for serialization and traceability only. It must not include raw credentials.

## 7. TokenReference handoff boundary

`tokenReference` is an opaque identifier, not a token.

Allowed:

- pass the reference id to Hosted transport as a handoff marker
- include safe metadata such as `expiresAt` and `scopes`
- mask or omit the reference in trace output if needed

Not allowed:

- resolving the reference into a raw token in this boundary
- logging the reference value as if it were a secret
- exposing access tokens, refresh tokens, or authorization headers

## 8. Auth state mapping

The adapter should map the following states:

- `ready` when authentication is present and sufficiently described
- `missing` when authentication is absent or incomplete
- `expired` when the tokenReference is known to be expired
- `unknown` when the available information is insufficient
- `not_configured` when Hosted execution is not enabled for the site or auth setup is unavailable

These states are safe to surface to the transport boundary and to contract tests.

## 9. Missing auth behavior

Missing auth should be treated as a recoverable state.

Expected behavior:

- `state: "missing"`
- `reasonCode: "AUTH_REQUIRED"` or `TOKEN_REFERENCE_MISSING`
- `userActionRequired: true`
- `retryable: true`
- safe message only

## 10. Expired auth behavior

Expired auth should be treated as recoverable but blocked until reauth.

Expected behavior:

- `state: "expired"`
- `reasonCode: "AUTH_EXPIRED"`
- `userActionRequired: true`
- `retryable: true`
- no refresh attempt in this boundary

## 11. Unknown auth behavior

Unknown auth should remain explicit instead of being guessed.

Expected behavior:

- `state: "unknown"`
- `reasonCode: "AUTH_STATE_UNKNOWN"`
- `retryable` may remain true if the calling layer wants a retry path
- no throw
- no raw credential disclosure

## 12. not_configured behavior

When Hosted execution is not configured, the adapter should surface:

- `state: "not_configured"`
- `reasonCode: "HOSTED_AUTH_NOT_CONFIGURED"` or `SITE_SETTINGS_DISABLED`
- safe message
- recoverable only if the caller can actually enable Hosted execution

## 13. Token and secret suppression

The adapter must suppress secret-like keys in metadata and trace payloads.

Examples of fields that must not appear in output:

- `accessToken`
- `refreshToken`
- `idToken`
- `authorization`
- `authorizationHeader`
- `clientSecret`
- `secret`
- `secrets`
- `token`
- `rawToken`
- `jwt`
- `password`
- `cookie`

## 14. Trace and result safety

Trace and result payloads must remain JSON-safe and credential-free.

Allowed trace fields:

- request ids
- correlation ids
- agent run ids
- auth state
- auth mode
- user ids and site ids when needed
- tokenReference presence or masked summary

Not allowed:

- raw tokens
- raw auth headers
- raw stack traces
- raw MCP responses
- secret values

## 15. Relationship with HostedTableauMcpTransport

`HostedTableauMcpTransport` should consume the safe auth summary only.

This boundary gives the transport enough information to:

- recognize missing or expired auth safely
- avoid raw token handling
- keep the default path no-network
- remain compatible with `not_configured` first behavior

## 16. Relationship with metadata preconditions

The adapter complements, but does not replace, metadata preconditions.

The precondition layer still decides whether a metadata action can execute.
This adapter only prepares safe auth and user-context data for hosted execution paths.

## 17. Relationship with integration test gating

Hosted auth adapter tests should remain no-network and default-safe.

`#158` keeps Hosted integration tests optional and gated, while this boundary keeps auth summary tests fully local and deterministic.

## 18. Recommendations for #161 and #162

For `#161`:

- accept the safe auth context and user context directly from this boundary
- do not require raw tokens
- keep `describeDatasource` behind a feature flag

For `#162`:

- map auth-related failures into safe metadata error summaries
- keep raw credential details out of transport errors

## 19. Open questions

- Should `tokenReference.referenceId` stay visible in trace metadata, or be masked further?
- Which auth state should the transport treat as `not_configured` versus `missing`?
- Do we need a dedicated `locale` field in the transport request contract?
- Should `siteContentUrl` stay in the transport-facing user context summary?
- Will `#161` require any additional auth metadata beyond this boundary?
