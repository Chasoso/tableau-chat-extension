# v0.9.0 Wrap-up

## Summary

v0.9.0 closed the loop on structured metadata discovery for the Tableau chat extension.

The milestone delivered a narrow structured discovery surface, preserved the legacy free-form chat fallback, and kept `selected_mark_explanation` unchanged.

The result is a safe, auditable boundary that can support a later decision between LLM ResponseComposer and an Exploration Session without reopening the v0.8.0 safety decisions.

## Milestone outcome

The milestone progressed through the following sequence:

- #182 established the planning and structured discovery boundaries
- #183 defined the `metadata_discovery` intent
- #184 defined the datasource / workbook / view ambiguity model
- #185 defined clarification response contracts
- #186 defined the structured metadata discovery plan shape
- #187 connected safe datasource discovery cases to `describeDatasource`
- #188 defined the controlled `listFields` discovery contract
- #189 added truncation and permission-aware discovery response shaping
- #190 reduced legacy chat coverage for metadata discovery cases
- #191 protected `selected_mark_explanation` and the free-form chat fallback
- #192 documented the structured discovery result

Taken together, those issues converted a broad discovery space into a narrow, safe, regression-tested metadata discovery surface.

## Actual boundary reached

v0.9.0 reached the following actual boundary:

- structured discovery is limited to metadata discovery
- executable structured discovery is limited to eligible datasource metadata cases
- `describeDatasource` remains the first validated hosted metadata boundary
- `listFields` remains a controlled candidate, not an execution surface
- workbook / view resolution remains limited and deferred
- free-form chat replacement did not happen
- arbitrary query execution did not happen
- underlying data access did not happen
- write tools did not happen

This boundary is intentionally narrow.

## Structured discovery result

The structured discovery result is documented in `docs/v0.9-structured-discovery-result.md`.

At a high level:

- eligible discovery cases moved into a structured path
- unsupported and ambiguous cases remained on the legacy path
- `metadata_discovery` became the structured orchestration intent
- clarification, plan shape, and hosted routing were defined explicitly
- `describeDatasource` was reused instead of introducing a new raw MCP surface
- `listFields` stayed deferred as a controlled candidate

## Legacy chat reduction result

Legacy chat was reduced, not removed.

The final state is:

- eligible metadata discovery cases use structured orchestration
- unsupported, ambiguous, and non-discovery cases stay on legacy fallback
- `ChatService` remains in place
- `runLightweightAgentLoop` remains in place
- no broad free-form chat replacement occurred

## Regression and safety result

## Remaining risks

The milestone intentionally leaves a narrow surface in place.

Remaining risks include:

- structured discovery stays narrow by design
- workbook / view resolution remains limited and deferred
- `listFields` remains contract-first and execution-deferred
- permission-aware behavior still depends on transport and Tableau permission behavior
- the structured / legacy split will continue to need regression coverage
- future LLM composition may drift without evidence discipline
- future session memory may broaden state, privacy, and test complexity

## v0.10.0 decision inputs

v0.9.0 provides the following decision inputs for v0.10.0:

- the natural language quality problem is still present, but the core discovery boundary is now safe and explicit
- the interaction continuity problem is still present, but the current milestone did not require session state
- the routing and execution boundaries are narrow enough to support a later decision without reopening raw MCP exposure
- the current system already has a deterministic baseline and evidence discipline that should not be lost

## Option A: LLM ResponseComposer

This path is the better next step if the main product pressure is output quality.

Potential benefits:

- more natural user-facing phrasing
- better answer composition across structured and fallback paths
- room to improve summaries without widening execution scope

Main risks:

- unsupported claims if composition is not evidence-bounded
- leakage of raw execution details if guardrails are weak
- drift away from deterministic baselines

Required boundaries:

- deterministic baseline remains the default
- composition stays optional and guarded
- response material stays evidence-bounded
- no raw MCP payloads
- no raw transport output
- no arbitrary query generation
- no underlying data access

## Option B: Exploration Session

This path is the better next step if the main product pressure is multi-step continuity.

Potential benefits:

- better continuity across related discovery steps
- a more natural exploration flow for metadata questions
- a place to carry safe state across short interactions

Main risks:

- session state complexity
- privacy and persistence concerns
- stale or misleading context
- harder no-network testing

Required boundaries:

- no uncontrolled memory
- no sensitive raw data persistence
- no underlying data, field values, or row data by default
- no arbitrary query
- no write tools
- no-network tests remain possible
- session lifecycle stays explicit and trace-safe

## AgentCore deferred decision

AgentCore remains deferred.

The v0.9.0 result did not require a broader agent runtime to achieve the milestone boundary.

AgentCore should be revisited only if one or more of the following becomes true:

- multi-step runtime complexity materially increases
- cross-run session or memory becomes a product requirement
- observability needs exceed the current trace contract
- timeout constraints become a real operational bottleneck
- cost, IAM, and deployment trade-offs become clearer

## Recommended next step

The next milestone should start with a planning issue that chooses between LLM ResponseComposer and Exploration Session.

If the main pain is response quality, choose LLM ResponseComposer first.

If the main pain is multi-step continuity, choose Exploration Session first.

If neither is clearly urgent, spend one short planning pass making the decision explicit before expanding implementation scope.
