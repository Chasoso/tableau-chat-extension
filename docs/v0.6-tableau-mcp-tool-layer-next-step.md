# Tableau MCP Tool Layer Next Step

## 1. Purpose

This document defines the next step for Tableau MCP migration after the v0.5.0 Tool Layer rebuild.

The goal is not to implement Tableau MCP migration yet. The goal is to decide the smallest useful migration unit, the registration strategy, and the boundary between v0.6 planning and later implementation.

## 2. Background

v0.5.0 intentionally deferred Tableau MCP migration.

v0.5.0 focused on context-derived pseudo tools:

- `context.selectedMarks`
- `context.summaryDataPreview`
- `context.filters`
- `context.parameters`

v0.6.0 is currently focused on AgentRunner and AgentCore comparison work. Tableau MCP migration should therefore be planned carefully so it does not become scope creep.

## 3. Current Tableau MCP usage audit

The current runtime still treats Tableau MCP as a deferred capability rather than a first-class Tool Layer migration target.

Audit points to keep in mind:

- Tableau MCP planning belongs to the legacy runtime and any existing agent loop logic, not to the selected-mark deterministic path
- the current selected_mark_explanation path does not depend on Tableau MCP
- failure / retry / fallback / timeout handling for Tableau MCP should stay isolated from the deterministic path
- raw MCP results should not be exposed in user-facing responses or trace payloads

The practical conclusion is that Tableau MCP is still a separate capability from the current selected-mark explanation path.

## 5. Tool registration strategy

Recommended strategy:

- do not expose raw MCP tools directly as user-facing ToolDefinitions
- register high-level wrapper tools aligned with app intent and use case
- keep the registry contract stable across LambdaAgentRunner and AgentCoreRunner

Recommended naming style:

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`
- `tableau.metadata.describeCurrentView`
- `tableau.query.previewSummary`

Recommended metadata:

- category: `tableau`
- capability: `read_only`
- safety: `safe_for_preview` or equivalent
- externalAccess: true
- requiresAuthentication: true
- raw data policy: summary only

The rationale is that the app should control the user-visible contract rather than exposing raw MCP granularity.

## 6. Input schema boundaries

The input schema should stay explicit and narrow.

Allowed fields should include only what the tool actually needs:

- siteId or siteName
- workbookId or workbookName
- viewId or viewName
- datasourceId or datasourceName
- worksheetName
- fieldName / fieldNames
- aggregation
- filters
- parameters
- selectedMarks summary
- rowLimit
- columnLimit
- query intent
- reason / user intent
- correlationId / agentRunId

Rules:

- do not accept arbitrary SQL or VizQL from the model
- treat ambiguous identifiers as precondition failures
- default row and column limits must exist
- `allowUnderlyingData` should be false by default
- raw row output should require explicit, high-risk preconditions

## 7. Output schema boundaries

The output schema should be JSON-safe and summary-first.

Required output shape elements should include:

- status
- result summary
- row count
- column count
- field list
- schema summary
- datasource / workbook / view references
- truncated flag
- warning list
- error summary
- permission / access notes
- evidence references
- raw output omitted / included flag
- trace metadata

Rules:

- do not expose raw MCP result blobs directly
- large tables must be summarized
- errors must be normalized
- permission-related data should be summarized, not leaked

## 8. Precondition boundaries

Recommended preconditions:

- authenticated Tableau session
- Connected App / Direct Trust token available
- Tableau site / workbook / view resolved
- datasource identifier resolved when required
- user capability check
- row / column limits present
- budget remaining
- timeout budget available
- no write-capable operation
- explicit confirmation for high-risk operations

Preconditions should be checked before execution whenever possible, and failures should become safe blocked / fallback outcomes rather than raw exceptions.

## 10. Recommended smallest migration unit

Recommended smallest migration unit:

- `tableau.metadata.describeDatasource`

Why:

- it is read-only
- it has low output risk
- it is good for identifier / schema / precondition design
- it does not require arbitrary query generation
- it can be wrapped in a safe ToolDefinition

`tableau.metadata.listFields` is a good second candidate if field discovery is more important in the product roadmap.

## 11. v0.6 or later decision

Recommendation:

- keep #107 as plan only in v0.6
- do not implement Tableau MCP execution in v0.6
- move the actual migration to v0.7 or a later issue unless there is a very narrow, high-confidence read-only contract that clearly supports the current product goals

This keeps v0.6 focused on AgentRunner and AgentCore comparison work.

## 12. Relationship with AgentRunner and AgentCoreRunner

Guidance:

- Tableau MCP tool definitions, schemas, and preconditions should be AgentRunner-independent
- both `LambdaAgentRunner` and `AgentCoreRunner` should be able to use the same Tool Layer abstraction
- AgentCore should not own the Tableau MCP migration decision by itself
- Tableau MCP execution summaries can later be reflected in `AgentRunResult` and comparison metrics

Recommended relationship:

- design the tool contract once
- keep the runtime integration shared
- delay execution migration until the product value is clear

## 13. Risks and constraints

- raw MCP outputs could become too large or too sensitive
- query-style tools could drift into arbitrary query execution if boundaries are weak
- Extension API overlap could make migration unnecessary for some cases
- datasource / workbook / view ambiguity can create noisy precondition failures
- permission and governance rules can make the first migration unit deceptively complex
- adding MCP execution too early could distract from the AgentRunner spike
