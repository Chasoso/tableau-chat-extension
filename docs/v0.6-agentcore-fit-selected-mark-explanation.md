# AgentCore Fit Assessment for selected_mark_explanation

## 1. Purpose

This document assesses whether `selected_mark_explanation` is a good fit for an AgentCore-based runner in v0.6.0.

The goal is not to prove that AgentCore can answer better. The goal is to decide whether AgentCore meaningfully improves operations, observability, and session management for this specific deterministic path compared with the current Lambda-backed runner.

## 2. Current LambdaAgentRunner path

Current path after #103:

```text
explicit selected-mark action
-> AgentRunInput construction
-> LambdaAgentRunner
-> selected_mark_explanation structured orchestration
-> ToolRouter / ToolRegistry / ToolExecutionWrapper
-> ResponseComposer skeleton
-> AgentRunResult normalization
-> existing selected-mark response shape
```

Current characteristics:

- the path is explicit and user-triggered
- `AgentRunInput` is constructed from Tableau context summaries, not raw context bodies
- `LambdaAgentRunner` is a thin wrapper around the existing deterministic orchestration path
- trace, fallback, and budget summaries are already normalized at the runner boundary
- the user-facing response is a deterministic placeholder / context collection summary
- LLM, Bedrock, Tableau MCP, and AgentCore are not part of this path yet

In other words, the current path is already comparable and relatively small.

## 3. selected_mark_explanation characteristics

`selected_mark_explanation` has the following properties:

- it uses a fixed plan
- it is context-derived rather than open-ended
- it centers on pseudo tools, not external action tools
- it already works as a deterministic summary path without LLM generation
- it has a clear required/optional context split
- required context is limited to selected marks
- optional context is limited to summaryDataPreview, filters, and parameters
- its failure modes are simple and explicit
- trace comparison is straightforward
- budget usage is predictable

These properties make it a strong baseline for comparing runtimes, but they also mean the path is not inherently a heavy agentic workload.

## 4. Intended AgentCore shapes

### Option A: AgentCore as a thin runner

```text
API / selected mark action
-> AgentCoreRunner
-> AgentCore runtime
-> custom structured orchestration remains mostly in code
-> AgentRunResult normalization
```

Pros:

- preserves the current deterministic orchestration logic
- keeps the existing Tool Layer and response material model intact
- reduces migration risk
- makes AgentCore a runtime envelope rather than a wholesale rewrite

Cons:

- may not deliver the full value of AgentCore memory / gateway / runtime features
- can still leave most business logic custom
- may create another wrapper layer without much simplification

### Option B: AgentCore owns more orchestration

```text
API / selected mark action
-> AgentCoreRunner
-> AgentCore runtime
-> AgentCore tools / gateway / observability
-> AgentRunResult normalization
```

Pros:

- could increase managed runtime observability
- could reduce custom orchestration code over time
- may fit a future more dynamic agent model

Cons:

- highest implementation and migration risk
- likely overkill for a fixed deterministic path
- makes local debugging and reproducibility harder
- requires more decisions about tools, boundaries, and memory policy

### Option C: Keep LambdaAgentRunner as the comparison baseline

```text
API / selected mark action
-> LambdaAgentRunner
-> custom structured orchestration
-> AgentRunResult
```

Pros:

- lowest risk
- highest reproducibility
- already implemented and working
- best for measuring a baseline before any AgentCore decision

Cons:

- does not provide AgentCore runtime benefits
- leaves observability and session work on the custom stack

## 5. What should remain custom

The following should remain custom for now:

- Tableau Extension context normalization
- selected mark summary policy
- raw data suppression policy
- IntentResolver and fixed plan definitions
- ToolDefinition / ToolRegistry / ToolPrecondition semantics
- context-derived pseudo tool strategy
- responseMaterial schema
- deterministic placeholder response wording
- API compatibility and frontend action semantics
- contract and path tests
- fallback behavior

These are product-specific and already tuned to the existing app.

## 6. What AgentCore may improve

AgentCore may improve:

- runtime-level observability
- session and memory readiness
- managed execution visibility
- tool invocation visibility
- comparison metrics across runs
- operational trace correlation
- production-oriented runtime boundaries

AgentCore is less likely to improve:

- the actual user-facing quality of `selected_mark_explanation`
- the deterministic path's correctness
- the amount of tool work required

## 7. Observability comparison

| Observation | LambdaAgentRunner current | AgentCore candidate | Assessment |
| --- | --- | --- | --- |
| run-level trace | Already available through orchestration + runner trace summaries | Likely stronger runtime-native trace correlation | AgentCore may improve visibility |
| tool execution visibility | Already visible in Tool Layer traces and result summaries | Could improve managed runtime introspection | Incremental improvement |
| intermediate output inspection | Available through response material and trace metadata | Could become easier if AgentCore exposes richer runtime artifacts | Potentially better |
| failure debugging | Good for local reproduction and deterministic tests | Could help with cloud-side correlation | Mixed |
| latency metrics | Already measurable in runner metadata and tests | Could add richer managed metrics | Better metadata, not necessarily lower latency |
| timeout diagnosis | Existing runner summaries and API timeout constraints are explicit | Could improve timeout observability and cloud traces | Better diagnosis |
| budget/cost visibility | Already summarized in runner and execution results | Could improve platform-level visibility | Incremental improvement |
| local reproduction | Strong | Weaker, because managed runtime adds remote behavior | Lambda is better |
| CloudWatch correlation | Limited to current custom logs and trace summaries | Likely stronger if AgentCore integrations are adopted | Better |
| future memory/session support | Not present yet | One of AgentCore's strongest future benefits | Clear advantage |

Current path already does well on deterministic behavior, local reproduction, and testability. AgentCore's main upside here is observability and runtime management, not answer quality.

## 8. Latency and timeout comparison

The current Lambda/API Gateway path is simple and already deterministic. For `selected_mark_explanation`, the work done is mostly:

- request normalization
- intent / plan / tool orchestration
- summary-only response composition
- trace aggregation

That makes the path relatively lightweight.

AgentCore may improve timeout diagnosis, but it may not improve latency for a simple deterministic path.

Points to consider:

- API Gateway timeouts remain relevant for the current frontend-triggered path
- adding AgentCore can introduce extra runtime hops
- the selected-mark path is not currently a long-running multi-step workload
- any latency increase from AgentCore may outweigh its benefits for this path
- if the path later becomes longer-running or memory-heavy, AgentCore becomes more attractive

Assessment:

- for now, AgentCore is not a latency win
- AgentCore may be a diagnostics win
- the path does not yet justify a latency-cost trade for production migration

## 9. Operational trade-offs

Current custom runtime advantages:

- simpler deploy pipeline
- easier local development
- easier CI test coverage
- lower blast radius
- easier rollback
- more direct debugging
- fewer AWS moving parts

AgentCore advantages:

- managed runtime patterns
- future session/memory primitives
- potentially better production observability
- clearer runtime standardization if the app becomes more agent-heavy

AgentCore trade-offs:

- more AWS resource and IAM complexity
- more environment-specific debugging
- potentially higher cost
- more moving parts in demo/PoC flows
- harder local reproduction
- more vendor coupling
- more migration work before clear product value is proven

For a demo / PoC centered on a deterministic selected-mark explanation path, AgentCore looks operationally heavy unless observability and runtime management are a primary goal.

## 10. Risks and constraints

- AgentCore could add complexity without improving the user-visible path
- managed runtime behavior may be harder to reproduce locally
- additional runtime layers may blur the current clear boundary between Tool Layer and orchestration
- AgentCore memory / session features can tempt us to move too much logic too early
- timeout and cost comparison could become noisy if the path is too small
- the current custom deterministic response flow is already stable and easy to test

## 11. Fit assessment

**Fit assessment: Conditional fit**

Reasoning:

- `selected_mark_explanation` is deterministic and already well served by the current custom runtime
- AgentCore is unlikely to improve answer quality for this path
- AgentCore could improve observability, session readiness, and runtime standardization
- the added implementation and operational complexity is real
- the current path is still small enough that custom code remains the clearest choice

Conclusion:

- AgentCore is worth keeping as a comparison target
- AgentCore is not yet the default recommendation for this path
- the right next step is to preserve the current Lambda baseline and compare carefully rather than migrate immediately
- the next design step should stay thin and comparison-oriented
- the comparison metrics for that decision are documented separately in `docs/v0.6-agent-runner-comparison-metrics.md`

## 12. Recommended next step

Recommended next step: **A. #105 should define a thin AgentCoreRunner design only**

Why:

- the current Lambda baseline is already functioning
- the fit is conditional, not strong enough for immediate implementation
- we first need to define how AgentCore would map onto the current `AgentRunInput` / `AgentRunResult` contract
- a design-only step keeps the spike low risk
- it preserves the current deterministic baseline for later comparison

What #105 should focus on:

- AgentCoreRunner shape
- AgentCore observability mapping
- what remains custom vs what gets delegated
- unsupported / not configured result behavior
- no migration of the selected-mark path yet
