# AgentCore Spike Decision Record

## 1. Status

Accepted

## 2. Decision

AgentCore is not adopted as the primary runtime in v0.6.0.

The project will keep `LambdaAgentRunner` as the default runtime for `selected_mark_explanation`, retain the `AgentRunner` abstraction, and keep `AgentCoreRunner` as a thin optional future candidate rather than an active migration target.

## 3. Date

2026-07-02

## 4. Context

v0.6.0 was planned as the AgentCore spike for the project.

By the end of the spike, the project had:

- audited runner boundaries
- defined the `AgentRunner` contract
- added `LambdaAgentRunner`
- routed the explicit `selected_mark_explanation` path through `LambdaAgentRunner`
- evaluated AgentCore fit for `selected_mark_explanation`
- designed a thin `AgentCoreRunner`
- defined comparison metrics
- documented Tableau MCP and LLM composer next steps

The current `selected_mark_explanation` path is deterministic, placeholder-based, and LLM-free. That makes it a strong comparison baseline, but not a strong reason to adopt AgentCore immediately for answer quality.

## 5. Options considered

### Option A: Full adoption

- AgentCore becomes the primary runtime
- broader paths would move gradually toward AgentCore
- `selected_mark_explanation` would be one of the first migration cases

Pros:

- strongest use of managed runtime
- clearer long-term standardization if the project becomes agent-heavy

Cons:

- highest implementation and migration risk
- more AWS setup, IAM, and operational overhead
- more difficult local reproduction
- not justified by the current deterministic selected-mark path

### Option B: Partial adoption

- keep `LambdaAgentRunner` as default
- keep a thin optional `AgentCoreRunner` candidate
- compare runtimes using the same `AgentRunInput` / `AgentRunResult`

Pros:

- preserves the current baseline
- keeps the comparison path open
- allows observability and session-readiness work later

Cons:

- AgentCore benefits remain deferred
- the project still needs future integration work if AgentCore is adopted later

### Option C: Deferred adoption

- no AgentCore implementation in v0.6
- keep the spike focused on comparison readiness, docs, and boundaries

Pros:

- best scope control for v0.6
- avoids broad runtime migration too early
- keeps local testing simple

Cons:

- AgentCore runtime benefits are not tested yet
- observability improvements are only planned, not validated

### Option D: Not recommended now

- continue with the custom runtime only
- do not position AgentCore as an active future option

Pros:

- simplest short-term path
- lowest operational overhead

Cons:

- discards the comparison work already done
- leaves future runtime evolution unprepared
- does not preserve a clean path to observability or managed-runtime adoption

## 6. Decision drivers

- `selected_mark_explanation` is deterministic and lightweight
- the current path already works through `LambdaAgentRunner`
- AgentCore value is mostly observability, session readiness, and managed runtime comparison
- AgentCore may add latency, configuration complexity, IAM complexity, and local test overhead
- v0.6 should not expand beyond the runner spike
- Tableau MCP migration should be handled separately
- LLM ResponseComposer should remain optional and later
- demo stability and no-network local tests matter

## 7. Comparison findings summary

- `selected_mark_explanation` is a good comparison target, but not a strong immediate adoption target
- `LambdaAgentRunner` is sufficient for the current deterministic path
- `AgentCoreRunner` could improve runtime observability and future session handling
- `AgentCoreRunner` would also complicate deployment, IAM, local reproduction, and rollback
- latency gains are not likely for the current simple deterministic flow
- the strongest AgentCore benefit here is comparison-ready observability, not response quality

## 8. Consequences

### Positive consequences

- v0.6 scope stays controlled
- the existing `selected_mark_explanation` path remains stable
- `LambdaAgentRunner` stays the default runtime
- the `AgentRunner` abstraction remains useful
- future `AgentCoreRunner` adoption remains possible
- local tests remain simple and no-network friendly
- Tableau MCP and LLM composer can evolve independently

### Negative consequences / trade-offs

- AgentCore observability is not validated in production yet
- session and memory capabilities remain untested
- managed runtime benefits are deferred
- future AgentCore work will still require integration effort
- the spike produced planning artifacts more than measured evidence

## 9. What this decision does not decide

- AgentCore is not permanently ruled out
- `AgentCoreRunner` is not fully designed to implementation detail
- Tableau MCP migration order is not decided here
- LLM ResponseComposer adoption is not decided here
- free-form chat migration is not decided here
- v0.7 exploration scope is not decided here
- production deployment architecture is not decided here

## 10. Follow-up path

Immediate follow-up:

- complete `#110` v0.6.0 wrap-up
- link the spike outputs together
- keep the baseline runner path stable

Post-v0.6 candidates:

- Tableau MCP read-only metadata tool definitions
- Tableau MCP metadata ToolRegistry registration
- LLM ResponseComposer contract / strategy
- fake LLM composer tests
- optional AgentCoreRunner stub / not-configured result
- AgentRunner comparison measurement prototype
- trace persistence / debug UI
- exploration session design

## 11. Revisit conditions

Revisit AgentCore when:

- selected_mark_explanation becomes only one of several longer-running tool paths
- Tableau MCP tool execution grows and observability value increases
- session / memory / cross-run exploration becomes necessary
- Lambda/API Gateway limits become a real bottleneck
- a no-network test strategy for a stubbed AgentCore path is available
- cost, IAM, and deployment strategy become clearer

## 12. Relationship with Tableau MCP Tool Layer

Tableau MCP migration is independent from this decision.

The current decision keeps Tableau MCP as a later Tool Layer step, starting from read-only metadata contracts rather than broad query migration.

## 13. Relationship with LLM ResponseComposer

LLM ResponseComposer is also independent from this decision.

The current decision keeps deterministic response composition as the baseline and leaves optional LLM generation for a later milestone.
