# v0.5.0 Tool Layer Rebuild Plan

## 1. Purpose

v0.5.0 は、v0.4.0 で作った structured orchestration の上に、実行可能な Tool Layer を安全に載せるための準備フェーズです。

v0.4.0 では `Intent -> Plan -> Tools -> Response` の骨組みを作りましたが、ToolRouter と ExecutionEngine はまだ骨組みであり、ToolRegistry / ToolDefinition / 実ツール実行は未着手です。

v0.5.0 の目的は、MCP、REST API、Notion、context-derived data を同じ道具の扱いとして整理し、selected_mark_explanation から最小の実行可能な tool path に進める土台を作ることです。

## 2. Current state

### Orchestration layer

- `IntentResolver` は `selected_mark_explanation` を deterministic に解決できる
- `PlanDefinition` と fixed plan は存在する
- `ToolRouter` は allowed / skipped / blocked / unavailable を返す門番として動く
- `ExecutionEngine` は plan step を順に読み、ToolRouter に可否を問い合わせるだけで、実際の tool execution はしない
- orchestration trace events は intent / plan / execution / routing の骨格を持つ

### Tool execution surface

- Tableau MCP 呼び出しは、まだ orchestration の正式な tool layer に入っていない
- Notion や REST API 呼び出しも、ToolRegistry 経由の統一モデルにはなっていない
- context-derived data は frontend context preview と backend result の間で扱われているが、tool としての定義はない

### Frontend surface

- AI Context Preview Panel は selected-mark action suggestion と orchestration result を表示できる
- selected_mark_explanation はユーザー明示操作のみで起動する
- free-form chat は旧経路のまま維持されている

## 3. Problems to solve

1. Tool 実行先がバラバラで、同じ「道具」でも扱いが揃っていない
2. Tool 実行前の schema / precondition / availability の考え方が統一されていない
3. MCP と REST API と Notion と context-derived data の境界が曖昧
4. ToolRouter と ExecutionEngine は骨組みなので、実行可能な layer に進める必要がある
5. selected_mark_explanation 以外の tool path を増やす前に、最小の registry / definition contract を固めたい

## 4. Target architecture

```text
Intent
  ↓
Plan
  ↓
ToolRouter
  ↓
ToolRegistry
  ↓
ToolDefinition
  ↓
Tool execution
  ↓
ResponseComposer
```

### Design principles

- MCP も REST API も Notion も同じ Tool として扱う
- context-derived data は実ツールと分けて扱う
- ToolRouter は実行可否と安全性の判定に集中する
- ToolRegistry は lookup / availability / schema / execution wrapper を担う
- ExecutionEngine は plan の進行管理を担う
- selected_mark_explanation を最初の対象にする

## 5. ToolDefinition responsibilities

### What it owns

- tool name
- description
- category
- safety level
- input schema
- output schema
- preconditions
- timeout / budget hints
- execution wrapper
- trace metadata

### What it does not own

- plan selection
- intent resolution
- UI state
- chat job creation
- response composition policy

### Policy

- `inputSchema` / `outputSchema` は最小限から始める
- schema 実装は既存依存を優先し、追加 package が必要なら別 Issue に分ける
- read-only tool と write-capable tool を分ける
- preview-safe tool と explicit-action-only tool を区別する

## 6. ToolRegistry responsibilities

### Core responsibilities

- tool register / unregister
- tool lookup by name
- tool listing by category / capability
- availability checks
- precondition evaluation
- input validation
- output validation
- execution wrapper
- trace metadata emission

### Boundary

- ToolRegistry は tool を「実行できる状態か」を判定する
- 実行順序や plan 進行は ExecutionEngine 側に残す
- ToolRegistry は ToolDefinition を統一されたインターフェースとして見せる

## 7. Tool categories

v0.5.0 では、まずカテゴリを次のように分けるのが安全です。

- `context`
- `tableau_mcp`
- `rest_api`
- `notion`
- `internal`

### Notes

- MCP を特別扱いしすぎない
- ただし write-capable tool は明示的な安全確認を要求する
- preview 表示や deterministic な context 参照は `context` カテゴリとして扱う

## 8. Context-derived pseudo tools

context-derived data は外部ツールではありませんが、plan から参照しやすいので、v0.5.0 では「pseudo tool」扱いにするかどうかを整理します。

### Candidate data

- selected marks
- summary data preview
- filters
- parameters
- dashboard metadata
- workbook metadata

### Recommended approach

**Hybrid** を推奨します。

- 実行可能 tool とは分ける
- ただし plan step から参照できる `context` source として扱う
- ToolRegistry 上では pseudo tool を「実行」ではなく「参照可能な context capability」として見せる

### Why

- external tool と data context の責務が混ざりにくい
- selected_mark_explanation の最小経路で必要な材料を安全に扱える
- 将来、context tool を実体 tool に昇格させる余地を残せる

## 9. Tableau MCP tools migration strategy

### Current situation

- Tableau MCP 呼び出しは既存 flow の一部として存在する
- v0.4.0 では tool registry 化されていない

### Migration policy

1. まず tool definition を `tableau_mcp` カテゴリで定義する
2. read-only な metadata / query 系を先に登録する
3. write-capable / side-effect ありの tool は後回しにする
4. selected_mark_explanation に必要な最小 tool だけを登録する

### Notes

- MCP を特別扱いしないが、安定性は優先する
- tool 名は provider 固有より capability ベースに寄せる

## 10. Tool preconditions

### Candidate preconditions

- selected marks がある
- summary data preview がある
- dashboard / workbook / worksheet context がある
- permission がある
- Tableau MCP が available
- datasource identifier がある
- query input がある
- write operation には explicit confirmation がある
- plan がその tool category を許可している

### Minimal v0.5.0 scope

- selected marks / context availability / tool allowlist / budget を最初に扱う
- permission と explicit confirmation は write-capable tool を扱う際に追加する
- schema validation は registry contract に合わせて導入する

## 11. Tool input/output schema policy

### Policy

- input/output は JSON-safe を前提にする
- まずは最小の schema contract を定義する
- Issue #76 では schema library を導入しない
- input/output validation は後続 Issue に回す
- schema library の追加が必要なら別 Issue に分ける
- output は raw data ではなく、必要最小限の preview / evidence を返す

### Recommendation

- `inputSchema` / `outputSchema` は contract として定義する
- 実装は既存依存で足りる範囲から始める
- 依存追加が必要なら `Define tool input/output schema policy` を独立 Issue にする

## 12. Tool execution trace design

### Minimal trace events

- `tool_registry.lookup`
- `tool_precondition.passed`
- `tool_precondition.failed`
- `tool_execution.started`
- `tool_execution.completed`
- `tool_execution.failed`

### Why minimal

- v0.4.0 ですでに orchestration trace がある
- tool 由来のイベントだけを増やしすぎると trace が肥大化する
- lookup / precondition / execution の三段で十分に追跡できる

## 13. selected_mark_explanation minimal tool path

最初に必要なのは、selected_mark_explanation の説明に必要な context を安全に集める最小 path です。

```text
selected_mark_explanation
  ↓
context.selectedMarks
  ↓
context.summaryDataPreview
  ↓
context.filters / context.parameters
  ↓
必要なら tableau metadata query
  ↓
safe response / deterministic summary
```

### Notes

- v0.5.0 では LLM による本格回答生成はまだ不要
- まずは context を tool layer で扱えるようにする
- response は deterministic / placeholder から始める

## 14. Minimal v0.5.0 scope

1. ToolDefinition contract
2. ToolRegistry contract
3. Tool precondition model
4. context-derived pseudo tools の方針確定
5. selected_mark_explanation に必要な最小 tool の登録
6. ToolRouter から ToolRegistry への接続
7. tool execution wrapper
8. tool execution trace events
9. selected_mark_explanation 用の deterministic response の入口

## 15. Out of scope

- すべての MCP ツールの一括移行
- ToolRegistry のフル自動 discovery
- 高度な schema generator
- LLM 由来の tool planning
- free-form chat の全面移行
- AgentCore
- trace persistence / DB 保存
- UI の大幅な追加

## 16. Risks and constraints

- tool 定義が先に肥大化しやすい
- schema / precondition / execution wrapper を一気に作ると複雑化する
- MCP と REST API の差分を埋めようとして抽象化しすぎるリスクがある
- context-derived data を tool と同列に扱うと責務がぼやける

## v0.5.0 Context-Derived Pseudo Tools

See [`docs/v0.5-context-derived-pseudo-tools.md`](./v0.5-context-derived-pseudo-tools.md).

- context-derived data is not an external tool
- pseudo tools are a capability surface for routing and trace
- raw selected marks and full summary data stay out of trace
- `context.selectedMarks` is required for `selected_mark_explanation`
- `context.summaryDataPreview`, `context.filters`, and `context.parameters` are optional helpers

## 20. ToolExecutionWrapper

Tool execution must stay behind a wrapper boundary.

- the wrapper receives a handler map or executable binding, not a required `execute` function on `ToolDefinition`
- the wrapper normalizes output into JSON-safe data
- the wrapper handles timeout, budget, and error cases
- the wrapper keeps raw context bodies out of trace metadata
- selected-mark pseudo tools should be the first execution target
- full tool execution migration remains out of scope until the next tool-layer phase

## 21. Tool execution trace events

- tool execution trace events are part of the v0.5 trace-first contract
- emit `tool_registry.lookup`, `tool_precondition.passed`, `tool_precondition.failed`, `tool_execution.started`, `tool_execution.completed`, and `tool_execution.failed`
- keep trace payloads summary-only; do not copy raw selected marks, full summary rows, or full tool outputs
- trace persistence and UI trace panels remain out of scope
- the trace layer should stay compatible with the selected_mark_explanation path and later ToolRegistry / ToolExecutionWrapper integration

## 22. selected_mark_explanation context execution

- `selected_mark_explanation` is the first execution target for context-derived pseudo tools
- the path should collect `context.selectedMarks`, `context.summaryDataPreview`, `context.filters`, and `context.parameters`
- `ExecutionResult.responseMaterial` should carry JSON-safe selected-mark summary material for a future ResponseComposer skeleton
- raw selected marks and raw summary rows remain out of trace and response material
