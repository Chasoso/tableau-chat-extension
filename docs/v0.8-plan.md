# v0.8.0 Plan

## 1. Purpose

This document defines the v0.8.0 plan before any hosted metadata execution work begins.

Its purpose is to:

- connect the roadmap to a concrete v0.8.0 execution plan
- keep the first Hosted / remote Tableau MCP rollout gated and explicit
- preserve the fake / no-network path and stdio fallback
- keep `selected_mark_explanation` behavior stable
- keep the free-form chat legacy path unchanged
- prepare a clear handoff to v0.9.0 Structured Data Discovery and Legacy Chat Reduction

## 2. Background

The v0.7.0 milestone established the Hosted Tableau MCP migration foundation:

- transport abstraction
- transport configuration strategy
- hosted readiness and requirements audit
- hosted auth / user-context / site-settings boundary
- read-only metadata tool definitions
- metadata schemas and preconditions
- fake / no-network handlers
- transport-aware execution boundary
- metadata output normalization
- trace event shapes
- migration path documentation

v0.8.0 uses that foundation to connect the first read-only metadata tool to a gated Hosted / remote Tableau MCP execution path.

## 3. Current state after v0.7.0

After v0.7.0, the repo should have:

- `ToolRegistry` entries for read-only metadata wrappers
- metadata precondition evaluation
- transport configuration resolution
- transport-aware execution boundaries
- fake / no-network metadata handlers
- normalized metadata output
- trace event candidates
- a documented stdio-to-hosted migration path

What still remains intentionally unresolved:

- real Hosted MCP execution
- hosted auth implementation details
- hosted site settings verification in production
- stdio fallback policy under real hosted execution
- hosted error normalization against live responses
- readiness evaluation for `tableau.metadata.listFields`
- any broader chat migration work

## 4. v0.8.0 milestone goal

v0.8.0 connects the first read-only Tableau metadata tool to a gated Hosted / remote Tableau MCP execution path, starting with `tableau.metadata.describeDatasource`, while preserving fake / no-network tests, stdio fallback, and all v0.7.0 safety boundaries.

The milestone is successful if we can safely decide whether Hosted metadata execution is ready for the next phase, without widening scope into arbitrary query execution or free-form chat migration.

## 5. Scope

The v0.8.0 scope is intentionally narrow:

- Hosted / remote MCP execution planning and boundaries
- endpoint and site-settings verification
- hosted integration test gating
- HostedTableauMcpTransport or RemoteTableauMcpTransport adapter
- hosted auth context adapter boundary
- `tableau.metadata.describeDatasource` hosted execution behind a feature flag
- hosted metadata error normalization
- hosted metadata trace coverage
- fake / no-network tests preservation
- stdio fallback preservation
- `tableau.metadata.listFields` hosted readiness evaluation
- v0.8.0 wrap-up

## 6. Non-goals

v0.8.0 explicitly does not include:

- free-form chat full migration
- `ChatService` replacement
- `runLightweightAgentLoop` replacement
- arbitrary query execution
- underlying data access
- write-capable tools
- raw MCP tool exposure
- broad Tableau MCP tool exposure
- LLM-generated Tableau MCP query execution
- AgentCore implementation
- AgentCoreRunner implementation
- LLM ResponseComposer implementation
- UI trace panel
- trace persistence
- broad frontend redesign

## 7. Architecture target

The target path for the first hosted metadata execution looks like this:

```text
metadata tool request
-> ToolRegistry
-> tableau.metadata.describeDatasource
-> precondition evaluator
-> transport config resolver
-> transport-aware execution boundary
-> HostedTableauMcpTransport / RemoteTableauMcpTransport
-> Hosted Tableau MCP
-> normalized metadata result
-> metadata trace events
```

Fallback path:

```text
metadata tool request
-> ToolRegistry
-> metadata execution boundary
-> fake / no-network transport or stdio fallback
-> normalized metadata result
```

## 8. First hosted metadata tool decision

The first hosted metadata tool should be `tableau.metadata.describeDatasource`.

Why this one:

- the output can stay summary-first
- the schema is lower-risk than `listFields`
- hosted auth and transport handling can be validated on a read-only path
- precondition and normalization work can be exercised without broadening scope
- it gives us a concrete host-execution target for v0.8.0

`tableau.metadata.listFields` should stay a readiness-evaluation item in v0.8.0 rather than a required hosted execution target.

## 9. Hosted MCP readiness checkpoints

Before we rely on hosted execution, we should have clarity on:

- Hosted MCP availability
- endpoint shape
- site settings requirement
- OAuth / DCR / static registration decision
- user-context delegation model
- token handling boundary
- backend Lambda connectivity assumptions
- feature flag strategy
- CI fake / no-network strategy
- integration test gating
- rollback to fake / stdio behavior
- no arbitrary query / underlying data / write tool approval

## 12. Safety and governance boundaries

v0.8.0 must preserve the v0.7.0 safety boundary and add no broader access:

- read-only metadata first
- raw MCP tool exposure is forbidden
- arbitrary query execution is forbidden
- underlying data access is forbidden
- write-capable tools are forbidden
- datasource ambiguity must be blocked or explained
- auth and site-settings preconditions must be explicit
- Hosted MCP must remain explicitly opt-in
- CI must stay no-network by default
- local development must keep fake / no-network behavior
- stdio fallback must remain available
- output must remain normalized and JSON-safe
- no tokens, secrets, or raw stack traces in user-facing results
- `selected_mark_explanation` must remain unchanged
- free-form chat must remain on the legacy path

## 13. Validation strategy

Validation should match the issue type:

- backend lint
- backend typecheck
- backend tests
- no-network tests
- fake transport tests
- stdio fallback tests
- hosted transport skeleton tests
- hosted integration tests only when gated
- contract tests for the hosted metadata adapter
- error normalization tests
- trace event tests
- `selected_mark_explanation` regression tests
- free-form chat legacy regression tests

Docs-only issues should validate by checking that the required sections exist and that the document stays aligned with the roadmap and v0.7 handoff.

## 14. Rollback and fallback strategy

If hosted execution is not ready, the system should fall back safely:

- hosted feature flag off -> fake / stdio path
- hosted not configured -> safe `not_configured`
- hosted auth missing -> safe auth-required result
- hosted timeout -> safe timeout result
- site settings disabled -> blocked
- permission denied -> safe permission-denied result
- CI always uses fake / no-network unless explicitly gated
- local dev can continue to use fake / no-network
- `selected_mark_explanation` stays unaffected
- free-form chat stays unaffected

## 15. Risks and mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Hosted endpoint details remain unclear | v0.8 stalls | Keep endpoint and site-settings verification ahead of transport implementation |
| OAuth or user-context boundary is unclear | Auth risk | Isolate auth context adapter work before live integration |
| Hosted integration tests require secrets | CI instability | Keep hosted tests gated and off by default |
| `describeDatasource` output differs from assumptions | Normalization mismatch | Add contract tests and safe output mapping |
| Hosted errors differ from assumptions | Unsafe response handling | Normalize errors before surfacing them |
| Fallback breaks fake path | Local and CI regression | Keep explicit fake / stdio regression tests |
| `selected_mark_explanation` changes | Baseline regression | Keep dedicated regression coverage |
| Free-form chat changes accidentally | User-facing regression | Do not touch the legacy path in v0.8.0 |

## 16. Handoff to v0.9.0

v0.9.0 is expected to focus on Structured Data Discovery and Legacy Chat Reduction.

v0.8.0 should hand off:

- hosted metadata execution results
- `describeDatasource` viability
- hosted error normalization results
- trace coverage
- fallback strategy
- `listFields` readiness evaluation
- remaining auth, permission, and site-settings gaps

## 17. Open questions

The following questions should be resolved or documented by the end of v0.8.0:

- What is the exact Hosted MCP access model?
- What OAuth / user-context / token handling approach is required?
- Can backend Lambda connect to Hosted MCP directly?
- Is hosted execution gated by site settings, feature flags, or both?
- What timeout and retry behavior is safe for hosted execution?
- What error shape does Hosted MCP return in practice?
- Does `describeDatasource` mapping need additional normalization?
- Is `listFields` ready for v0.8.0 or better deferred to v0.9.0?
- How far should stdio fallback remain the default path?
