# v0.8.0 Plan

## 1. Purpose

This document locks the v0.8.0 scope, non-goals, and decision criteria before any Hosted execution work starts.

Its purpose is to:

- fix the v0.8.0 boundary before code-bearing work begins
- bridge the v0.7.0 foundation into v0.8.0 implementation planning
- make the child issue mapping explicit
- record the hosted execution assumptions up front
- define the safety and governance boundaries before implementation
- keep the validation path docs-first

## 2. Background

v0.7.0 completed the Hosted Tableau MCP Migration Foundation.

That milestone established:

- transport abstraction
- transport configuration strategy
- auth / user-context / site-settings boundary
- read-only metadata tool definitions
- input/output schemas
- preconditions and governance boundaries
- fake / no-network handlers
- transport-aware execution boundary
- output normalization and trace events
- stdio to Hosted migration path

v0.8.0 uses those foundations to gate the first read-only metadata tool into Hosted / remote Tableau MCP execution without attempting a full production migration.

## 3. Current state after v0.7.0

After v0.7.0, the current state should look like this:


Also true at this point:

- real Hosted MCP execution is not implemented
- real Hosted MCP transport is not implemented or not connected
- real stdio MCP replacement is not implemented
- fake / no-network path is preserved
- `selected_mark_explanation` keeps its existing behavior
- free-form chat remains on the legacy path
- AgentCore remains in deferred partial adoption

## 4. v0.8.0 milestone goal

```text
v0.8.0 connects the first read-only Tableau metadata tool to a gated Hosted / remote Tableau MCP execution path, starting with describeDatasource, while preserving fake / no-network tests, stdio fallback, selected_mark_explanation behavior, and the free-form chat legacy path.
```

The milestone should answer a simple question: is Hosted metadata execution ready enough to carry into v0.9.0 without widening scope into query, write, or chat-migration work?

## 8. First hosted metadata tool decision

The first hosted metadata tool should be:

```text
tableau.metadata.describeDatasource
```

Why this tool:

- output is small
- summary-first fit is good
- output size risk is lower than `listFields`
- auth, transport, and error handling can be validated on a read-only path
- v0.7.0 already prepared schema, precondition, and normalization boundaries
- `listFields` has larger field-count, truncation, omission, and permission concerns

`tableau.metadata.listFields` should remain a readiness evaluation item in v0.8.0, and only move to real Hosted execution if later safety checks justify it.

## 9. Hosted execution assumptions

Hosted execution in v0.8.0 assumes:

- Hosted MCP endpoint shape must be verified
- site settings requirement must be verified
- OAuth / DCR / static registration requirement must be verified
- user-context delegation model must be clear
- `tokenReference` boundary must be preserved
- raw access token and refresh token must not appear in result or trace
- Hosted execution must be explicit opt-in
- default CI must not require Hosted MCP
- hosted integration tests must be optional and gated
- fake / no-network path must remain first-class
- stdio fallback must remain available
- `describeDatasource` should be the first real hosted tool
- `listFields` requires additional readiness evaluation

Anything still unverified is `Unknown / needs verification`.

See `docs/v0.8-hosted-mcp-endpoint-site-settings-verification.md` for the endpoint / site-settings / admin-enablement gate that should decide whether #159 can proceed with real Hosted execution or remain on guarded skeleton and fallback paths.

See `docs/v0.8-hosted-mcp-integration-test-gating.md` for the companion test-gating policy that keeps default CI no-network while Hosted integration tests remain explicit opt-in.

See `docs/v0.8-hosted-tableau-mcp-transport-skeleton.md` for the hosted transport skeleton contract that #159 adds before any hosted tool wiring begins.

## 11. Decision criteria

Success criteria:

- `describeDatasource` connects through a feature-flagged Hosted / remote transport
- fake / no-network tests remain intact
- stdio fallback remains intact
- hosted integration tests are separated from default CI
- Hosted MCP errors normalize into safe metadata error summaries
- trace events cover Hosted execution
- `selected_mark_explanation` remains unchanged
- free-form chat legacy path remains unchanged
- arbitrary query, underlying data, and write tools are not added

Block criteria:

- Hosted MCP endpoint cannot be confirmed from official sources
- site settings requirements cannot be confirmed
- OAuth / user-context / token boundary cannot be defined safely
- default CI would require secrets or Hosted MCP access
- raw MCP result, token, or stack trace cannot be safely suppressed
- `selected_mark_explanation` shows regression risk

## 13. Safety and governance boundaries

v0.8.0 keeps the v0.7.0 boundaries intact:

- read-only metadata first
- raw MCP tool exposure is forbidden
- arbitrary query execution is forbidden
- underlying data access is forbidden
- write-capable tools are forbidden
- datasource ambiguity is blocked
- authenticated Tableau context is required for real execution
- site settings disabled blocks Hosted execution
- permission denied is a safe failure
- Hosted MCP explicit opt-in
- CI no-network
- local fake / no-network
- stdio fallback
- output normalization
- JSON-safe output
- no tokens, secrets, or raw stack trace
- `selected_mark_explanation` unchanged
- free-form chat legacy unchanged
- AgentCore deferred

## 14. Validation strategy

Docs-only issues:

- docs exist
- required headings are present
- no implementation is introduced

Code-bearing issues:

- backend lint
- backend typecheck
- backend tests

No-network tests:

- fake transport tests
- fallback tests
- metadata tool tests
- normalization tests
- trace event tests

Hosted tests:

- optional
- gated
- not required by default CI
- no secrets in default CI

Regression:

- `selected_mark_explanation` regression
- free-form chat legacy non-regression
- `ToolRegistry` registration regression

## 15. Rollback and fallback strategy

Rollback and fallback should be safe and boring:

- hosted feature flag off -> fake / stdio path
- Hosted endpoint missing -> `not_configured`
- auth missing -> auth required
- token expired -> auth expired / reauth required
- site settings disabled -> blocked
- permission denied -> safe permission denied
- timeout -> safe timeout result
- protocol error -> safe protocol error
- unknown Hosted error -> safe unknown / transport failed
- CI default -> fake / no-network
- local default -> fake / no-network or stdio
- `selected_mark_explanation` unaffected
- free-form chat unaffected

## 16. Risks and mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Hosted MCP endpoint details remain unclear | v0.8 cannot proceed safely | Verify official endpoint and site settings before implementation |
| OAuth / DCR / static registration unclear | auth design stays risky | Keep auth boundary isolated and docs-first |
| site settings requirement unclear | hosted execution may be blocked later | Confirm requirement before transport wiring |
| Hosted integration tests require secrets | CI instability | Keep hosted tests optional and gated |
| `describeDatasource` output differs from schema | normalization mismatch | Add contract checks and keep normalization conservative |
| Hosted errors differ from assumptions | unsafe responses | Map to safe summary-only failures |
| fallback breaks fake path | local and CI regression | Preserve fake and stdio regression tests |
| stdio fallback behavior is unclear | rollout risk | Keep stdio fallback explicit and documented |
| `selected_mark_explanation` changes | user-visible regression | Keep dedicated regression coverage |
| free-form chat changes accidentally | legacy behavior breaks | Do not touch the legacy path in this issue |
| `listFields` output is too large | truncation or omission risk | Keep it as readiness evaluation only |
| raw MCP output leaks | security and privacy risk | Never surface raw transport payloads |
