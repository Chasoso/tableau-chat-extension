# v0.7.0 Wrap-up

## 1. Purpose

This document summarizes the outcomes of the v0.7.0 "Hosted Tableau MCP Migration Foundation" milestone.

Its goals are to:

- gather the v0.7.0 results in one place
- make the v0.8.0 handoff explicit
- record how far the Hosted Tableau MCP migration foundation has progressed
- document what intentionally remained unchanged or deferred
- provide material for the develop -> main release summary

## 2. Milestone summary

v0.7.0 did not migrate the app to Hosted Tableau MCP. Instead, it established the foundation required to migrate safely: current stdio audit, hosted readiness analysis, transport boundaries, configuration strategy, auth / user-context boundaries, read-only metadata tool definitions, fake / no-network handlers, transport-aware metadata execution, output normalization, trace event shapes, and a staged migration handoff to v0.8.0.

In short, v0.7.0 turned Hosted Tableau MCP from an assumption into a documented, testable migration path.

## 4. Architecture outcome

The current free-form chat path remains unchanged:

```text
free-form chat
↓
ChatService / runLightweightAgentLoop
↓
legacy Tableau MCP usage
↓
stdio Tableau MCP
```

The v0.7.0 metadata foundation path is now established:

```text
ToolRegistry
↓
tableau.metadata.describeDatasource / tableau.metadata.listFields
↓
metadata precondition evaluator
↓
transport config resolver
↓
transport-aware execution boundary
↓
fake / no-network transport path
↓
metadata output normalizer
↓
safe normalized metadata result
↓
trace event candidates
```

The future hosted candidate path is now documented but not implemented:

```text
ToolRegistry
↓
metadata tool handler
↓
transport-aware execution boundary
↓
HostedTableauMcpTransport / RemoteTableauMcpTransport
↓
Hosted Tableau MCP
↓
normalized metadata result
```

v0.7.0 therefore ends with a dual-state architecture:

- legacy stdio Tableau MCP remains the current live path
- the metadata tool chain now has a transport-agnostic foundation ready for hosted execution

## 5. Transport abstraction status

The transport abstraction is defined and documented.

Key outcomes:

- transport contract exists
- supported kinds are `stdio`, `hosted`, `remote`, `fake`, and `unknown`
- normalized request, result, error, warning, timing, and trace shapes are documented
- authContext and userContext are summaries or references, not token containers
- raw tokens and secrets are not part of the transport result contract
- real stdio / hosted / remote transports are intentionally not implemented yet

The current code path supports fake and stubbed transport behavior for no-network tests and future adapter compatibility.

## 6. Transport configuration status

Transport selection is now a first-class configuration problem instead of an implicit runtime assumption.

Key outcomes:

- selection modes are documented: `auto`, `stdio`, `hosted`, `remote`, `fake`, `disabled`
- runtime environments are part of the selection model
- a pure config resolver is defined
- safe defaults prefer fake / no-network in CI and explicit opt-in for hosted
- hosted adoption is not silent
- misconfiguration is handled safely
- secrets are not required in the config result itself

## 7. Hosted MCP readiness status

Hosted Tableau MCP readiness is documented, but not fully resolved.

Current readiness position:

- Hosted MCP availability assumptions are recorded
- site settings assumptions are recorded
- backend Lambda connectivity assumptions are recorded
- local dev and CI constraints are explicit
- unknowns remain intentionally visible

The hosted path is therefore readiness-documented, not production-ready.

## 8. Auth, user-context, and site-settings status

The hosted auth boundary is now explicit.

Key outcomes:

- OAuth responsibility boundary is documented
- user-context delegation boundary is documented
- Cognito ↔ Tableau user relationship is called out
- site settings boundary is explicit
- token storage and refresh remain deferred
- permission and scope assumptions are visible
- no raw token, secret, or stack trace is allowed in Tool Layer, transport results, or trace payloads
- CI and no-network tests do not require OAuth

## 9. Metadata tool status

The first read-only Tableau metadata tools are now defined as app-specific wrappers:

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`

Key outcomes:

- raw MCP exposure is avoided
- read-only metadata only
- input and output schemas are defined
- ambiguity / resolution fields are defined
- warning / error summaries are defined
- truncation / omission signals are defined
- ToolRegistry registration exists
- fake / no-network handlers exist

## 10. Metadata execution status

Metadata execution is now transport-aware.

Key outcomes:

- preconditions run before transport execution
- transport config is resolved before execution
- fake transport path is first-class
- transport result mapping exists
- timeout / error normalization exists
- transport candidates remain interface-compatible
- no real Hosted MCP execution exists
- no real stdio MCP process execution exists
- no OAuth / token requirement exists for the fake path
- `selected_mark_explanation` regression protection is preserved

## 11. Output normalization and trace status

Metadata output is normalized before it is surfaced.

Key outcomes:

- normalized metadata result shape exists
- raw result suppression exists
- JSON-safe sanitizer exists
- warning / error summaries are preserved in safe form
- truncation / omission signals are preserved
- trace event candidates exist:
  - `tableau_metadata_tool.started`
  - `tableau_metadata_tool.completed`
  - `tableau_metadata_tool.failed`
- no raw MCP result is exposed
- no raw transport result is exposed
- no secrets, tokens, or raw stack traces are included
- no UI trace panel or trace persistence is introduced

## 12. Safety and governance outcome

v0.7.0 fixed the following safety and governance boundaries:

- read-only metadata first
- raw MCP tool exposure forbidden
- arbitrary query execution forbidden
- underlying data access forbidden
- write-capable tools forbidden
- Hosted MCP explicit opt-in
- fake / no-network first-class
- CI does not require Hosted MCP
- local dev can run without Hosted MCP
- datasource ambiguity is blocked
- auth / site settings / permission assumptions are explicit
- output is summary-first
- output is JSON-safe
- raw result / token / stack trace are suppressed
- `selected_mark_explanation` remains unchanged
- free-form chat legacy path remains unchanged

## 16. v0.8.0 handoff

v0.8.0 should be the first milestone to implement gated Hosted Tableau MCP metadata execution.

Recommended v0.8.0 theme:

```text
v0.8.0 Hosted Tableau MCP Metadata Execution
```

Recommended v0.8.0 goal:

```text
Implement a gated Hosted / remote Tableau MCP metadata execution path for the first read-only metadata tool, while preserving fake / no-network tests, stdio fallback, existing selected_mark_explanation behavior, and the free-form chat legacy path.
```

Recommended first hosted metadata tool:

```text
tableau.metadata.describeDatasource
```

Why:

- smaller summary-first output
- easier to bound than `listFields`
- better initial validation target for auth / transport / error mapping
- `listFields` can follow once truncation and omission behavior are trusted

Recommended v0.8.0 scope:

- implement `HostedTableauMcpTransport` or `RemoteTableauMcpTransport`
- use explicit hosted feature flags
- connect `tableau.metadata.describeDatasource` first
- preserve fake / no-network tests
- preserve stdio fallback
- normalize hosted errors into the existing metadata error summaries
- reuse the trace event shapes from #137
- verify auth / user-context / site-settings checkpoints
- no arbitrary query
- no underlying data
- no write tools
- no free-form chat full migration

## 18. Release notes draft

### Summary

v0.7.0 completed the Hosted Tableau MCP migration foundation. It did not make Hosted MCP production-ready; instead, it documented and implemented the transport, auth, metadata, governance, fake/no-network, execution, and normalization boundaries needed for a safe later migration.

### Added

- current stdio Tableau MCP usage audit
- hosted readiness audit
- transport abstraction and configuration strategy
- Hosted auth / user-context / site-settings boundary
- read-only metadata tool definitions and schemas
- metadata preconditions and governance boundaries
- fake / no-network ToolRegistry handlers
- transport-aware metadata execution boundary
- metadata output normalization and trace events
- stdio to Hosted migration path documentation

### Changed

- metadata work now runs through a transport-aware boundary
- metadata results are normalized before being surfaced
- trace payloads are now bounded and JSON-safe
- fallback and rollback behavior are documented

### Not changed

- frontend UI
- backend API behavior
- `ChatService`
- `runLightweightAgentLoop`
- `selected_mark_explanation` behavior
- legacy stdio Tableau MCP behavior
- AgentCore remains deferred partial adoption
- infrastructure and GitHub Actions

### Validation

- docs-only issues were validated by document existence and heading checks
- code-bearing issues #135, #136, and #137 reported backend lint, typecheck, and test success
- no-network tests exist for the metadata foundation path

### Known limitations

- Hosted MCP production execution is not yet implemented
- OAuth implementation is still deferred
- token storage / refresh policy is still open
- hosted endpoint details remain under verification
- stdio fallback remains the live safety net

### Next milestone

The next milestone should be v0.8.0 Hosted Tableau MCP Metadata Execution, beginning with gated hosted execution for `tableau.metadata.describeDatasource`.

## 20. Final decision

v0.7.0 is complete as a Hosted Tableau MCP migration foundation milestone. The project should not treat Hosted MCP as production-ready yet. The next milestone should proceed to v0.8.0 only as a gated Hosted metadata execution phase, starting with read-only `describeDatasource`, while preserving fake / no-network tests, stdio fallback, `selected_mark_explanation` behavior, and the free-form chat legacy path.

AgentCore remains deferred partial adoption. v0.8.0 should not introduce AgentCore unless a separate decision record is created.
