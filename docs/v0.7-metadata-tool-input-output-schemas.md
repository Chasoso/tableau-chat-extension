# Metadata Tool Input and Output Schemas

## 1. Purpose

This document defines the input and output schema boundaries for the read-only Tableau metadata tools introduced in v0.7.0.
The goal is to keep the metadata contract summary-first, JSON-safe, bounded, and free of raw MCP result leakage.

## 2. Background

The v0.7.0 milestone introduces app-specific wrappers for Tableau metadata discovery.
Those wrappers need explicit schema boundaries so transport, auth, and execution details can remain separate from the tool contract.

## 3. Schema principles

- prefer summary-first output over raw payloads
- keep every schema JSON-safe
- bound list sizes and summary counts
- allow ambiguity to be represented explicitly
- keep raw MCP results out of the schema surface
- keep tokens, secrets, and headers out of the schema surface

## 4. Shared identifier schemas

Shared identifier types are used to describe site, workbook, view, and datasource references without carrying raw Tableau context or transport data.

| Type                                  | Purpose                                |
| ------------------------------------- | -------------------------------------- |
| `TableauMetadataSiteIdentifier`       | Site identity and content URL boundary |
| `TableauMetadataWorkbookIdentifier`   | Workbook and project identity boundary |
| `TableauMetadataViewIdentifier`       | View identity boundary                 |
| `TableauMetadataDatasourceIdentifier` | Datasource identity boundary           |

## 5. Shared request context

`TableauMetadataToolRequestContext` carries only bounded orchestration context:

- request and correlation identifiers
- agent run identifier
- site / workbook / view / datasource references
- locale
- bounded counts and feature flags
- JSON-safe metadata

It must not carry raw tokens, raw auth headers, raw Tableau Extension context, SQL, VizQL, or raw MCP payloads.

## 6. `tableau.metadata.describeDatasource` input schema

Minimum input:

- `datasource` is required
- `requestContext` is optional
- `workbook`, `view`, and `site` are disambiguation context
- `includeFieldsSummary` and `includeConnectionSummary` are bounded summary toggles
- `maxFieldsForSummary` caps the sample size

The schema must not accept arbitrary query text, raw tool payloads, or token fields.

## 7. `tableau.metadata.describeDatasource` output schema

The output is summary-first and JSON-safe:

- `status`
- `summary`
- `resolution`
- `fieldsSummary`
- `connectionSummary`
- `warnings`
- `error`
- `truncation`
- `omissions`

The output does not expose raw rows, raw datasource blobs, or raw MCP results.

## 8. `tableau.metadata.listFields` input schema

Minimum input:

- `datasource` is required
- `requestContext` is optional
- `workbook`, `view`, and `site` are disambiguation context
- `maxFields` caps the returned list
- `includeHidden` and `includeTechnicalMetadata` toggle bounded detail
- `fieldNameFilter` is a simple bounded filter only

The schema must not accept arbitrary query strings, expressions, token fields, or raw MCP payloads.

## 9. `tableau.metadata.listFields` output schema

The output is a bounded list of field summaries:

- `status`
- `datasource`
- `resolution`
- `fields`
- `fieldCountSummary`
- `warnings`
- `error`
- `truncation`
- `omissions`

Each field summary is metadata only. The schema does not expose field values, sample rows, or raw MCP output.

## 10. Ambiguity and resolution fields

The schema can represent:

- resolved
- ambiguous
- missing
- not found
- not checked

Ambiguity is a schema concern, not a transport concern.
The concrete resolution strategy belongs to later precondition and execution work.

## 11. Warning and error summaries

Warnings and errors are normalized summaries, not raw exceptions.
They are designed to be safe for user-facing messages and traces once later normalization work is added.

## 12. Truncation and omission signals

The schema carries explicit truncation and omission indicators so that bounded results remain honest about what was omitted.

Examples include:

- output limit
- field limit
- hidden-by-default omissions
- technical metadata omissions

## 13. JSON-safe output boundary

The output boundary permits only JSON-safe values:

- strings
- numbers
- booleans
- null
- arrays
- plain objects
- bounded metadata objects

The boundary excludes raw `Date`, `Error`, `Map`, `Set`, `BigInt`, functions, streams, and raw response blobs.

## 14. Raw MCP result exposure boundary

The schema does not include raw MCP result fields such as:

- `rawMcpResult`
- `rawResult`
- `stdout`
- `stderr`
- `stack`
- `accessToken`
- `refreshToken`
- `authorizationHeader`

Raw results, if they ever exist internally, belong behind a later normalization boundary.

## 15. Relationship with ToolDefinition

The ToolDefinition layer owns the wrapper tool names and safety metadata.
This schema layer owns the input and output boundary for those wrappers.

The two layers are intentionally separate so transport and auth concerns do not leak into the tool contract.

## 16. Relationship with preconditions and governance

This document defines the shape of the data, not the execution policy.
The actual authenticated-context, datasource-ambiguity, site-settings, and permission checks belong to later precondition work.

## 17. Relationship with output normalization

Output normalization should map transport results into this schema without reintroducing raw blobs or secrets.
This document is the contract that normalization must satisfy.

## 18. Recommendations for #134, #135, #136, and #137

- #134 should interpret ambiguity and resolution summaries as precondition inputs
- #135 should return fake metadata outputs that match these schemas
- #136 should map transport results into these bounded schema shapes
- #137 should normalize warnings, errors, truncation, and omissions into the same boundary

## 19. Open questions

- should `describeDatasource` and `listFields` share a stricter common envelope later
- how much technical metadata should remain optional versus omitted by default
- whether the field-name filter should stay a plain string or later become a bounded pattern type
- whether additional metadata tools need their own schema family or can extend this one
