# Hosted Tableau MCP Requirements and Constraints Audit

## 1. Purpose

This audit documents the requirements, constraints, and unresolved questions for using Hosted / remote Tableau MCP in this repository before any real Hosted MCP connection is attempted.

Its purpose is to:

- compare Hosted MCP readiness against the current stdio audit in `docs/v0.7-stdio-tableau-mcp-usage-audit.md`
- create the assumptions needed for #129 transport abstraction
- create the assumptions needed for #130 transport configuration strategy
- create the assumptions needed for #131 OAuth / user-context / site-settings boundary
- keep the milestone docs-first and avoid accidental implementation work
- make the infrastructure boundary explicit before any integration begins

## 2. Background

The v0.7.0 milestone assumes:

- the current implementation is still stdio Tableau MCP based
- Hosted / remote Tableau MCP is the future target candidate
- v0.7.0 is not the Hosted MCP full migration milestone
- read-only metadata comes first
- arbitrary query, underlying data access, and write-capable tools are out of scope
- AgentCore remains deferred partial adoption
- free-form chat remains on the legacy path

The current repository already has a stdio audit, so this document focuses on what must be true before Hosted MCP can be considered safe and practical.

## 5. Hosted MCP access model

The access model for Hosted MCP is not yet confirmed, so this section separates confirmed platform facts from migration assumptions.

Confirmed platform facts from official docs:

- Tableau Cloud Connected Apps are configured by a site admin from `Settings > Connected Apps`
- direct trust uses JWT-based trust and can authorize REST API / Metadata API access
- OAuth 2.0 trust uses an external authorization server and an authorization-code style flow
- the current user context in Tableau is represented by the authenticated Tableau Cloud user subject

Migration assumptions for Hosted MCP:

- Hosted MCP likely needs a remote MCP transport rather than local stdio process startup
- the endpoint may be site-scoped, user-scoped, or tenant-scoped, but this is currently unknown
- the backend Lambda should be assumed to be the primary client until Tableau docs say otherwise
- direct browser-side access from the Tableau Extension frontend would increase auth and CORS risk, so it should remain an assumption to challenge rather than a default
- a backend proxy may be required if hosted MCP cannot be called directly from Lambda or if token exchange needs isolation
- session lifecycle, concurrency, and timeout behavior must be explicit before implementation
- stdio fallback should remain available while hosted readiness is being proven
- fake / no-network transport is needed so tests can run when hosted access is unavailable

## 6. Tableau Cloud site settings

Official docs confirm that Connected Apps are managed from Tableau Cloud site settings.

What the docs show:

- a site admin signs in to Tableau Cloud
- Connected Apps are configured under `Settings > Connected Apps`
- a new app is enabled manually after creation
- direct trust apps are created from the site settings UI
- OAuth 2.0 trust apps are also registered from the same settings area
- direct trust apps are set to disabled by default until enabled

What remains unclear for Hosted MCP:

- whether a hosted MCP-specific site setting exists
- whether the setting differs between dev, staging, and prod sites
- whether a REST API exists to inspect or change hosted MCP enablement
- whether an admin-only flag must be coordinated with embedding settings, connected-app settings, or a separate feature flag

## 7. OAuth and user-context requirements

The official docs confirm that Tableau Cloud Connected Apps support two trust styles:

- direct trust
- OAuth 2.0 trust

They also confirm that the authenticated Tableau Cloud user is represented in the JWT subject claim, and that OAuth 2.0 trust uses an external authorization server.

For Hosted MCP readiness, the unresolved design questions are:

- whether Hosted MCP expects OAuth 2.1 specifically or simply OAuth-style delegated access
- whether dynamic client registration is required
- whether static client registration is enough
- whether the backend should act as a confidential client
- whether a public-client model is ever appropriate for the Tableau Extension frontend
- whether user-delegated access is mandatory
- whether a service-account-like model is allowed
- how the current Cognito user maps to the Tableau user identity
- how the Tableau Extension auth context relates to backend auth context
- how the current Connected App / Direct Trust / JWT setup differs from the hosted path
- whether PAT is excluded, supported, or simply not the right model for hosted MCP
- what scopes and permissions are required
- who owns consent, reauth, expiry, and revocation
- how local development and CI should avoid depending on real OAuth traffic

## 8. Token handling boundary

The token handling boundary should be explicit before any Hosted MCP connection starts.

Current requirement direction:

- token acquisition should happen outside the Tool Layer
- token persistence should not be introduced as part of this issue
- any token passed to hosted transport should be short-lived and ephemeral
- tokens must not appear in trace output, logs, or user-facing errors
- secrets should stay in the existing secret-management boundary rather than becoming tool data
- CI and no-network tests should never require real tokens

What remains undefined:

- whether the backend obtains the token directly or whether a separate auth service does
- whether the hosted transport accepts a bearer token, session token, or another credential shape
- whether token refresh happens in the backend, in a proxy, or in a delegated auth service
- whether revocation needs explicit handling in the repo or only via the upstream platform

This audit intentionally does not implement token storage or refresh handling.

## 9. Backend Lambda connectivity assumptions

If the backend Lambda is the hosted MCP client, the following assumptions should be validated:

- the Lambda runtime can reach the hosted endpoint over outbound internet
- any VPC attachment still permits the necessary DNS and TLS egress
- the hosted call must fit within the Lambda and API Gateway timeout budget
- cold starts and connection reuse must be understood before relying on persistent sessions
- remote MCP session lifecycle must be compatible with stateless Lambda invocations
- retry and backoff must not create duplicate or unsafe side effects
- failure modes should be normalized before they reach the UI
- CloudWatch logs should be sufficient to correlate failures without exposing tokens
- local no-network tests must remain possible even if the hosted path exists

Status summary:

| Assumption | Status | Notes |
| --- | --- | --- |
| Lambda outbound network access | Unknown / needs verification | Must be validated against deployment topology |
| VPC / NAT impact | Unknown / needs verification | Could be irrelevant or a hard blocker depending on deployment |
| DNS / TLS reachability | Unknown / needs verification | Must be checked before hosted integration |
| API Gateway timeout budget | Confirmed as a constraint to respect | Exact hosted call budget must fit the deployed request path |
| Connection reuse across invocations | Unknown / needs verification | Likely limited by Lambda lifecycle |
| Session lifecycle for remote MCP | Unknown / needs verification | Must be documented before connection code exists |

## 10. Local development and CI strategy

Local development and CI should remain usable even when Hosted MCP is unavailable.

Recommended direction:

- keep fake / no-network transport first-class
- keep CI independent of Hosted MCP
- allow local developers to select fake, stdio, or hosted modes explicitly
- make hosted integration tests optional or gated
- ensure unit and contract tests pass without secrets
- retain stdio fallback until Hosted MCP is proven stable
- avoid increasing developer setup complexity more than necessary

Status summary:

| Requirement | Status | Notes |
| --- | --- | --- |
| Fake / no-network transport | Required direction | Needed for CI and offline development |
| CI without Hosted MCP | Required direction | The repo should not depend on hosted availability |
| Local mode switching | Required direction | Mode selection should be explicit and safe |
| Hosted integration tests gated | Recommended | They should not be required for normal CI |
| Secrets-free tests | Required direction | Unit and contract tests should not require real tokens |
| Stdio fallback retained | Recommended | It reduces migration risk while hosted readiness is proven |

## 11. Security and governance notes

Hosted MCP raises the security bar. The readiness audit should keep the following governance points visible:

- read-only metadata first
- least privilege
- user-context enforcement
- permission preconditions
- site / workbook / datasource access control
- no arbitrary query execution
- no underlying data access
- no write-capable tools
- no raw MCP tool exposure
- token secrecy
- no secrets in trace
- no raw stack traces in user-facing responses
- explicit output truncation and summarization
- auditability of hosted access
- admin enablement should be visible in docs and config

These constraints are especially important because the current repository already uses connected-app style Tableau access and a stdio transport path. The hosted path should not relax those boundaries.

| --- | --- | --- | --- |
| Transport | Local stdio child-process transport | Remote transport to a hosted endpoint | Transport abstraction is required |

| Auth | Connected App / JWT / direct trust assumptions | Likely delegated or OAuth-based access | Auth boundary must be separated from tool logic |

| User-context | `tableauSubject` and Cognito-adjacent backend auth are used | Hosted flow may need a clearer delegated-user model | User mapping must be formalized |
| Timeout | `listTools` timeout and legacy deadline logic exist | Remote round trips may take longer or fail differently | Timeout budgets need redesign |
| Retry | Retry-like behavior is distributed in the legacy loop | A hosted boundary should normalize retry policy | Retry behavior should move to the transport boundary |

| Error handling | Errors are normalized around the stdio path | Hosted errors must be normalized by transport | Error shape should be standardized |

## 13. Infrastructure boundary

v0.7.0 does not change infrastructure.

However, the hosted readiness work should acknowledge the infra elements that may later be needed:

- outbound network access from Lambda
- NAT or other egress configuration if Lambda is in a VPC
- secret management for any hosted OAuth / token material
- redirect or callback handling if the final auth flow requires it
- deployment configuration for feature flags and transport selection
- rollback strategy back to stdio fallback
- environment variables to control transport and auth boundaries
- staged rollout controls for hosted enablement

These are documented here so they stay visible without being implemented in v0.7.0.

## 14. Requirements before real Hosted MCP connection

Before a real Hosted MCP connection is added, the following should be true:

- Hosted MCP availability is confirmed by official docs or vendor guidance
- Tableau Cloud site settings are confirmed
- the OAuth / delegated-access flow is confirmed
- the user-context model is decided
- the token handling boundary is decided
- backend connectivity from Lambda is confirmed
- fake / no-network transport exists
- transport abstraction is defined
- read-only metadata tool schemas are defined
- no-network tests pass
- secret handling policy is defined
- fallback / rollback strategy is defined
- output normalization policy is defined
- security boundaries have been reviewed

## 15. Recommendations for #129, #130, and #131

### For #129 Define Tableau MCP transport abstraction

- the transport interface should handle stdio, hosted, and fake modes with one contract
- transport results should normalize auth, timeout, and network error cases
- transport responses should never expose raw tokens
- ToolDefinition should stay transport-agnostic
- fake / no-network transport should be first-class rather than test-only

### For #130 Define stdio vs hosted transport configuration strategy

- the default should stay safe, with fake or stdio fallback as the baseline
- hosted transport should be enabled only through an explicit feature flag or environment variable
- CI should not require Hosted MCP
- misconfiguration should fall back safely rather than fail open
- missing secrets should not leak into user-facing error messages

### For #131 Define Hosted MCP OAuth user-context and site-settings boundary

- define the OAuth boundary before implementation
- clarify current Cognito / Tableau user mapping
- design token lifecycle, refresh, and revocation before transport work
- verify site settings and admin enablement requirements
- connect permission boundaries to ToolPrecondition rather than ad hoc checks
