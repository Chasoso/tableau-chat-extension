# Metadata Discovery Clarification Contracts

This document defines the safe clarification response contract used by v0.9.0 structured metadata discovery.

## Purpose

When `metadata_discovery` is ambiguous, incomplete, or unsafe, the runtime returns a JSON-safe clarification response instead of an execution candidate.

This contract is used by #185 and is designed to feed #186 structured plan shape without exposing raw transport payloads or raw MCP data.

## Response kinds

- `clarification_required`
- `unsupported`

## Stable reason codes

- `missing_target`
- `ambiguous_target_type`
- `missing_identifier`
- `multiple_target_candidates`
- `target_not_resolvable_from_context`
- `unsafe_data_access_request`
- `unsupported_write_request`
- `metadata_query_boundary_conflict`
- `unsupported_discovery_request`

## Required safety boundaries

- No raw MCP result or transport payload.
- No token, secret, stack trace, or internal exception dump.
- No underlying data, field values, or row data.
- No execution plan or tool call payload.

## Resume contract

The caller can resume by re-running `metadata_discovery` with the missing fields from `resumeContract.requiredFields`.

Typical examples:

- `targetType`
- `identifier`
- `candidateTargetTypes`

Unsupported or unsafe requests should not be resumed as execution candidates. They should be rephrased as safe metadata discovery or handled by the legacy fallback path.
