# Read-only Tableau Metadata Tool Definitions

## 1. Purpose

This document defines the first read-only Tableau metadata tools for v0.7.0 as app-specific ToolDefinitions.
The goal is to keep the wrapper contract stable while deferring execution, transport, auth, precondition, and output-normalization work to later issues.

## 2. Background

v0.7.0 is the Hosted Tableau MCP migration foundation milestone.
The migration path starts with read-only metadata discovery because it is lower risk than query execution or write-capable operations.

The tools defined here are wrappers owned by the application, not raw MCP exports.

## 3. Tool definition principles

- use app-specific wrapper names
- keep the tools read-only and preview-safe
- require authentication without embedding raw tokens
- avoid exposing raw MCP tool names to the registry or to the model
- keep transport details out of the ToolDefinition surface

## 4. Tool set

| Tool name                             | Purpose                              | Category      | Capability              | Safety                      |
| ------------------------------------- | ------------------------------------ | ------------- | ----------------------- | --------------------------- |
| `tableau.metadata.describeDatasource` | Summarize datasource metadata        | `tableau_mcp` | `read_tableau_metadata` | read-only, safe for preview |
| `tableau.metadata.listFields`         | Summarize fields within a datasource | `tableau_mcp` | `read_tableau_metadata` | read-only, safe for preview |

## 5. `tableau.metadata.describeDatasource`

This tool describes a datasource at a metadata level.
It may surface safe summary fields such as display name, ownership, workbook or project context, field-count summary, and other non-sensitive metadata.

It must not:

- query underlying rows
- execute arbitrary Tableau queries
- mutate Tableau content
- expose raw MCP payloads

## 6. `tableau.metadata.listFields`

This tool lists fields for a datasource at a metadata level.
It may surface safe summary fields such as field name, type, role, and truncation indicators.

It must not:

- query underlying rows
- execute arbitrary Tableau queries
- mutate Tableau content
- expose raw MCP payloads

## 7. ToolDefinition fields

The v0.7.0 contract uses the existing `ToolDefinition` shape:

- `name`
- `description`
- `category`
- `capabilities`
- `safety`
- `availability`
- `inputSchema`
- `outputSchema`
- optional `metadata`
- optional `traceMetadata`

The metadata tool definitions use placeholder contract schemas so the detailed input/output contract can be finalized in #133.

## 8. Category, capability, and safety metadata

- category: `tableau_mcp`
- capability: `read_tableau_metadata`
- safety level: `read_only`
- safeForPreview: `true`
- requiresAuthentication: `true`
- externalAccess: `true`

These tools are metadata wrappers, not arbitrary query tools.

## 9. Authentication and external access

The tools declare that authentication is required, but they do not carry raw token values, token references, or auth flow logic.
That boundary belongs to #131 and later transport and precondition work.

## 10. safeForPreview semantics

`safeForPreview: true` means the tools do not mutate state.
It does not mean the tools are public or permissionless.
Authentication, site settings, and permission preconditions still apply.

## 11. Raw MCP exposure boundary

- raw MCP tool names are not used as ToolDefinition names
- raw MCP results are not exposed directly
- the registry should receive only app-specific wrapper definitions
- execution and transport remain outside this issue

## 12. Relationship with input/output schemas

Input and output schema details are defined in #133.
The schema contract covers identifier shapes, truncation flags, omission signals, warning and error summaries, and JSON-safe output boundaries.

## 13. Relationship with preconditions

Precondition design is intentionally minimal here.
The full authenticated-context, datasource-resolution, and governance boundary is deferred to #134.

## 14. Relationship with transport abstraction

These ToolDefinitions are transport-agnostic.
They do not know whether execution will come from stdio, hosted remote MCP, or a fake no-network transport.

## 15. Recommendations for #133, #134, #135, and #136

- #133 should define the input and output schema boundaries for the wrapper tools
- #134 should define the authentication, datasource resolution, and governance preconditions
- #135 should register the tools with fake no-network handlers
- #136 should connect the metadata handler to the transport boundary

## 16. Open questions

- should the two tools share a single metadata capability or split into more specific capabilities later
- should `availability` remain `conditional` until transport registration exists
- what minimal placeholder schema fields are needed to keep #133 focused
- whether future query-style tools should be a separate family rather than extending these wrappers
