# stdio Tableau MCP Usage Audit

## 1. Purpose

This audit documents the current stdio Tableau MCP usage before any hosted migration boundary is defined.

Its goals are to:

- preserve the current legacy chat path while migration work is planned
- make current stdio assumptions visible for transport abstraction work
- expose overlap between Tableau MCP usage and the Tool Layer
- provide the baseline that #128 and #129 will build on
- keep v0.7.0 focused on foundation work rather than implementation

## 2. Background

The milestone context is:

- v0.6.0 left AgentCore as deferred partial adoption
- `LambdaAgentRunner` remains the default runner
- `selected_mark_explanation` continues through the structured path and Tool Layer
- free-form chat still uses the legacy path
- Tableau MCP is still centered on stdio usage
- Hosted Tableau MCP is a new assumption introduced by the v0.7 roadmap

Because the migration target is changing, we need an explicit audit of where stdio is embedded today before we define a hosted boundary.

## 5. Current transport assumptions

The current transport model is strongly stdio-oriented.

- `TABLEAU_MCP_TRANSPORT` defaults to `stdio`
- the launch path is process-based rather than endpoint-based
- the stdio path uses a child-process style model with command and args
- `process.execPath` is the default command fallback when the runtime is Node
- `@tableau/mcp-server` is the default command target when args are resolved
- the transport is closed in a `finally` block after lookup completes
- stderr is piped and sanitized for diagnostics
- MCP communication uses the SDK client connection model rather than a custom remote protocol
- `listTools` and tool invocation are bounded by per-call timeouts
- the `http` path is currently a stubbed alternative, not a full hosted transport
- the code assumes remote MCP is not the primary path yet

Places that would be sensitive to a hosted switch include:

- local process startup and shutdown
- stderr-based diagnostics
- command / arg resolution
- env injection for a local child process
- transport-specific error logging
- per-request startup assumptions

## 6. Current authentication assumptions

Current Tableau access is based on backend-owned Tableau authentication, not hosted MCP OAuth.

Observed assumptions:

- Tableau auth is tied to Connected App style sign-in
- JWT generation is handled in the backend Tableau REST client path
- the backend holds the inputs needed to create a Tableau session
- `TABLEAU_DEFAULT_SUBJECT` and `TABLEAU_SCOPES` are part of the auth contract
- the MCP process receives auth-related env such as client ID, secret material, and subject data
- `tableauSubject` must exist for the stdio MCP path to proceed
- `AUTH_REQUIRED` and Cognito-related backend auth are part of the wider request context, but the Tableau MCP path itself is not yet hosted-OAuth aware
- the current code does not show a PAT-centered path
- the current code does not yet define a hosted user-context or token refresh boundary

Hosted MCP will likely need a clearer split between:

- backend identity
- Tableau user context
- site-level settings
- token issuance and refresh
- transport-level credential passing

That split is not formalized yet in the current stdio path.

## 7. Tableau MCP usage from legacy chat

The free-form chat path still uses Tableau MCP as part of its legacy orchestration.

Key behaviors observed:

- `ChatService` drives the current chat job flow
- `runLightweightAgentLoop` remains the main loop for free-form chat
- the loop can incorporate Tableau MCP-derived context into planning and response building
- `chatAgent.ts` re-evaluates plans when metadata is incomplete or warnings are present
- MCP-derived context is surfaced as execution/debug information
- legacy chat can fall back when the context lookup does not complete successfully
- the flow still distinguishes planning, tool execution, and finalization stages
- `selected_mark_explanation` is not the same path; it remains a structured path / Tool Layer / `LambdaAgentRunner` concern

The important migration constraint is that this legacy path must remain stable while hosted transport work is introduced elsewhere.

## 8. Timeout, retry, and failure handling

Timeout and failure handling are distributed across the legacy chat path and the Tableau MCP context provider.

Observed behavior:

- `listTools` uses a timeout from MCP config
- the HTTP stub path uses `AbortSignal.timeout(...)`
- startup validation can fail before transport execution
- transport and process startup failures are converted into structured failure states
- the context provider records failure stage / reason information
- the legacy chat flow has its own overall deadline fallback behavior
- plan re-evaluation gives the appearance of retry-like behavior when evidence is insufficient
- retries are not yet centralized at a transport abstraction boundary
- user-facing errors are normalized instead of exposing raw transport stack traces

The current model is workable for stdio, but it will need clearer normalization before hosted transport can be introduced safely.

## 9. Evidence, trace, and logging behavior

Evidence and trace behavior is currently mixed into service-level logging and execution debug data.

Observed patterns:

- stdout / stderr diagnostics are captured for stdio startup
- transport events are logged with MCP-specific event names
- chat service logs record context lookup completion details
- `mcpExecutionDebug` carries execution metadata forward
- tool call counts, planned tools, executed tools, and fallback reasons are surfaced in debug data
- stderr tails are sanitized before logging
- raw secrets are not intended to be logged
- raw MCP output can still influence downstream observations and summaries

This suggests the current trace model is operationally useful, but it is not yet a clean Tool Layer trace contract.

## 10. Result parsing and output handling

The current implementation parses and normalizes MCP-derived results at the context-provider layer.

Observed behavior:

- tool results are collected into structured context objects
- metadata / evidence / observation fields are separated during context assembly
- the provider keeps both normalized and raw-ish result data for internal use
- output shape is tuned for downstream chat reasoning rather than transport portability
- output truncation and safety constraints are implied by config and planning logic
- JSON-safe summary data is preferred over unbounded raw payloads

This is sufficient for the current legacy flow, but it is a good candidate for extraction into a transport-aware contract later.

## 12. Overlap with Tool Layer responsibilities

There is clear overlap between the legacy Tableau MCP path and the Tool Layer.

| Responsibility | Legacy Tableau MCP path | Tool Layer path | Overlap / Gap |
| --- | --- | --- | --- |
| tool definition | Defined implicitly in MCP lookup and allowed-tool lists | Owned explicitly by `ToolDefinition` / `ToolRegistry` | Gap exists; Tool Layer should own it |
| tool selection | Planned during legacy orchestration | Can be handled by ToolRouter / policy | Overlap exists today |
| preconditions | Handled in Tableau context lookup and fallback checks | Owned by `ToolPrecondition` | Overlap exists; should be normalized |
| execution wrapper | Child-process and transport lifecycle logic live in the context provider | `ToolExecutionWrapper` is the intended home | Gap exists; legacy path still owns execution |
| timeout handling | Split across chat loop and MCP calls | Should be centralized in wrapper / transport | Overlap and duplication exist |
| error normalization | Normalized in context provider and chat flow | Should be contract-driven in Tool Layer | Overlap exists |
| trace events | Service-level logs and execution debug data | Tool Layer trace should own tool-level events | Current trace is fragmented |
| output normalization | Context-provider level normalization | Tool Layer should standardize response shape | Overlap exists |
| response composition | ChatService / ResponseComposer consume the result | ResponseComposer owns final formatting | Legacy path still feeds it directly |
| budget handling | Chat loop budget and deadline logic | Tool Layer should not own all budget logic | Partial overlap |
| safety policy | Enforced by context provider and plan logic | Tool Layer should encode explicit capability/safety flags | Gap exists |

The main takeaway is that the current stdio path still owns transport, execution, and much of the safety handling. The Tool Layer already owns some adjacent concerns, but the boundary is not yet clean.

## 13. Responsibilities that must remain stable during migration

The following responsibilities should remain stable while hosted migration work begins:

- free-form chat legacy path
- existing Tableau MCP call behavior
- auth and token assumptions
- user-facing fallback messages
- evidence and trace behavior
- timeout behavior
- `selected_mark_explanation` structured path
- Tool Layer context tools
- `LambdaAgentRunner` as the default runtime
- no AgentCore dependency
- local development path
- CI path without hosted MCP

## 14. Risks and constraints for Hosted MCP migration

| Risk | Current source | Impact | Mitigation candidate |
| --- | --- | --- | --- |
| stdio assumption is embedded in the client path | `tableauMcpContextProvider.ts` launch logic | Remote transport becomes harder to introduce | Define a transport abstraction before execution is refactored |
| auth is coupled to the current MCP startup model | Backend Connected App / JWT setup | Hosted OAuth may not fit the current boundary | Document the auth boundary in #131 |
| legacy chat directly invokes MCP-backed context lookup | `ChatService` / `runLightweightAgentLoop` | Tool Layer bypass persists | Migrate in stages rather than replacing everything at once |
| raw MCP output can influence downstream summaries | Context assembly and logging | Safety and privacy risk | Normalize output before it reaches user-facing layers |
| timeout handling is distributed | Chat loop + transport call sites | Inconsistent failure behavior | Centralize timeout and error normalization |
| tests may require a real MCP endpoint | Current process model | CI instability | Keep fake / no-network paths first-class |
| datasource ambiguity is handled opportunistically | Precondition checks and fallback responses | Unsafe discovery behavior | Formalize preconditions before hosted execution |

## 15. Recommendations for #128 and #129

### For #128 Hosted Tableau MCP requirements and constraints

- document the hosted auth model in detail
- compare current Connected App / JWT assumptions with hosted OAuth and user-context needs
- confirm Tableau Cloud site settings that must be present
- verify whether backend Lambda can reach hosted MCP directly
- define local dev and CI behavior without a real hosted endpoint
- list unknowns early so implementation does not start with hidden auth constraints

### For #129 Tableau MCP transport abstraction

- hide stdio-specific launch logic behind a transport boundary
- use one result shape for stdio, hosted, and fake transports
- normalize timeout, error, and trace metadata at the transport layer
- keep ToolDefinition transport-agnostic
- make no-network fake transport a first-class path
- avoid replacing the legacy path all at once

## 16. Open questions

The following questions remain open after this audit:

- How exactly is the current stdio MCP server started and reused?
- Is Tableau auth token generation owned by the backend or by the MCP server process?
- Do we have the full set of legacy MCP tool names and input shapes mapped already?
- How will hosted OAuth connect to the current Cognito / Tableau auth setup?
- Where should fake / no-network transport live in the architecture?
- Should Tool Layer migration happen before legacy chat migration, or alongside it?
- What needs to be confirmed before code-bearing issues can begin?
