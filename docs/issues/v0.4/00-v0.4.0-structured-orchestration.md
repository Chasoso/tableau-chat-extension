# v0.4.0 Structured Orchestration

## Background

v0.3.0 completed the Tableau context-first layer. The frontend can now surface preview state, listen for selection changes, and show safe action suggestions.

The next step is to move the backend toward structured orchestration:

`Intent -> Plan -> Tools -> Response`

For v0.4.0, we want a narrow and safe first path that can connect the selected-mark action suggestion to the `selected_mark_explanation` intent without replacing the existing chat flow all at once.

## Goal

Introduce the minimal orchestration boundary needed to support deterministic intent resolution, fixed plan selection, explicit tool routing, structured execution, and trace-first visibility.

## Scope

- Define the v0.4 orchestration boundary and current migration constraints.
- Add the minimal contracts needed for intent resolution, plan execution, tool routing, and orchestration tracing.
- Connect the selected-mark action suggestion flow to a deterministic orchestration entry.
- Preserve the current chat job flow and `runLightweightAgentLoop` as a fallback path during migration.

## Out of scope

- Full replacement of `ChatService`
- Full replacement of `runLightweightAgentLoop`
- AgentCore migration
- Tool registry redesign
- Large frontend UX changes
- Multi-intent orchestration expansion beyond the first fixed path

## Child issues

1. Orchestration audit and migration boundaries
2. Define `IntentResolver` contract
3. Implement minimal `IntentResolver`
4. Define plan execution contract
5. Add `ToolRouter` skeleton
6. Add `ExecutionEngine` skeleton
7. Connect selected-mark actions to orchestration intent
8. Add trace-first orchestration events
9. Connect `selected_mark_explanation` fixed plan
10. v0.4 wrap-up and v0.5 handoff

## Acceptance criteria

- The orchestration boundary is documented and easy to follow.
- The first deterministic path from a selected-mark action to backend orchestration is defined.
- The existing chat job flow remains intact during the migration.
- Trace and budget responsibilities are clearly assigned.

## Validation

- Review each child issue and confirm it fits a single PR.
- Confirm the scope does not include a full orchestration rewrite.
- Confirm the docs make the v0.4 / v0.5 boundary explicit.
