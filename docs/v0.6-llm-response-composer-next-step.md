# LLM ResponseComposer Next Step

## 1. Purpose

This document defines the next step for a future optional LLM-based `ResponseComposer`.

The goal is not to implement LLM response generation yet. The goal is to define when an LLM composer would be safe, how it would relate to the deterministic composer, and what information it may or may not consume.

## 2. Background

v0.5.0 added a deterministic, placeholder-only `ResponseComposer` skeleton.

The current `selected_mark_explanation` path is:

```text
AI Context Preview Panel
-> selected_mark_explanation
-> fixed plan
-> ToolRouter
-> ToolRegistry lookup
-> ToolExecutionWrapper
-> context.selectedMarks required
-> context.summaryDataPreview optional
-> context.filters optional
-> context.parameters optional
-> ExecutionResult responseMaterial
-> ResponseComposer skeleton
-> deterministic / placeholder response
```

That baseline is intentionally stable and comparison-friendly.

## 3. Current deterministic ResponseComposer audit

The current deterministic composer should be treated as:

- a safe summary formatter for context collection results
- a fallback / warning presentation layer
- a baseline for regression tests
- a low-cost response path for `selected_mark_explanation`

Its expected responsibilities are:

- accept safe `responseMaterial`
- summarize selected mark counts and worksheet names
- summarize summaryDataPreview, filters, and parameters
- present fallback and warning states
- keep raw selected marks and full rows out of the message
- make it clear that LLM generation is not connected yet
- return JSON-safe output suitable for `AgentRunResult`

The deterministic composer is the right default baseline because it is stable, cheap, and easy to compare across runners.

## 4. Deterministic composer vs LLM composer split

### DeterministicResponseComposer

Responsibilities:

- safe context collection summary
- fallback and warning presentation
- stable regression baseline
- no hallucination risk from generation
- low latency and low cost
- comparison baseline for `selected_mark_explanation`

### LlmResponseComposer

Responsibilities:

- turn safe `responseMaterial` into a natural-language explanation
- optionally refine selected mark explanation wording
- include evidence-aware narrative
- support locale-aware output
- present warnings and limitations naturally

Recommended default:

- keep deterministic composer as the default / baseline
- make LLM composer optional
- use deterministic composer as the fallback when LLM generation is disabled or fails

## 5. ResponseComposer strategy design

Recommended long-term strategy:

```ts
interface ResponseComposer {
  compose(input: ResponseComposerInput): Promise<ResponseComposerResult>;
}
```

Potential implementations:

- `DeterministicResponseComposer`
- `LlmResponseComposer`
- `FallbackResponseComposer`
- `CompositeResponseComposer`

Recommended selection strategy:

- `modelPolicy.allowLlmGeneration`
- `responseStrategy`
- `intentId`
- feature flag
- environment variable
- budget / timeout availability
- explicit user action

Recommended behavior:

- deterministic composer remains the baseline
- LLM composer is optional and strategy-driven
- LLM failure falls back to deterministic output
- result type remains `ResponseComposerResult`
- the caller should be able to tell which strategy produced the output

## 6. responseMaterial boundary

Safe `responseMaterial` may include:

- selected mark count
- worksheet names
- field names
- compact selected mark summaries
- summaryDataPreview row count / column count / column names
- truncated preview summary
- active filter names / counts
- parameter names / value summary
- warnings
- fallback reasons
- trace summary
- evidence references
- no raw data notes
- user intent / action label
- locale

Unsafe or disallowed material:

- raw selected marks full values
- full summary rows
- underlying data
- secrets / tokens / auth headers
- raw MCP output
- raw Error object / stack trace
- large raw trace payloads
- unrelated internal infrastructure details

Boundary policy:

- `responseMaterial` must be compact, bounded, and safe enough to be prompt input
- it should explicitly mark `truncated`, `omitted`, or `rawDataSuppressed` state where needed
- LLM composer must not infer information that was not provided
- limitations must be eligible to appear in the final response

## 7. Prompt boundary

The prompt boundary should separate:

- system prompt: safety, honesty, and no overclaiming
- developer prompt: app-specific response policy and output format
- user prompt: the actual user-facing request or intent

Prompt design guidance:

- use intent-specific prompt text if needed
- keep `selected_mark_explanation` prompt narrow and factual
- embed only safe `responseMaterial`
- ask for structured output or a constrained template if that helps validation
- include explicit instructions to distinguish confirmed facts from inference
- require limitations when the context is incomplete
- define locale handling up front
- version prompts so comparison tests can lock to a known text

Prompt policy example:

> Explain the selected marks using only the provided safe Tableau context summary. Do not claim to have inspected raw rows or underlying data. Do not infer causes unless they are explicitly supported. Mention limitations when context is missing.

## 8. Safety and hallucination concerns

Main risks:

- claiming to inspect data that was not provided
- over-interpreting selected marks
- inventing causal explanations
- ignoring filters or parameters
- speaking as if truncated data were complete
- leaking raw data, PII, or secrets into the prompt
- confusing Extension API context with Tableau MCP output
- overstating confidence in the final message

Mitigations:

- strict `responseMaterial` boundaries
- deterministic fallback path
- prompt restrictions
- explicit limitation section
- max model calls and timeout budget
- no raw data policy
- prompt version logging
- output validation
- regression tests with golden cases

## 9. Observability and trace reuse

LLM composer should reuse existing run-level observability:

- `agentRunId`
- `intentId`
- `planId`
- `responseMaterial` summary
- tool execution trace summary
- warnings / fallback reasons
- model policy
- model call usage
- duration
- prompt version
- composer strategy
- safety flags
- rawDataSuppressed flag

Future trace events may include:

- `response_composer.started`
- `response_composer.completed`
- `response_composer.failed`
- `llm_response_composer.started`
- `llm_response_composer.completed`
- `llm_response_composer.fallback`

No new trace implementation is added in this issue; this is the next-step design only.

## 10. Runner compatibility implications

Recommended compatibility model:

- keep the `ResponseComposer` contract runner-independent
- keep deterministic composer as the common baseline
- allow LLM composer to be selected by policy / feature flag
- let runner capability determine the model invocation backend if needed

Implications:

- `LambdaAgentRunner` and `AgentCoreRunner` can share the same composer contract
- `AgentRunBudget` and `AgentRunBudgetUsage` should reflect model calls if LLM generation is enabled
- `selected_mark_explanation` consistency metrics should compare deterministic baseline behavior separately from optional LLM behavior

Recommended compatibility shape:

- deterministic composer is shared across runners
- LLM composer is optional and strategy-based
- the actual model invocation may differ by runner capability, but the contract should not

## 11. Migration strategy

Suggested stages:

1. docs only next-step plan
2. `LlmResponseComposer` contract / strategy type only
3. prompt boundary and prompt version docs
4. fake LLM composer tests
5. Bedrock-backed LLM composer behind feature flag
6. optional `selected_mark_explanation` LLM response
7. deterministic fallback on failure
8. comparison metrics update
9. decision record

For v0.6, only the docs-only planning stage is appropriate.

## 12. Next-step decision

Recommended next step:

- v0.6 should remain docs only for LLM composer
- deterministic composer should remain the baseline
- optional LLM composer should move to v0.7 or a later milestone

Reason:

- v0.6 is already focused on AgentRunner / AgentCore comparison
- LLM composer would add cost, latency, and hallucination risk
- deterministic baseline is important for comparison quality
- Bedrock changes are out of scope for this issue
- prompt and safety boundaries should be finalized before any implementation
