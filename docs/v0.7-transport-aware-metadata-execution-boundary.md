# Transport-aware Metadata Tool Execution Boundary

## 1. Purpose

This boundary makes Tableau metadata tool execution transport-aware without exposing raw transport details to Tool Layer callers.

It exists to:

- route `tableau.metadata.describeDatasource` and `tableau.metadata.listFields` through a boundary that can accept fake, stdio, hosted, or remote transport implementations
- keep the fake no-network path first-class for local development and tests
- normalize transport success, partial, timeout, unsupported, not-configured, and failure outcomes into safe metadata tool outputs
- attach trace metadata without leaking raw process, network, token, or stack-trace details
- preserve the current ToolRegistry registration and selected-mark explanation paths while enabling future transport replacement

## 2. Background

Issue #135 registered the metadata tools and added fake no-network handlers.

Issue #136 extends that work by ensuring the handlers no longer hardcode a single execution shape. Instead, they execute through a boundary that can accept a transport implementation by dependency injection.

This is still not production Hosted MCP execution.

## 3. Execution boundary responsibilities

The boundary is responsible for:

- accepting tool input and execution context
- evaluating metadata preconditions before any transport call
- building a JSON-safe `TableauMcpTransportRequest`
- invoking a `TableauMcpTransport`
- normalizing the returned `TableauMcpTransportResult`
- converting transport errors into safe metadata error summaries
- attaching execution trace metadata and timing summaries
- ensuring fake no-network results remain deterministic

## 4. Non-goals

This boundary does not:

- create a real Hosted MCP client
- spawn a real stdio MCP server process
- call Tableau REST, Metadata, or OAuth APIs
- implement arbitrary query support
- expose raw MCP responses
- change selected-mark explanation planning
- perform output normalization for real production hosted data beyond the metadata boundary

## 5. Execution flow

The execution flow is:

1. metadata tool input is normalized
2. preconditions are evaluated
3. a transport request is built from safe summaries only
4. the selected transport is called
5. the transport result is normalized
6. a safe metadata tool output is returned

If preconditions fail, the transport is not called and the handler returns a safe failed output with a user-facing message.

## 6. Dependency injection and transport selection

The runtime accepts a transport implementation through dependency injection.

Current behavior:

- default runtime uses a fake no-network transport
- tests can inject custom stubs that behave like stdio, hosted, remote, or timeout/failure candidates
- the executor does not build real stdio or hosted transports
- the executor does not read environment variables directly

This keeps transport selection separate from the metadata execution boundary.

## 7. Transport request mapping

The boundary maps metadata tool input into a JSON-safe transport request.

| Field | Purpose | Notes |
| --- | --- | --- |
| `requestId` | Correlation for the call | Safe identifier only |
| `toolName` | Tool routing | App-specific wrapper name |
| `input` | Normalized metadata input | No raw MCP payload |
| `timeoutMs` | Execution budget hint | Derived from preconditions/context |
| `correlationId` | Trace linking | Safe summary only |
| `agentRunId` | Orchestration linking | Safe summary only |
| `userContext` | User/site summary | No raw Tableau context |
| `authContext` | Auth summary/reference | No raw token |
| `trace` | Safe trace options | No payload or secrets |
| `metadata` | Execution summary | JSON-safe only |

## 8. Transport result mapping

The boundary normalizes transport results into metadata output shapes.

| Transport status | Metadata outcome |
| --- | --- |
| `success` | Return a success output, merging safe transport metadata |
| `partial` | Return a partial output with warnings |
| `failed` | Return a failed output with a safe error summary |
| `timeout` | Return a failed output with `TIMEOUT` |
| `not_configured` | Return a failed output with `TRANSPORT_NOT_CONFIGURED` |
| `unsupported` | Return a failed output with `TRANSPORT_FAILED` |
| `cancelled` | Return a failed output with `TRANSPORT_FAILED` |

The boundary does not pass through raw transport errors or raw stack traces.

## 9. Fake transport path

The fake transport path is first-class and deterministic.

It:

- does not require Hosted MCP
- does not require stdio MCP
- does not require OAuth
- does not use the network
- returns placeholder metadata only
- marks results as fake/no-network in safe metadata

This allows contract tests and local development to exercise the full boundary without live Tableau connectivity.

## 10. stdio / hosted compatibility

The boundary is compatible with stdio and hosted candidates through the shared transport interface.

Compatibility means:

- the executor only depends on the transport contract
- transport kind is normalized in trace metadata
- `not_configured`, `unsupported`, `timeout`, and `failed` are safe outcomes
- no transport-specific process or network details leak into the metadata response

This allows later issues to add real transports without changing the metadata handler shape.

## 11. Timeout and error normalization

Timeout and error handling is normalized into safe metadata errors.

Guidelines:

- transport timeout becomes `TIMEOUT`
- transport not configured becomes `TRANSPORT_NOT_CONFIGURED`
- auth-related transport failures map to auth-safe metadata errors
- permission-related transport failures map to `PERMISSION_DENIED`
- unknown failures map to `TRANSPORT_FAILED` or `UNKNOWN_ERROR` depending on context
- `retryable` and `userActionRequired` are preserved when available

Raw `Error` objects, stack traces, and transport internals are not returned.

## 12. Trace metadata boundary

The boundary attaches only safe trace metadata.

Allowed fields include:

- `requestId`
- `correlationId`
- `agentRunId`
- `toolName`
- `transportKind`
- `transportStatus`
- `startedAt`
- `completedAt`
- `durationMs`
- `timeoutMs`
- `warningCount`
- safe trace identifiers such as `transportEventId`

Not allowed:

- raw MCP payloads
- access tokens
- refresh tokens
- auth headers
- stack traces
- stdout or stderr
- raw process identifiers unless explicitly required and safe

## 13. Precondition failure handling

If metadata preconditions fail, the transport is not called.

The handler returns:

- a safe failed metadata output
- a user-facing message
- a normalized error summary
- trace metadata that records the precondition block

This applies to missing auth, ambiguous identifiers, disabled site settings, insufficient budget, and policy blocks.

## 14. ToolRegistry handler integration

`ToolRegistry` continues to register the two wrapper tools:

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`

Handlers route through the boundary rather than returning opaque placeholder objects directly. The registry itself does not learn about stdio or hosted transport details.

## 15. Safety boundaries

The boundary preserves these safety rules:

- no raw token in request, response, or trace
- no raw MCP response exposure
- no arbitrary query support
- no write-capable metadata actions
- no underlying data access
- no free-form chat migration
- no stdio or Hosted MCP process execution in this issue

## 16. Relationship with output normalization

Transport-aware execution is lower level than final metadata output normalization.

This boundary:

- receives transport results
- maps them into safe metadata tool outputs
- keeps the output JSON-safe and summary-first

Later output normalization work can refine truncation, omission, and summarization without changing the transport contract.

## 17. Recommendations for #137 and #138

For #137:

- centralize output normalization rules for success, partial, and failed metadata outputs
- keep transport warnings and execution trace metadata bounded and JSON-safe
- ensure raw transport payloads never appear in user-facing outputs

For #138:

- add a dedicated trace/logging boundary only if it can preserve the no-secrets rule
- keep transport internals out of persistent traces unless they are explicitly summarized
- keep the fake path and no-network tests deterministic

## 18. Open questions

- Should future real stdio and hosted transports live behind the same boundary or separate adapter classes?
- Should timeout normalization be owned by the transport adapter or the metadata executor?
- Should selected transport kind be resolved before the handler or passed in through context?
- Should trace metadata remain only in tool outputs, or also be surfaced in an internal execution log?
