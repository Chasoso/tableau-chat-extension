# Tableau MCP Transport Configuration Strategy

## 1. Purpose

This document defines how the repository should choose between `stdio`, `hosted`, `remote`, `fake`, and `disabled` Tableau MCP transport modes in a safe and predictable way.

Its purpose is to:

- keep transport selection deterministic across production, local development, and CI
- make hosted transport explicit rather than implicit
- keep fake / no-network execution first-class for tests
- preserve existing stdio behavior while hosted readiness is still being established
- keep transport selection details out of the Tool Layer

This is a configuration strategy document only. It does not implement a resolver or transport.

## 2. Background

The repository currently uses stdio-based Tableau MCP behavior as the active path, while the v0.7 roadmap introduces transport abstraction and hosted readiness as separate concerns.

At this point:

- the transport abstraction has already been defined in `docs/v0.7-tableau-mcp-transport-abstraction.md`
- hosted readiness constraints have been documented in `docs/v0.7-hosted-tableau-mcp-requirements-audit.md`
- current stdio behavior has been audited in `docs/v0.7-stdio-tableau-mcp-usage-audit.md`
- config behavior already exists for Tableau and MCP in `docs/configuration.md` and `backend/src/config.ts`

The missing piece is the selection strategy: how to decide which transport is safe to use in each runtime environment.

## 3. Configuration goals

The selection strategy should:

- prefer a safe default
- avoid silent hosted adoption
- keep CI independent from real hosted access
- keep local development usable without secrets
- allow explicit opt-in when hosted transport is eventually ready
- provide safe fallback behavior when configuration is incomplete
- return developer-safe metadata rather than raw secrets

## 4. Non-goals

This document does not define:

- a real Hosted MCP connection
- a real stdio transport implementation
- a real remote transport implementation
- secret storage
- OAuth or token refresh implementation
- infrastructure changes
- ToolRegistry integration
- metadata tool execution
- frontend changes

## 5. Transport selection modes

Recommended mode enum:

```ts
export type TableauMcpTransportSelectionMode =
  | "auto"
  | "stdio"
  | "hosted"
  | "remote"
  | "fake"
  | "disabled";
```

Mode meaning:

- `auto`: choose a safe transport based on runtime context and configuration
- `stdio`: use the current local / self-hosted stdio path
- `hosted`: use Hosted Tableau MCP only when it is explicitly enabled and configured
- `remote`: use a generic remote transport candidate when explicitly enabled
- `fake`: use deterministic no-network behavior
- `disabled`: do not use Tableau MCP transport at all

## 6. Runtime environments

Recommended runtime enum:

```ts
export type TableauMcpRuntimeEnvironment =
  | "production"
  | "development"
  | "test"
  | "ci"
  | "local"
  | "unknown";
```

Runtime meaning:

- `production`: deployed runtime with real user traffic
- `development`: developer machine or local backend process
- `test`: unit / contract / no-network test execution
- `ci`: automated pipeline execution
- `local`: explicit local development mode if the repo wants to distinguish it from generic development
- `unknown`: fallback when the runtime cannot be classified safely

## 7. Configuration input and resolution shape

The strategy should be driven by a pure input shape rather than direct environment-variable reads.

Recommended input:

```ts
export type TableauMcpTransportConfigInput = {
  mode?: TableauMcpTransportSelectionMode;
  runtimeEnvironment?: TableauMcpRuntimeEnvironment;
  hostedEnabled?: boolean;
  hostedEndpointConfigured?: boolean;
  hostedAuthConfigured?: boolean;
  stdioAvailable?: boolean;
  fakeAllowed?: boolean;
  noNetwork?: boolean;
  ci?: boolean;
  allowFallbackToStdio?: boolean;
  allowFallbackToFake?: boolean;
  allowHostedInCi?: boolean;
  metadata?: Record<string, unknown>;
};
```

Recommended resolution result:

```ts
export type TableauMcpTransportConfigResolutionStatus =
  | "selected"
  | "fallback"
  | "disabled"
  | "not_configured"
  | "invalid";

export type TableauMcpTransportConfigWarningCode =
  | "HOSTED_NOT_CONFIGURED"
  | "HOSTED_DISABLED"
  | "STDIO_FALLBACK_USED"
  | "FAKE_FALLBACK_USED"
  | "NO_NETWORK_MODE"
  | "CI_FORCES_FAKE"
  | "UNKNOWN_RUNTIME_ENVIRONMENT"
  | "UNSUPPORTED_SELECTION_MODE";

export type TableauMcpTransportConfigErrorCode =
  | "INVALID_TRANSPORT_MODE"
  | "HOSTED_SELECTED_BUT_NOT_CONFIGURED"
  | "STDIO_SELECTED_BUT_UNAVAILABLE"
  | "FAKE_SELECTED_BUT_NOT_ALLOWED"
  | "HOSTED_NOT_ALLOWED_IN_CI"
  | "NO_AVAILABLE_TRANSPORT";

export type TableauMcpTransportConfigResolution = {
  status: TableauMcpTransportConfigResolutionStatus;
  selectedTransportKind?: TableauMcpTransportKind;
  requestedMode?: TableauMcpTransportSelectionMode;
  runtimeEnvironment?: TableauMcpRuntimeEnvironment;
  reason: string;
  warnings?: Array<{
    code: TableauMcpTransportConfigWarningCode;
    message: string;
    metadata?: Record<string, unknown>;
  }>;
  errors?: Array<{
    code: TableauMcpTransportConfigErrorCode;
    message: string;
    metadata?: Record<string, unknown>;
  }>;
  fallbackFrom?: TableauMcpTransportSelectionMode;
  fallbackTo?: TableauMcpTransportKind;
  hostedEnabled?: boolean;
  noNetwork?: boolean;
  metadata?: Record<string, unknown>;
};
```

Contract rules:

- no token or secret values in input or output
- no `process.env` dependency in the pure resolver contract
- JSON-safe output only
- developer-safe metadata only
- user-facing error text should be created by higher layers

## 8. Environment variable strategy

The repo already uses `TABLEAU_MCP_TRANSPORT`, `TABLEAU_MCP_COMMAND`, `TABLEAU_MCP_ARGS`, and related configuration variables in `docs/configuration.md`.

For the new selection strategy, the candidate env-var surface should be explicit and separate from secret material:

- `TABLEAU_MCP_TRANSPORT_MODE=auto|stdio|hosted|remote|fake|disabled`
- `TABLEAU_MCP_HOSTED_ENABLED=true|false`
- `TABLEAU_MCP_HOSTED_ENDPOINT=...`
- `TABLEAU_MCP_HOSTED_AUTH_MODE=oauth_delegated|token_reference|unknown`
- `TABLEAU_MCP_STDIO_ENABLED=true|false`
- `TABLEAU_MCP_FAKE_ENABLED=true|false`
- `TABLEAU_MCP_NO_NETWORK=true|false`
- `TABLEAU_MCP_ALLOW_STDIO_FALLBACK=true|false`
- `TABLEAU_MCP_ALLOW_FAKE_FALLBACK=true|false`
- `TABLEAU_MCP_ALLOW_HOSTED_IN_CI=true|false`

Strategy notes:

- the env-var adapter itself is not required in this issue
- secret values should remain in a separate boundary
- the hosted endpoint may remain optional until #128 / #131 finalize the availability and auth model
- `auto` should be the default mode candidate, but not a silent hosted default

## 9. Feature flag strategy

Feature flags should control whether hosted transport may be selected at all.

Recommended rules:

- hosted should be off by default
- fake / no-network should be explicit and safe for CI
- stdio fallback should remain available during migration
- a runtime flag should be able to disable hosted immediately for rollback
- feature flags should not silently enable hosted behavior
- feature flags should not replace auth or site-setting validation

Recommended interpretation:

- env vars describe what is configured
- feature flags describe what the runtime is allowed to select

## 10. Default transport rules

Recommended safe defaults:

- CI and test environments should prefer `fake`
- no-network mode should prefer `fake`
- production `auto` should choose hosted only when hosted is explicitly enabled and fully configured
- development and local mode should prefer `fake` or `stdio`
- hosted should never be selected implicitly
- fake should not become the production default
- disabled mode should remain available for opt-out or shutdown cases

## 11. Local development mode

Local development should be optimized for low friction and no secrets.

Recommended behavior:

- default to `fake` or `stdio`
- allow hosted only with explicit opt-in
- keep no-network development working without Tableau hosted access
- allow stdio fallback only when the current local MCP server setup exists
- if stdio is unavailable locally, fall back to fake or disabled rather than failing open
- keep local setup complexity lower than hosted setup complexity

## 12. CI and no-network mode

CI should not depend on Hosted MCP.

Recommended behavior:

- CI should default to `fake`
- no network access should be required
- OAuth should not be required
- secrets should not be required
- stdio process startup should not be required unless a dedicated integration test wants it
- hosted should be disabled by default in CI
- hosted CI execution should require an explicit opt-in and be gated

## 13. Fallback strategy

Recommended fallback table:

| Scenario | Expected result | Fallback | Notes |
| --- | --- | --- | --- |
| `hosted` selected but not configured | `not_configured` or `invalid` | stdio only if explicitly allowed | Avoid silent hosted adoption |
| `hosted` selected but auth missing | `not_configured` | stdio or fake if allowed | Do not infer auth material |
| `hosted` selected but endpoint missing | `not_configured` | stdio or fake if allowed | Keep endpoint requirements explicit |
| `stdio` selected but unavailable | `not_configured` or `invalid` | fake if allowed | Preserve current behavior only when available |
| `fake` selected but fake not allowed | `invalid` | none | Do not fall back to fake when it is disallowed |
| `auto` in CI | `selected` | fake | CI should stay no-network |
| `auto` in local | `selected` | fake or stdio | Prefer predictable local defaults |
| `auto` in production | `selected` | hosted only if explicitly enabled and configured | Otherwise use stdio only if fallback is allowed |
| `disabled` mode | `disabled` | none | Explicit opt-out |
| unknown runtime environment | `fallback` or `invalid` | stdio or fake if allowed | Prefer safe behavior with warning |
| invalid mode | `invalid` | none | Fail closed |

## 14. Safe misconfiguration handling

Misconfiguration should fail safely and predictably.

Rules:

- do not include token or secret values in errors
- do not silently select hosted when hosted is not configured
- do not silently select hosted in CI
- do not fall back to fake in production if fake is disallowed
- treat invalid mode as invalid
- treat no available transport as disabled or not_configured, not as a hidden success
- return developer-safe metadata only from configuration resolution
- let higher layers create user-facing messages

## 15. Relationship with transport abstraction

This strategy sits above the transport contract defined in `docs/v0.7-tableau-mcp-transport-abstraction.md`.

Relationship rules:

- the resolver chooses a transport kind or a safe no-transport result
- the transport contract executes the selected kind
- the resolver should not know how stdio / hosted / fake are implemented
- the resolver should not read secrets or open network connections
- transport selection details should remain hidden from the Tool Layer

## 16. Relationship with auth and token handling

Transport selection must respect the auth boundary without implementing auth.

Rules:

- hosted should require explicit auth configuration before selection
- the configuration strategy should only know whether auth is configured, not store tokens
- token refresh, token persistence, and OAuth callback handling belong to later issues
- misconfiguration should not leak auth material into logs or error text
- current stdio / direct-trust assumptions should remain backward-compatible until #131 defines the hosted auth boundary

## 17. Recommendations for #131, #135, and #136

### For #131 Define Hosted MCP OAuth user-context and site-settings boundary

- finalize which auth signals mean hosted is allowed
- define the minimum hosted auth configuration that the selector can trust
- separate auth configuration from secret values

### For #135 Register metadata tools with fake no-network handlers

- make fake the safe default for CI and test contexts
- keep fake transport available even when stdio is present
- allow fake to represent no-network local development clearly

### For #136 Add transport-aware metadata tool execution boundary

- let the execution layer consume only the selected transport kind and normalized config outcome
- do not let metadata handlers re-implement selection logic
- keep runtime fallback behavior in the config boundary, not in the tool handler

## 18. Open questions

- Should `hosted` and `remote` remain distinct selection modes in implementation?
- Should production `auto` prefer stdio fallback when hosted is not configured, or should it return `not_configured`?
- Should local development default to `fake` or `stdio` when both are available?
- Should `disabled` be a true no-MCP mode or a fallback state with warnings?
- Should CI allow an explicit hosted opt-in for a separate integration workflow?
- Should the env-var adapter be added in #130, or deferred to a later config-wiring issue?
