# v0.9.0 Structured Data Discovery and Legacy Chat Reduction Plan

## Summary

v0.9.0 narrows the next step of the Tableau chat extension around discovery-oriented structured intents.

The milestone is about moving a subset of metadata discovery away from free-form legacy chat and into structured orchestration while preserving the safety posture established in v0.8.0.

The key idea is to keep the hosted metadata boundary narrow:

- `describeDatasource` remains the first validated hosted boundary
- `listFields` remains a controlled, truncated, permission-aware candidate
- `selected_mark_explanation` stays unchanged
- legacy free-form chat remains available as a fallback

## Background

The roadmap from v0.7.0 through v0.10.0 positions v0.9.0 as the milestone where structured discovery starts to matter more than transport plumbing.

By the end of v0.8.0, the repository had already established:

- a hosted / transport-aware metadata execution boundary
- safe auth and user-context handling
- output normalization and trace safety
- fake / no-network execution as the default posture
- stdio fallback as a preserved local-development path
- a clear refusal to expose raw MCP tools or arbitrary query execution

That means v0.9.0 does not need to solve hosted transport again.
It needs to use the existing boundary to route structured discovery safely.

## Relationship to v0.8.0

v0.8.0 validated the first hosted metadata boundary for `describeDatasource`.

Its wrap-up established the following carry-forward assumptions:

- read-only metadata comes first
- hosted execution stays explicit and opt-in
- default CI remains no-network
- fake / stdio fallback remains preserved
- `listFields` is deferred because of size, truncation, hidden-field, and permission risk
- `selected_mark_explanation` and legacy free-form chat behavior remain stable

v0.9.0 should reuse those decisions instead of reopening them.
In particular, v0.9.0 should treat the hosted `describeDatasource` path as the first validated metadata boundary for structured discovery, not as a license to broaden hosted access.

## Goals

- Add `metadata_discovery` as a structured orchestration intent.
- Clarify the datasource / workbook / view ambiguity model.
- Define clarification response contracts.
- Define a structured metadata discovery plan shape.
- Reuse the v0.8.0 hosted `describeDatasource` boundary.
- Treat `listFields` as a controlled, truncated, permission-aware candidate.
- Move eligible metadata discovery cases from legacy free-form chat to structured orchestration.
- Preserve `selected_mark_explanation` and legacy chat fallback behavior.

## Non-goals

- Full `ChatService` replacement.
- Full `runLightweightAgentLoop` replacement.
- Arbitrary data query execution.
- LLM-generated Tableau MCP query execution.
- Underlying data access.
- Field values or row data retrieval.
- Write-capable tools.
- Broad raw MCP tool exposure.
- Production Hosted MCP migration.
- OAuth full implementation.
- Token storage or token refresh.
- AgentCore implementation.
- LLM ResponseComposer implementation.
- Frontend redesign.
- Default CI hosted integration.

## Scope

v0.9.0 should cover:

- structured metadata discovery intent design
- ambiguity and clarification contracts
- discovery plan shape definition
- a safe routing boundary to the hosted `describeDatasource` path
- a controlled `listFields` discovery contract
- truncation and permission-aware response shaping
- gradual reduction of legacy chat coverage for metadata discovery cases
- regression protection for `selected_mark_explanation` and the fallback chat path

The milestone should not broaden beyond discovery-oriented cases.

## Safety and governance boundaries

v0.8.0 established the safety posture that v0.9.0 must keep:

- read-only metadata first
- hosted execution only for validated boundary cases
- no raw MCP tool exposure
- no arbitrary query execution
- no underlying data access
- no field values or row data retrieval
- no write-capable tools
- output normalization remains summary-first
- output must stay JSON-safe
- no raw MCP result exposure
- no raw transport result exposure
- no tokens, secrets, or stack traces in user-facing output or traces
- hosted execution stays explicit opt-in
- default CI stays no-network
- fake and stdio fallback remain preserved
- `selected_mark_explanation` remains unchanged
- free-form chat legacy fallback remains available
- AgentCore remains deferred

Some of the code-bearing items may start as contract-first work if the implementation boundary is still too early for a full execution change.
That is acceptable as long as the issue body makes the boundary explicit.

## Validation strategy

- Docs-only issues require no lint, typecheck, or test run.
- Contract-first issues should validate the contract shape with the smallest safe tests available.
- Code-bearing issues should keep default validation no-network.
- Hosted integration remains optional and gated.
- Regression coverage must protect `selected_mark_explanation`, legacy fallback behavior, and the no-network baseline.

## Completion criteria

v0.9.0 is complete when:

- structured discovery is routed through explicit contracts instead of ad hoc legacy behavior
- `metadata_discovery` is defined and safe to use
- ambiguity and clarification rules are documented and enforced
- the hosted `describeDatasource` boundary is reused for eligible cases
- `listFields` remains controlled and permission-aware
- legacy discovery coverage is reduced without breaking fallback behavior
- `selected_mark_explanation` remains unchanged
- default CI remains no-network
- no new raw MCP, arbitrary query, underlying data, or write-tool surface is introduced

## Notes for v0.10.0

v0.10.0 should make its decision from the boundary that v0.9.0 leaves behind.

Likely follow-up questions:

- should the next step be safer LLM ResponseComposer support
- should the next step be an exploration-session model
- should AgentCore remain deferred for one more milestone

The v0.9.0 outcome should make that decision easier by keeping the structured discovery surface narrow, explicit, and regression-safe.
