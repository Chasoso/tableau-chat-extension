# v0.9.0 Structured Data Discovery and Legacy Chat Reduction Plan

## Summary

v0.9.0 introduces structured metadata discovery as the next safe step in the Tableau chat extension.

The milestone does not try to replace legacy chat in one jump.
Instead, it narrows metadata discovery into a structured path, reduces legacy chat coverage for discovery-oriented cases, and keeps the fallback path intact.

The key boundaries remain:

- `describeDatasource` is the first validated hosted metadata boundary
- `listFields` stays a controlled, truncated, permission-aware candidate
- `selected_mark_explanation` remains unchanged
- legacy free-form chat remains available where structured discovery is not appropriate

## Background

The roadmap from v0.7.0 through v0.10.0 places v0.9.0 after hosted metadata execution groundwork, but before any broader chat replacement.

By the end of v0.8.0, the repository had already established:

- hosted metadata execution for `describeDatasource` behind explicit feature-flag and configuration checks
- safe auth and user-context handling
- output normalization and trace safety
- fake / no-network execution as the default posture
- stdio fallback as a preserved local-development path
- optional, gated Hosted integration tests
- a clear refusal to expose raw MCP tools, arbitrary query execution, or underlying data access

That means v0.9.0 is not a transport migration milestone.
It is a structured discovery milestone that reuses the validated hosted metadata boundary safely.

## Relationship to v0.8.0

v0.8.0 validated the first hosted metadata boundary for `describeDatasource`.

Its wrap-up established the following carry-forward assumptions:

- read-only metadata first
- hosted execution stays explicit and opt-in
- default CI remains no-network
- fake fallback remains the default safety posture
- stdio fallback remains preserved
- hosted integration tests remain optional and gated
- output normalization stays summary-first and JSON-safe
- raw MCP output, raw transport output, tokens, secrets, and stack traces stay out of user-facing output and traces
- `listFields` was evaluated but deferred because of output-size, truncation, hidden-field, and permission risk
- `selected_mark_explanation` and free-form chat legacy behavior remain stable

v0.9.0 reuses those decisions instead of reopening them.

## Goals

- Add `metadata_discovery` as a structured orchestration intent.
- Clarify the datasource / workbook / view ambiguity model.
- Define clarification response contracts.
- Define structured metadata discovery plan shape.
- Reuse the v0.8.0 hosted `describeDatasource` boundary.
- Treat `listFields` as a controlled / truncated / permission-aware candidate.
- Move eligible metadata discovery cases from legacy free-form chat to structured orchestration.
- Preserve `selected_mark_explanation` and legacy chat fallback behavior.

## Non-goals

- Full `ChatService` replacement.
- Full `runLightweightAgentLoop` replacement.
- Arbitrary data query execution.
- LLM-generated Tableau MCP query execution.
- Underlying data access.
- Field values or row data retrieval.
- Write-capable tools.
- Broad raw MCP tool exposure.
- Production Hosted MCP migration.
- OAuth full implementation.
- Token storage / token refresh.
- AgentCore implementation.
- LLM ResponseComposer implementation.
- Frontend redesign.
- Default CI hosted integration.
- GitHub Actions changes, unless explicitly required by a later issue.

## Scope

v0.9.0 should cover:

- structured metadata discovery intent design
- ambiguity and clarification contracts
- discovery plan shape definition
- a safe routing boundary to the hosted `describeDatasource` path
- a controlled `listFields` discovery contract
- truncation and permission-aware response shaping
- gradual reduction of legacy chat coverage for metadata discovery cases
- regression protection for `selected_mark_explanation` and the fallback chat path

The milestone should not broaden beyond discovery-oriented cases.

## Structured discovery boundary

v0.9.0 structured discovery is limited to metadata discovery.

It should follow this order:

- a user request is recognized as discovery-oriented
- intent and ambiguity are evaluated first
- clarification is returned when the target is unclear
- a structured plan is formed when the request is safe to execute
- only then does the flow reach the ToolLayer and the hosted metadata boundary

Important guardrails:

- do not convert a discovery request directly into raw MCP tool execution
- do not bypass intent, ambiguity, clarification, or plan-shape checks
- connect to `describeDatasource` only when the preconditions are satisfied
- return to clarification for unclear datasource / workbook / view targets
- do not move on to arbitrary query execution or underlying data access

## Legacy chat reduction boundary

Legacy chat is reduced, not removed.

The milestone should:

- keep `ChatService` and `runLightweightAgentLoop` in place
- move only metadata discovery cases that can be safely structured
- leave unsupported, ambiguous, or non-discovery cases on the legacy fallback path
- preserve the existing `selected_mark_explanation` behavior
- keep free-form chat available for the parts of the product that are not yet ready for structured orchestration

This means the milestone is a narrowing step, not a full replacement step.

## Hosted MCP and Tableau MCP safety boundaries

v0.8.0 established the safety posture that v0.9.0 must keep:

- Hosted execution is explicit opt-in
- default CI is no-network
- fake fallback is preserved
- stdio fallback is preserved
- no raw MCP tool exposure
- no raw MCP result exposure
- no raw transport result exposure
- no tokens, secrets, or stack traces in user-facing output or traces
- output normalization remains summary-first
- output must stay JSON-safe
- no arbitrary query execution
- no underlying data
- no field values or row data
- no write-capable tools
- `selected_mark_explanation` remains unchanged
- free-form chat legacy fallback remains available
- AgentCore remains deferred
- LLM ResponseComposer implementation is not part of this milestone

## Fallback and no-network strategy

The repository must continue to work without Hosted MCP, Tableau Cloud, OAuth, or secrets in the default test path.

That implies:

- local development stays no-network friendly
- CI stays no-network by default
- hosted integration tests remain optional and gated
- fake metadata execution remains the default safety posture
- stdio fallback remains available for configured environments
- the v0.8.0 fake / stdio fallback regression posture must remain intact in later issues

## Notes for v0.10.0

v0.10.0 should make its decision from the boundary that v0.9.0 leaves behind.

- should the next step be safer LLM ResponseComposer support

The v0.9.0 outcome should make that decision easier by keeping the structured discovery surface narrow, explicit, and regression-safe.
