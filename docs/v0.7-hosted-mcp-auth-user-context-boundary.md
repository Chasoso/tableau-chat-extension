# Hosted MCP Auth, User Context, and Site Settings Boundary

## 1. Purpose

This document fixes the Hosted MCP auth boundary before any Hosted MCP implementation begins.

Its purpose is to:

- separate OAuth, user-context, site-settings, token storage, and refresh responsibilities
- clarify the current Cognito / Tableau auth relationship
- keep ToolPrecondition, transport selection, and token storage boundaries distinct
- ensure local development and CI do not depend on real OAuth tokens
- keep later metadata tool design from hard-coding auth assumptions

This is a boundary document only. It does not implement OAuth, token persistence, or Hosted MCP connectivity.

## 2. Background

The v0.7 milestone is the Hosted Tableau MCP migration foundation.

Current state:

- v0.6.0 left AgentCore as deferred partial adoption
- v0.7.0 introduces hosted readiness, transport abstraction, and auth boundary design
- current Tableau MCP behavior is stdio-based
- hosted / remote MCP would shift auth and tenant-context concerns away from a local child-process model
- #131 is docs-only boundary definition
- real Hosted MCP connection is not required in v0.7
- AgentCore, LLM ResponseComposer, and free-form chat migration are out of scope

## 4. Hosted MCP auth problem statement

Hosted MCP changes the problem from local process startup auth to remote service access auth.

Key differences:

- hosted MCP is a remote service, not a local stdio child process
- user-context must be intentionally propagated rather than implied by a local backend session
- backend Lambda may be acting on behalf of the user, so the delegation boundary must be explicit
- service-account style access can simplify operations but increases privilege risk if overused
- delegated OAuth tokens may be required, depending on the official Hosted MCP model
- token storage, refresh, and revocation need a clear owner
- site settings and admin enablement may gate whether hosted access is usable at all
- permission enforcement may happen in multiple places: Tableau, the hosted transport, and ToolPrecondition
- Tool Layer should not depend on raw token implementation details

## 5. OAuth responsibility boundary

OAuth should be treated as a separate capability from transport and tool execution.

| Responsibility | Candidate owner | v0.7 decision | Notes |
| --- | --- | --- | --- |
| OAuth flow owner | Future auth integration layer | Undefined / boundary only | No OAuth flow is implemented in v0.7 |
| OAuth client registration owner | Deployment / auth operations | Undefined / boundary only | Client registration is not performed here |
| Callback / redirect owner | Frontend or dedicated auth handler, if required | Undefined / boundary only | Depends on the official Hosted MCP auth model |
| Token acquisition owner | Auth layer, not Tool Layer | Undefined / boundary only | Transport must not manufacture tokens |
| Token refresh owner | Auth layer or token service | Undefined / boundary only | Refresh is out of scope for this issue |
| Token revocation owner | Auth layer / platform policy | Undefined / boundary only | Needs platform confirmation |
| Consent / reauth owner | Auth layer | Undefined / boundary only | Might differ between delegated and service-account flows |
| Error handling owner | Auth layer for auth errors, higher layers for user messaging | Undefined / boundary only | Normalize without leaking secrets |
| Token reference issuer | Auth layer or secrets service | Undefined / boundary only | Reference only, not raw token |
| Hosted MCP transport handoff | Transport layer | Summary / reference only | Raw tokens must not cross this boundary |

This issue does not implement OAuth.

## 6. User-context delegation boundary

Hosted MCP should receive only the minimum user-context summary needed for authorization and traceability.

Recommended summary elements:

- Cognito user id
- Tableau user id
- email
- site id
- site name
- scopes / permissions summary

Rules:

- do not pass raw Tableau Extension context verbatim
- do not pass raw access tokens in user-context
- do not pass refresh tokens in user-context
- prefer `TableauMcpUserContextSummary` style summaries over raw identity objects
- avoid impersonation by default
- if a service account is used later, document the privilege and audit risks explicitly
- per-user delegated access should remain the preferred assumption until a later issue decides otherwise

## 7. Cognito and Tableau user relationship

The current repository has Cognito app authentication and Tableau subject-based access, but the exact identity mapping for Hosted MCP is not finalized.

Open items to keep explicit:

- whether email matching is the primary mapping
- whether Tableau user id is stored or fetched dynamically
- whether a user can belong to multiple Tableau sites
- how to handle user mismatch
- how to handle disabled or deleted Tableau users
- how to handle permissions mismatch
- how to represent fake users in local dev and tests

Current state should be treated as `Unknown / needs verification` where not directly documented.

## 8. Site settings boundary

Hosted MCP may require site-level enablement or admin approval.

Boundary rules:

- site settings ownership should remain with Tableau admin / operations, not the Tool Layer
- backend may read site-setting state as a precondition, but should not assume it can change it
- if Hosted MCP is disabled at the site level, the runtime should fail safely rather than attempt a hidden fallback
- dev / staging / prod sites may differ and should be documented separately
- CI should not require real site settings

## 9. Token storage and refresh boundary

Token handling must remain outside the Tool Layer and outside transport trace data.

Boundary rules:

- raw access tokens must not appear in Tool Layer, trace, logs, or user-facing responses
- refresh tokens must not be placed in transport requests
- if token references are used, they should be opaque references rather than secrets
- token storage owner should be an auth / secrets boundary, not the transport boundary
- expiry, refresh, and revocation must be handled before Hosted MCP transport sees the request
- CI / no-network tests must not require real tokens
- local dev should be able to use fake token references when needed

This issue does not implement token persistence.

## 10. Permission and scope assumptions

Hosted MCP should be read-only metadata first.

Permission assumptions:

- least privilege by default
- read-only metadata only
- no arbitrary query execution
- no underlying data access
- no write-capable tools
- permission preconditions should be explicit
- site settings preconditions should be explicit
- datasource ambiguity should still be checked before execution
- permission denied and insufficient scope should map to safe, normalized errors
- scope escalation should not happen implicitly

## 11. Transport configuration relationship

Auth and transport selection are related, but they are not the same boundary.

Rules:

- hosted transport should require auth to be configured
- hosted selected but auth missing should resolve to a safe not_configured / invalid outcome
- `auto` should not silently choose hosted
- CI and no-network should use fake and real tokens should not be required
- `authContext` should remain a summary / reference, not a raw token carrier
- raw token values must not appear in transport config, result, or trace
- normalized errors such as `AUTH_REQUIRED` and `AUTH_EXPIRED` should be produced at the transport / auth boundary, not by the Tool Layer
- Tool Layer should not know OAuth details

## 12. Local development and test constraints

Local dev and tests should work without Hosted MCP and without real OAuth tokens.

Required constraints:

- local default should be fake or stdio
- hosted should be explicit opt-in only
- CI default should be fake / no-network
- CI should not require OAuth tokens
- integration tests should be optional / gated
- fake user-context and fake token references should be acceptable in tests
- test fixtures should not contain secrets
- tests should remain deterministic
- hosted-unavailable environments should still pass the baseline suite

## 13. Security and governance rules

- no raw token in logs, trace, or errors
- no refresh token in transport request
- no secrets in test fixtures
- least privilege by default
- per-user delegated access preferred unless a later issue decides otherwise
- no service-account broad access by default
- read-only metadata first
- no arbitrary query execution
- no underlying data access
- no write-capable tools
- permission preconditions must be explicit
- site-setting preconditions must be explicit
- datasource ambiguity preconditions must remain in place
- safe user-facing errors only
- no raw stack traces
- auditability must be preserved

## 16. Recommendations for #132, #134, #135, and #136

### For #132 Define read-only Tableau metadata tool definitions

- tools should declare auth requirements but not token handling
- tools should remain read-only metadata only
- tools should not expose raw MCP or auth implementation
- tool definitions should include `requiresAuthentication` and `externalAccess` semantics

### For #134 Define metadata tool preconditions and governance boundaries

- include authenticated Tableau context precondition
- include site settings availability precondition
- include datasource ambiguity precondition
- include permission / scope insufficiency handling
- avoid raw auth errors in user-facing messages

### For #135 Register metadata tools with fake no-network handlers

- fake handlers should use fake userContext and fake tokenReference
- fake handlers must not require Hosted MCP / OAuth / real tokens
- tests must avoid secrets

### For #136 Add transport-aware metadata tool execution boundary

- metadata handlers should pass only authContext summary / tokenReference to transport
- transport should return normalized auth errors
- no raw token in trace / result
- hosted not configured should map to safe transport result
