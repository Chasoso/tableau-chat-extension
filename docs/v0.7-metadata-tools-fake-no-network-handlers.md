# Metadata Tools Fake No-network Handlers

## 1. Purpose

This document defines the fake, deterministic execution path for the first read-only Tableau metadata tools so they can be registered and exercised without Hosted Tableau MCP, stdio MCP, OAuth, or live Tableau connectivity.

## 2. Background

v0.7.0 establishes the migration foundation for hosted Tableau MCP. Before real transport work arrives, the metadata tools need a safe no-network path that keeps local development and CI deterministic and protects existing orchestration behavior.

## 3. Registered metadata tools

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`

Both tools are app-specific wrapper tools registered in `ToolRegistry`. Raw MCP tool names are not exposed.

## 4. Fake handler principles

- No network, no Hosted MCP, no stdio MCP process spawn
- No OAuth or token lookup
- Deterministic placeholder output only
- JSON-safe responses only
- Read-only and preview-safe behavior only
- Fail safely when preconditions are not satisfied

## 5. Fake `describeDatasource` behavior

- Returns a placeholder datasource summary
- Preserves datasource/workbook/site identifiers when provided
- Includes a summary-first response shape
- Emits a fake transport warning so the result is easy to identify
- Omits raw rows, raw MCP payloads, and any secret material

## 6. Fake `listFields` behavior

- Returns a placeholder field list with stable field names and metadata
- Honors `maxFields`, `includeHidden`, and simple `fieldNameFilter` behavior
- Emits truncation and omission signals when applicable
- Omits raw field values, raw rows, and raw MCP payloads

## 7. Fake precondition context

The fake path uses an explicit fake precondition context:

- authenticated Tableau context is marked as fake
- site settings are marked `not_required_for_fake`
- transport is marked `fake`
- permission is not verified, but the fake path remains deterministic

This keeps no-network tests self-contained.

## 8. ToolRegistry integration

The metadata tool definitions are registered through the existing in-memory registry pattern. The fake handler runtime exposes a registry plus an execution wrapper, mirroring other tool-layer patterns without changing selected mark orchestration.

## 9. No-network test strategy

- Verify registry lookup for both metadata tools
- Verify fake success output for both tools
- Verify missing and ambiguous datasource cases fail safely
- Verify JSON safety and placeholder-only output
- Keep selected mark context tools available and unchanged

## 10. selected_mark_explanation regression protection

The metadata runtime is isolated from the selected mark orchestration path. Existing selected mark context tools remain registered separately, and the metadata registration does not alter the fixed plan or orchestration flow.

## 11. Non-goals and safety boundaries

- No real Tableau connectivity
- No Hosted MCP execution
- No stdio MCP execution
- No OAuth flow
- No raw MCP exposure
- No arbitrary query execution
- No underlying data access
- No write-capable tools

## 12. Relationship with transport-aware execution

These fake handlers are a temporary no-network boundary for validation. Later transport-aware execution can replace the placeholder behavior without changing the wrapper tool names or the registry surface.

## 13. Recommendations for #136 and #137

- Keep transport-aware execution separate from tool registration
- Normalize output after execution, not inside the tool definition
- Preserve the fake path as a CI-friendly fallback
- Keep trace output free of tokens and raw MCP payloads
