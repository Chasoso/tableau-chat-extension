# v0.5.0 Tool Layer Audit and Boundaries

## 1. Purpose

v0.5.0 Tool Layer Rebuild の前提整理として、現在の tool-like responsibilities を棚卸しし、ToolDefinition / ToolRegistry / ToolRouter / ExecutionEngine の責務境界を明確にする。

v0.4.0 では structured orchestration の最小経路ができたが、MCP / REST API / Notion / context-derived data はまだ別々の層に散らばっている。
v0.5.0 では、それらを共通の Tool Layer に寄せる前に、どこまでを新 layer に移し、どこを既存 flow に残すかを決める。

## 2. Current v0.4 orchestration state

### Backend orchestration

- `backend/src/agent/intent.ts` と `backend/src/agent/minimalIntentResolver.ts` で `selected_mark_explanation` を deterministic に解決できる
- `backend/src/agent/plan.ts` で `selected_mark_explanation-v1` の fixed plan が定義されている
- `backend/src/agent/toolRouter.ts` は allowed / skipped / blocked / unavailable を返す skeleton
- `backend/src/agent/execution.ts` は plan step を順番に見て、ToolRouter で routing 判断をする skeleton
- `backend/src/agent/orchestrationTrace.ts` は intent / plan / execution / tool routing の trace-first metadata を持つ
- `backend/src/agent/selectedMarkOrchestration.ts` が selected-mark explanation の最小 orchestration path を接続している

### Current selected_mark_explanation path

```text
frontend AI Context Preview Panel
  ↓
selected-mark action suggestion
  ↓
frontend intent resolution / orchestration request
  ↓
backend MinimalIntentResolver
  ↓
selected_mark_explanation fixed plan
  ↓
ExecutionEngine skeleton
  ↓
ToolRouter skeleton
  ↓
trace-first result
  ↓
safe placeholder response
```

### Current plan shape

`selected_mark_explanation-v1` currently:

- requires `dashboard_context`, `context_preview`, `selected_marks`
- declares preconditions for dashboard context and selected marks
- uses `context` as the only allowed tool kind
- disallows `tableau-extension`, `tableau-rest`, `tableau-mcp`, `notion`, `llm`
- contains `validate-context`, `route-selected-marks-context`, `route-summary-data-preview`, `compose-response`
- keeps model calls at `0` and tool calls at a conservative minimum

### Frontend orchestration

- `frontend/src/App.tsx` builds an orchestration request from the selected mark action suggestion
- `frontend/src/api/orchestrationApi.ts` posts to `/intent/resolve`
- `frontend/src/components/AIContextPreviewPanel.tsx` is read-only and only emits the click callback
- `frontend/src/tableau/contextPreview.ts` keeps the context preview model as the source of truth for preview UI and suggestions

## 4. Current Tableau MCP tool usage

### Tool names in current codebase

- `list-datasources`
- `get-datasource-metadata`
- `query-datasource`
- `search-content`
- `list-workbooks`
- `list-views`

### Where the inputs are built

- `backend/src/services/tableauMcpToolPlanner.ts` chooses the tools and order from question interpretation and policy
- `backend/src/tableau/tableauMcpContextProvider.ts` assembles arguments, validates preconditions, and decides whether a tool should be attempted
- the current flow derives tool arguments from dashboard context, question interpretation, datasource hints, and auth context

### Where the outputs are interpreted

- `backend/src/tableau/tableauMcpContextProvider.ts` normalizes tool results into `TableauAdditionalContext`, `McpExecutionDebug`, and related summaries
- `backend/src/services/chatService.ts` consumes that context for answer generation, fast paths, and downstream logging
- `backend/src/types/tableau.ts` already models tool summaries and result summaries, but these are still service-level artifacts, not ToolDefinition contracts

### Precondition handling today

- auth subject availability
- MCP startup validation
- allowed tool list / deny list
- datasource identifier resolution
- result safety checks
- query-specific safeguards in planner/context provider

### ToolRegistry migration readiness

**Good candidates for ToolDefinition first**

- read-only metadata lookup tools
- query-like tools with explicit budget and result caps
- content search tools with bounded output

**Higher-risk candidates**

- write-capable tools
- tools that currently depend on planner heuristics and fallback chain behavior
- tools whose output is large and still needs summarization before exposing to orchestration trace

## 5. Current context-derived data usage

### Frontend context collector

`frontend/src/tableau/dashboardContext.ts` currently gathers:

- dashboard metadata
- workbook metadata
- view metadata
- worksheets
- filters
- parameters
- selected marks
- data sources
- summary data preview
- availability / warnings

### Preview model

`frontend/src/tableau/contextPreview.ts` normalizes that data into:

- `ContextPreviewModel`
- `ContextActionSuggestion`
- preview-safe selected marks and summary data preview
- `lastChangedWorksheet`
- `availability` and warnings

### Backend usage

- `backend/src/agent/selectedMarkOrchestration.ts` receives a `contextSummary` and the intent result to select a fixed plan
- `backend/src/services/chatService.ts` still treats dashboard context as an input to free-form chat
- no backend ToolRegistry exists yet for these context-derived values

### Boundary note

These values are **context sources**, not external tools. They should not be represented as full ToolDefinitions unless the execution model needs a capability handle. Their raw bodies should not be copied into trace events.

## 6. Current external API / Notion usage

### Notion

- `backend/src/notion/notionMcpClient.ts` uses Notion MCP / OAuth flows and performs write-capable operations such as page creation
- `backend/src/notion/notionService.ts` wraps those operations into application flows
- `backend/src/handlers/notionHandler.ts` exposes the HTTP surface

### Tableau REST / external APIs

- `backend/src/tableau/tableauRestClient.ts` performs direct REST requests to Tableau services
- auth and OAuth-related code also uses `fetch`, but that is infrastructure rather than tool execution

### ToolRegistry implication

- Notion and Tableau REST calls are easier to migrate into ToolDefinition than the current free-form chat loop
- write-capable operations should be gated by explicit confirmation / policy
- direct service flows should remain intact until the tool wrapper can preserve safety and traceability

## 7. Responsibility boundaries

### 7.1 ToolDefinition boundary

#### Should own

- tool name and description
- category and capability
- read-only vs write-capable
- preview-safe vs explicit-action-only
- input schema / output schema policy contract, not schema validation itself
- preconditions
- timeout / budget hints
- execution wrapper contract
- trace metadata shape

#### Should not own

- intent resolution
- plan selection
- plan step sequencing
- orchestration UI
- chat job creation
- response composition

### 7.2 ToolRegistry boundary

#### Should own

- register / unregister
- lookup / list / availability
- category filtering
- allowedTools / disallowedTools matching
- precondition evaluation entrypoint
- input/output validation entrypoint
- execution wrapper lookup
- tool lookup / missing-tool reporting
- tool execution trace hooks

#### Should not own

- intent resolution
- plan execution order
- response composition
- UI state

### 7.3 ToolRouter boundary

#### Should keep

- PlanStep-based routing decision
- allowed / skipped / blocked / unavailable
- plan policy application
- budget checks
- interpretation of ToolRegistry lookup outcomes

#### Should not do

- tool execution itself
- schema validation details beyond routing needs
- response composition
- intent resolution

### 7.4 ExecutionEngine boundary

#### Should keep

- ordered PlanStep processing
- ToolRouter invocation
- step result aggregation
- budget usage updates
- onFailure handling
- execution result generation

#### Should not do

- direct ToolDefinition execution semantics
- response generation
- intent resolution
- UI updates

### 7.5 ToolExecutionWrapper boundary

#### Should own

- timeout / cancellation handling
- execution time measurement
- result normalization
- error normalization
- JSON-safe output shaping
- trace metadata for actual execution
- handler-map based execution binding for pseudo tools and external tools
- keeping `ToolDefinition` JSON-safe and free of required executable functions

#### Should not own

- tool registration
- plan selection
- routing policy
- tool discovery
- raw context collection

### 7.6 ResponseComposer boundary

#### Should own

- deterministic placeholder / summary composition
- safe summarization of tool results
- response shape normalization

#### Should not own

- tool execution
- routing
- intent resolution
- raw context collection

## 8. Context-derived pseudo tools decision

### Options considered

- **A. Toolとして扱わない**
- **B. pseudo tool として ToolRegistry に登録する**
- **C. context source として ToolDefinition とは別枠にする**
- **D. hybrid approach**

### Recommendation

**D. hybrid approach** を推奨する。

#### Why

- 実装コストが低い
- trace しやすい
- `selected_mark_explanation` との相性がよい
- Tableau MCP など外部 tool との統一性も残せる
- ToolRegistry を context データで肥大化させにくい
- context 本体を trace に載せすぎるリスクを減らせる

#### How

- context-derived data はまず context source として扱う
- 必要なものだけ capability handle として registry 側に見せる
- 外部 tool と同じ execution model に無理に押し込めない

## 9. Tableau MCP migration strategy

### Current migration shape

- `TableauMcpToolPlanner` と `TableauMcpContextProvider` に散らばる tool selection / execution logic を、将来的に ToolDefinition + ToolRegistry に切り出す
- ただし、一気に全部移行しない

### Suggested migration order

1. 読み取り専用の metadata / search 系
2. bounded query 系
3. より大きな tool output を返すもの
4. write-capable / explicit confirmation が必要なもの

### ToolDefinition にしやすい単位

- `list-datasources`
- `get-datasource-metadata`
- `search-content`
- bounded `query-datasource`

### v0.5.0 で移行すべきもの

- selected_mark_explanation に必要な最小 context tool path
- 既存 planner と同じ意味を保てる read-only tool
- trace / budget がつけやすい tool

### v0.6 以降に回すべきもの

- 大規模な write-capable tool
- planner heuristics と強く結びつく複合 tool
- schema 追加が必要な高度な tool registry discovery

## 10. selected_mark_explanation minimal tool path

### Recommended minimal path

```text
selected_mark_explanation
  ↓
context.selectedMarks
  ↓
context.summaryDataPreview
  ↓
context.filters
  ↓
context.parameters
  ↓
deterministic ResponseComposer skeleton
```

### Necessary tool candidates

- `context.selectedMarks`
- `context.summaryDataPreview`
- `context.filters`
- `context.parameters`

### Optional external tool candidates

- `tableau.metadata`
- `tableau.query` or equivalent read-only MCP capability

### Guidance

- `selected_mark_explanation` の最初の段階では、context-derived data を優先する
- Tableau MCP は最初から必須にしない
- LLM を使わずに deterministic response で説明できる範囲を先に固める
- 不足分だけ後続の tool capability に回す

## 11. Trace and budget considerations

### Current trace state

- v0.4.0 trace already tracks orchestration / intent resolution / plan selection / execution / tool routing
- current trace metadata focuses on summary data and routing outcomes, not actual tool execution

### v0.5.0 minimal tool trace

- `tool_registry.lookup`
- `tool_precondition.passed`
- `tool_precondition.failed`
- `tool_execution.started`
- `tool_execution.completed`
- `tool_execution.failed`

### Budget handling

- keep `RunBudget` in the plan / execution layer
- pass budget snapshots into ToolRouter / ToolRegistry / ToolExecutionWrapper
- track `toolCallsUsed`, `modelCallsUsed`, `timeoutMs` without copying large context payloads into trace

## 12. Risks and constraints

- Tool abstraction can become too broad if MCP, REST, and context sources are merged too early
- Context-derived data can bloat registry semantics if treated as normal external tools
- Write-capable tools need explicit confirmation and stronger error handling
- Trace payloads should stay summary-only
- Free-form chat must remain on the existing flow for now

## 14. Recommendations

1. Define ToolDefinition / ToolRegistry contracts before adding any tool registration
2. Treat context-derived values as hybrid context capabilities, not external tools
3. Start with read-only tools and deterministic execution
4. Keep ToolRouter focused on routing and safety decisions
5. Keep ExecutionEngine focused on ordered step processing and result aggregation
6. Introduce tool execution tracing before broad tool migration
7. Delay write-capable tools until explicit confirmation policy is available

## v0.5.0 Context-Derived Pseudo Tools

See [`docs/v0.5-context-derived-pseudo-tools.md`](./v0.5-context-derived-pseudo-tools.md) for the adopted hybrid approach.

- context-derived data is not an external tool
- ToolRegistry can carry pseudo tool definitions for routing and trace
- ToolExecutionWrapper should resolve context references, not external services
- raw selected marks and full previews must stay out of trace
- `context.selectedMarks` is required; `context.summaryDataPreview`, `context.filters`, and `context.parameters` are optional helpers

## 16. Tool execution trace notes

- trace events for tool lookup, tool preconditions, and tool execution are summary-only
- registry lookup status, precondition status, execution status, duration, budget, and errors are traceable
- raw selected marks, summary rows, and full tool outputs stay outside trace payloads
- this trace layer is designed to support #82 now and #83 later without broadening execution scope

## 17. v0.5.0 wrap-up notes

- v0.5.0 completed the narrow selected_mark_explanation path for context-derived pseudo tools.
- ToolRegistry, ToolExecutionWrapper, and ResponseComposer remain intentionally bounded to the deterministic selected-mark path.
- context-derived pseudo tools stay separate from future external tool families until v0.6 planning defines a broader runtime boundary.
- the next phase should evaluate AgentCore against the current lightweight loop before any broader chat migration.
