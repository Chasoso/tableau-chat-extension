# Metadata Tool Preconditions and Governance Boundaries

## 1. Purpose

This document defines the precondition and governance boundary for the read-only Tableau metadata tools introduced in v0.7.0.

The goal is to make it explicit when metadata tools may execute, when they must fail safely, and which checks belong to policy rather than transport execution.

The boundary is intentionally narrow:

- it protects `tableau.metadata.describeDatasource` and `tableau.metadata.listFields`
- it keeps auth, site settings, datasource resolution, and budget checks explicit
- it keeps transport details out of the Tool Layer decision boundary
- it prepares the implementation path for #135, #136, and #137

## 2. Background

v0.7.0 is a Hosted Tableau MCP migration foundation, not a production migration.

The earlier milestone work established:

- #132: read-only metadata tool definitions
- #133: input/output schema boundaries

Those tools are read-only, app-specific wrappers, but they still need a governance layer before execution.

The governance layer must stay independent from:

- stdio vs hosted transport selection
- transport execution details
- output normalization
- trace emission

That separation keeps the metadata tools safe even while hosted readiness remains incomplete.

## 3. Precondition principles

The precondition layer follows these principles:

- fail safely when authentication is missing
- fail safely when identifiers are ambiguous or unresolved
- fail safely when policy does not allow the requested tool
- fail safely when the budget or timeout is too low
- keep the result JSON-safe and free of secrets
- avoid network, MCP, OAuth, or Tableau API calls inside the evaluator
- return a decision that can be consumed by later execution layers

Preconditions decide whether the tool may execute.

They do not execute the tool.

## 4. Governance principles

Governance describes what the tool is allowed to do.

For the metadata tools in v0.7.0, governance is intentionally conservative:

- read-only is required
- preview-safe is required
- external access must be allowed by policy
- underlying data access is blocked
- write operations are blocked
- raw MCP tool exposure is blocked
- arbitrary query support is blocked

The governance layer is policy-oriented, not transport-oriented.

## 5. Precondition input boundary

The precondition input should contain only summary-level information.

Recommended shape:

- `toolName`
- request identifiers such as `requestId`, `correlationId`, and `agentRunId`
- authenticated Tableau context summary
- site settings summary
- datasource / workbook / view resolution summaries
- tool policy summary
- budget / timeout summary
- transport configuration summary
- permission summary
- optional bounded metadata

The input must not contain:

- raw access tokens
- refresh tokens
- authorization headers
- raw Tableau Extension context objects
- raw MCP payloads
- raw child process output
- raw network responses

## 6. Precondition result boundary

The precondition result should communicate:

- whether execution may proceed
- why the decision was made
- which warnings are present
- which fallback action is appropriate
- what governance state was observed
- a safe user-facing message

Recommended result fields:

- `status`: `passed`, `warning`, `blocked`, or `not_checked`
- `canExecute`
- `failureCode`
- `message`
- `userFacingMessage`
- `warnings`
- `fallback`
- `governance`
- bounded `metadata`

The result must remain JSON-safe.

## 7. Allowed tool policy

Only the two app-specific wrapper tools are allowed here:

- `tableau.metadata.describeDatasource`
- `tableau.metadata.listFields`

Raw MCP tool names are not allowed.

Arbitrary query tools are not allowed.

Write-capable tools are not allowed.

Underlying data access tools are not allowed.

If a policy explicitly supplies an allowlist, any tool not on the allowlist must be blocked.

## 8. Authenticated Tableau context

The metadata tools require an authenticated Tableau context for real execution.

If the authenticated context is missing or marked unauthenticated, the evaluator must block execution.

Recommended summary fields:

- Cognito user id
- Tableau user id
- email
- site id
- site name
- auth mode summary

The evaluator must not require raw token values.

Fake / no-network paths may use a fake authenticated context, but the contract still requires the context to be explicitly present.

## 9. Site / workbook / view / datasource resolution

The metadata tools operate on summary resolution objects, not raw search results.

Recommended resolution statuses:

- `resolved`
- `ambiguous`
- `missing`
- `not_found`
- `not_checked`

For real execution:

- `resolved` may proceed
- `ambiguous` must block
- `missing` must block
- `not_found` must block
- `not_checked` should block unless a later layer explicitly relaxes the policy

The datasource resolution is the primary gate for these metadata tools.

Workbook and view resolutions are used as disambiguation context and should also fail safely when ambiguous.

## 10. Datasource ambiguity handling

Datasource ambiguity must never fall through to execution.

Safe handling means:

- missing datasource identifier blocks execution
- ambiguous datasource resolution blocks execution
- unresolved datasource resolution blocks execution
- not-found datasource resolution blocks execution

This is a governance decision, not a transport error.

The user-facing outcome should tell the user to choose one datasource explicitly.

## 11. Read-only operation check

The metadata tools are read-only by design.

The precondition layer should treat any non-read-only policy request as out of scope for these tools.

The tools must not:

- mutate Tableau state
- trigger write actions
- expose raw MCP commands
- execute arbitrary queries

## 12. safeForPreviewOnly policy

The metadata tools are intended to be safe for preview mode.

This means:

- safe for preview does not mean permission-free
- safe for preview does not mean external access is free
- safe for preview does not allow raw data access
- safe for preview does not bypass site settings

If a caller requires `safeForPreviewOnly`, these tools should qualify because they are read-only metadata tools.

## 13. External access policy

The metadata tools still rely on external Tableau access.

That means:

- the policy must explicitly allow external access
- the evaluator should block execution if external access is disabled by policy
- the tool must still avoid underlying data access and write operations

External access is allowed only to the extent needed for read-only metadata inspection.

## 14. Budget and timeout preconditions

The precondition layer should reject unsafe execution windows.

Recommended checks:

- a non-positive timeout blocks execution
- remaining execution time below a safe threshold blocks execution
- excessive max item requests should generate warnings and be reduced later

This keeps the metadata tools from starting when the surrounding request budget is already too tight.

## 15. Permission and capability notes

Permission verification is intentionally lightweight in v0.7.0.

The precondition result should distinguish between:

- permission verified
- permission not verified
- permission denied
- insufficient scope

The boundary should remain explicit even before full permission implementation exists.

For hosted execution, permission not verified should be treated conservatively.

## 16. Fallback and user-facing messages

The evaluator should return safe fallback guidance.

Recommended cases:

| Case | Recommended outcome | User-facing message |
| --- | --- | --- |
| auth missing | blocked | Tableau authentication is required before metadata can be inspected. |
| datasource missing | blocked | Please select or specify a Tableau datasource before running this metadata action. |
| datasource ambiguous | blocked | Multiple datasources match this request. Please choose one datasource first. |
| workbook ambiguous | blocked | Multiple workbooks match this request. Please choose one workbook first. |
| view ambiguous | blocked | Multiple views match this request. Please choose one view first. |
| site settings disabled | blocked | Hosted Tableau MCP is not enabled for this Tableau site. |
| site settings unknown | blocked or warning, depending on transport context | Tableau site settings are not verified for this metadata action. |
| tool not allowed | blocked | This metadata action is not allowed by the current tool policy. |
| budget exhausted | blocked | There is not enough remaining execution time to run this metadata action safely. |
| permission not verified | warning for fake / no-network, blocked for real hosted execution | Tableau permission could not be verified for this metadata action. |

The messages should be short, safe, and free of secrets.

## 17. Separation from transport concerns

Preconditions decide whether metadata tools are allowed to execute.

Transport decides how an allowed tool would be executed.

Output normalization decides how raw or transport-level results become safe metadata output.

Response composition decides how safe metadata output is presented to the user.

The precondition evaluator must not:

- open a network connection
- spawn a process
- call MCP
- read process environment variables
- fetch OAuth tokens
- inspect raw transport output

## 18. Relationship with metadata schemas

The precondition layer consumes the summary-level schema objects defined earlier in v0.7.0:

- datasource, workbook, view, and site identifiers
- ambiguity and resolution summaries
- warning and error summaries
- truncation and omission summaries

Those schemas supply the safe input the evaluator needs.

The evaluator should not reconstruct raw schema objects from transport responses.

## 19. Relationship with output normalization

Preconditions operate before transport execution and before output normalization.

That means:

- preconditions can stop unsafe execution early
- output normalization can assume an allowed execution path
- normalization remains responsible for raw-result suppression later in the flow

The precondition result should not contain raw MCP payloads or raw stack traces.

## 20. Recommendations for #135, #136, and #137

### #135 Register metadata tools with fake no-network handlers

- use the precondition evaluator before the handler runs
- keep fake no-network behavior consistent with the same governance result shape
- preserve the same safe user-facing messages

### #136 Add transport-aware metadata tool execution boundary

- call the precondition evaluator first
- only pass allowed requests to transport execution
- keep transport errors separate from governance failures

### #137 Add metadata output normalization and trace events

- normalize only after the precondition boundary has passed
- keep trace events free of tokens, stack traces, and raw MCP payloads
- reuse the governance and warning summaries when shaping safe output

## 21. Open questions

- Should permission not verified be a warning or a block for non-hosted paths?
- Should site settings unknown be a warning in more cases, or always blocked for hosted execution?
- Should budget reduction be a warning only, or should it also carry an explicit fallback action?
- Should datasource ambiguity and site ambiguity share the same fallback message pattern?
- Should the evaluator expose a richer capability summary for later policy checks?
