# v0.11.0 Plan: Usable Hosted Tableau MCP Metadata Discovery

## Summary

v0.11.0 is the milestone that makes Hosted Tableau MCP usable in the existing structured metadata discovery flow, not just a connection-validation milestone.

The goal is to let the current `metadata_discovery` / chat flow use Hosted Tableau MCP in an opt-in and gated way, limited to safe read-only metadata discovery through `describeDatasource`.

Key points:

- DCR-dependent OAuth 2.1 Remote MCP host flow is not adopted
- the application uses the existing backend-controlled `Connected App + Direct Trust` boundary
- the first hosted execution target is `describeDatasource`
- hosted execution remains optional and gated
- default CI remains no-network
- no raw MCP output, arbitrary query execution, underlying data access, or write-capable tools are introduced
- `selected_mark_explanation` remains protected
- LLM ResponseComposer behavior remains unchanged
- Exploration Session remains deferred
- AgentCore remains deferred

## Background

v0.7.0 through v0.10.0 built the foundation for this milestone:

- Hosted Tableau MCP readiness and transport boundaries
- read-only metadata tool boundaries
- `describeDatasource` boundary
- no-network and fake fallback behavior
- structured metadata discovery
- datasource / workbook / view ambiguity handling
- clarification contracts
- legacy chat reduction for eligible discovery cases
- `selected_mark_explanation` regression protection

v0.10.0 completed Branch A: LLM ResponseComposer and left Branch B: Exploration Session deferred.

v0.11.0 does not expand LLM composition further. Instead, it focuses on making Hosted Tableau MCP practically useful inside the existing metadata discovery path while preserving the current safety posture.

## Goal: usable Hosted metadata discovery

v0.11.0 is not only a connection-validation milestone.

The goal is to make Hosted Tableau MCP usable in the existing structured metadata discovery flow, limited to safe read-only `describeDatasource` usage.

Expected end state:

- a user asks a data-source or metadata-discovery question
- existing intent resolution, ambiguity checks, and precondition checks run
- only eligible cases proceed to Hosted execution
- Hosted Tableau MCP `describeDatasource` runs when enabled and configured
- the result is normalized before it reaches user-facing output
- raw MCP output is not exposed
- failure, timeout, auth failure, permission denial, and not-configured cases become safe normalized outcomes
- fake and stdio fallback behavior remains intact
- hosted integration tests remain gated
- default CI stays no-network

## Authentication decision

Selected authentication boundary: `Connected App + Direct Trust`

This milestone does not use:

- Dynamic Client Registration
- DCR-dependent OAuth 2.1 Remote MCP host flow
- OAuth 2.0 Trust / External Authorization Server
- PAT
- token storage / refresh as a new v0.11.0 feature

The auth boundary is intentionally narrower than a generic Remote MCP host flow.

The application is not trying to become a generic remote MCP host. It is a Tableau extension app with a backend-controlled auth boundary, so Direct Trust fits the architecture better.

## Why DCR-dependent OAuth 2.1 Remote MCP host flow is not used

OAuth 2.1 itself is not rejected.
The rejected path is the DCR-dependent Remote MCP host flow.

Reasoning:

- Hosted Tableau MCP Server does not support the DCR-based client registration flow expected by generic Remote MCP hosts such as Codex / Kiro
- a generic remote-host registration flow therefore does not complete cleanly against the Hosted Tableau MCP server
- this application should not build its v0.11.0 plan around a flow that the Hosted server does not support
- instead, the backend-controlled Connected App + Direct Trust boundary should be used

In practical terms:

- the app-authenticated user email is mapped to a Tableau Cloud user with the same email
- the backend issues the Direct Trust JWT
- the `sub` claim corresponds to the Tableau Cloud same-email user
- Tableau Cloud user permissions remain the final authorization boundary
- the application does not reimplement Tableau authorization rules on its own

## Connected App Direct Trust boundary

The Direct Trust boundary should be treated as the controlled auth layer for v0.11.0:

- backend receives an authenticated app user context
- backend derives a Direct Trust JWT for the same-email Tableau Cloud user
- the JWT is used only as the boundary for the Hosted metadata request
- Tableau Cloud permissions remain authoritative
- the app does not duplicate Tableau permission modeling
- raw JWTs, access tokens, refresh tokens, and secrets must not appear in user-facing output, traces, or logs
- token storage / refresh is out of scope for v0.11.0

The implementation issue must verify the exact token presentation expected by the Hosted MCP path. Direct Trust is the selected application auth boundary, but the hosted transport adapter must not assume that every Hosted MCP endpoint accepts the JWT in the same way as Tableau REST or embedding flows without validation.

This keeps the auth boundary compatible with no-network validation and gated hosted execution.

## Hosted MCP connection model

v0.11.0 should keep the connection model explicit:

- `fake`
- `stdio`
- `hosted`

Recommended behavior:

- default remains `fake` or the existing no-network fallback path
- `stdio` fallback remains available
- `hosted` is used only when explicitly enabled and configured
- the first hosted execution target is `describeDatasource`
- `listFields` execution is not added in v0.11.0
- raw MCP tools are not surfaced to users
- transport details remain behind the tool boundary
- hosted transport / adapter behavior should continue to normalize missing config, timeout, auth failure, permission denial, and transport errors safely

## Existing flow integration

The Hosted path should be inserted into the existing flow, not bypass it:

```text
user message
 ->
intent resolution
 ->
metadata_discovery
 ->
ambiguity / clarification / precondition checks
 ->
structured discovery plan
 ->
ToolLayer metadata execution boundary
 ->
Hosted describeDatasource when enabled and configured
 ->
normalization
 ->
safe response shaping
 ->
ResponseComposer
```

Important constraints:

- existing `metadata_discovery` flow is reused
- existing ambiguity model and clarification contracts are reused
- existing precondition checks are reused
- Hosted execution is limited to eligible datasource metadata discovery plans
- unsupported, ambiguous, or unsafe requests do not enter Hosted execution
- legacy free-form chat fallback remains available
- `selected_mark_explanation` remains unchanged
- LLM ResponseComposer is not expanded in v0.11.0

Chat flow integration means using the existing structured `metadata_discovery` path when it is already selected. It does not mean replacing `ChatService`, rewriting broad free-form chat routing, or sending broad free-form requests to Hosted MCP.

## Risks and open questions

The main risks and unknowns are:

- exact Hosted MCP endpoint shape and transport details
- whether the Direct Trust JWT is accepted by the Hosted MCP path exactly as expected
- what exact token presentation the Hosted MCP endpoint expects from this application
- email mapping mismatches between app identity and Tableau Cloud identity
- site content URL or site ID mismatches
- permission-denied behavior and whether it is surfaced as a safe fallback
- timeout and latency under real hosted conditions
- hosted adapter error normalization quality
- secrets and environment management for hosted tests
- extra complexity from switching between fake, stdio, and hosted modes
- whether the DCR-free assumption changes later
- Tableau-side Hosted MCP behavior changes
- regression risk when hosted execution is connected to the metadata discovery flow
- preserving safe user-facing output
- keeping the LLM ResponseComposer boundary stable while adding Hosted execution
