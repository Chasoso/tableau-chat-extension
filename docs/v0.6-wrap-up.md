# v0.6.0 Wrap-up

## 1. Purpose

This document wraps up the v0.6.0 AgentCore Spike and prepares the handoff to the next phase.

## 2. Release theme

v0.6.0 is not an AgentCore implementation release.

It is a release for:

- AgentRunner abstraction
- LambdaAgentRunner default runtime
- selected_mark_explanation runner path
- AgentCore fit / design / metrics / decision readiness

## 4. Main architecture outcome

```text
Explicit selected-mark action
↓
AgentRunInput
↓
LambdaAgentRunner
↓
selected_mark_explanation structured orchestration
↓
Tool Layer
↓
ResponseComposer deterministic summary
↓
AgentRunResult
↓
existing response shape
```

Key outcomes:

- `AgentRunner` abstraction was introduced
- `LambdaAgentRunner` was added
- `selected_mark_explanation` now goes through `LambdaAgentRunner`
- `AgentRunInput` / `AgentRunResult` provide comparison-ready shapes
- `AgentRunResult` can represent warnings, errors, fallback, trace, observability, and budget fields
- `AgentCoreRunner` is not implemented yet
- free-form chat remains on the legacy path

## 6. AgentCore decision summary

Decision:

- deferred partial adoption

Status:

- accepted as the v0.6 outcome

Rationale:

- `selected_mark_explanation` is deterministic and lightweight
- AgentCore is not expected to improve answer quality much for this path
- AgentCore could improve observability and session readiness later
- operational complexity is not yet justified for the current path

What is accepted now:

- `LambdaAgentRunner` remains the default runtime
- `AgentRunner` remains the comparison boundary
- `AgentCoreRunner` remains a thin optional future candidate

What is deferred:

- AgentCore implementation
- AgentCoreRunner implementation
- runtime migration

What remains open:

- whether AgentCore becomes valuable once longer-running or session-heavy paths appear
- whether AgentCore should be revisited after Tableau MCP or LLM composer work expands the runtime

Revisit conditions:

- longer-running tool paths appear
- observability or session needs increase
- API Gateway limits become a real bottleneck
- no-network stubbed AgentCore tests become useful

## 7. Comparison results summary

| Category | Current finding | Implication |
| --- | --- | --- |
| Latency | `selected_mark_explanation` is lightweight and deterministic. | AgentCore is unlikely to help latency and may add overhead. |
| Timeout risk | The current path should stay well within budget in normal cases. | AgentCore timeout value is not yet compelling. |
| Failure handling | LambdaRunner already provides safe fallback and normalization. | Current behavior is sufficient for now. |
| Trace completeness | Trace-first summary is already available in the custom stack. | AgentCore would need to prove an improvement. |
| Observability usefulness | LambdaRunner is good enough for baseline comparison, but not managed runtime observability. | AgentCore may help later if runtime visibility becomes harder to manage. |
| Implementation complexity | AgentCore increases AWS, IAM, and integration complexity. | Delay until the need is stronger. |
| Local testability | The current path stays no-network friendly and deterministic. | Preserve this property. |
| Cost | AgentCore cost is not justified for the current simple path. | Defer until value is clearer. |
| Debugging effort | LambdaRunner is easier to reproduce locally. | Keep the baseline for now. |
| selected_mark_explanation consistency | The deterministic summary path is already stable. | Do not trade away stability lightly. |

## 8. Risk summary

Remaining risks:

- AgentCore runtime behavior has not been validated in production
- comparison metrics are defined, but not automatically measured
- the selected-mark path is intentionally narrow
- free-form chat still uses the legacy flow
- Tableau MCP migration is still plan-only
- LLM ResponseComposer is still plan-only
- `AgentRunInput` / `AgentRunResult` may need adjustment if a real AgentCoreRunner appears later
- Tool Layer and AgentCore boundaries may need revisiting once more real tools exist
- trace persistence and debug UI are still missing
- cost estimates remain rough
- local testability must remain intact if AgentCoreRunner is added later

## 10. Next-phase handoff

Recommended next phase:

1. Tableau MCP Tool Layer Phase
2. LLM ResponseComposer Phase

Why:

- Tableau MCP migration is still the clearest Tool Layer next step
- LLM composer planning is useful, but should stay optional
- AgentCore should stay deferred until the runtime needs become stronger
