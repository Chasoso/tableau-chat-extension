# LLM ResponseComposer Safety Boundary

## Summary

This document defines the safety boundary for the optional v0.10.0 LLM ResponseComposer direction.

The deterministic ResponseComposer remains the baseline.
LLM composition is optional, guarded, and evidence-bounded.

## Deterministic baseline

The deterministic composer remains available in all cases.

It is the default fallback when:

- LLM composition is disabled
- configuration is missing
- the LLM is unavailable
- the LLM times out
- the LLM produces unsafe or invalid output
- the LLM output fails validation

## Optional / guarded LLM boundary

LLM composition must remain:

- optional
- feature-flagged or config-guarded
- no-network friendly for default tests
- separate from tool execution decisions
- separate from arbitrary query generation

The LLM composer may improve wording and explanation quality, but it must not widen execution scope.

## Composer input boundary

The LLM composer may consume only evidence-bounded, normalized, JSON-safe material such as:

- normalized metadata discovery summaries
- deterministic `selected_mark_explanation` summaries
- trace-safe summaries
- evidence IDs or evidence summaries
- truncation notes
- permission-aware limitation notes
- safe user request summaries
- safe response intent or type metadata

The LLM composer must not receive:

- raw MCP output
- raw MCP tool payloads
- raw transport output
- raw hosted MCP responses
- tokens
- secrets
- OAuth credentials
- stack traces
- internal exception dumps
- underlying data
- field values
- row data
- sample values
- domain values
- write payloads
- arbitrary query text intended for execution

## Composer output boundary

The LLM composer may produce only user-facing, non-executable wording such as:

- summary text
- explanation wording
- limitation wording
- clarification wording
- safe next-step wording
- evidence-bounded restatements

The LLM composer must not produce:

- tool calls
- MCP tool names or payloads
- Tableau MCP queries
- arbitrary queries
- SQL
- write actions
- hidden raw data
- row data
- field values
- sample values
- domain values
- unsupported claims
- fabricated counts
- fabricated permissions
- instructions to bypass safety

## Evidence discipline

LLM composition is a wording step, not a fact-generation step.

The composer must:

- stay within the provided evidence
- preserve truncation and limitation notes
- preserve unknown values as unknown
- avoid inventing counts, permissions, or metadata
- keep outputs traceable to the provided evidence material

## Fallback behavior

Fallback to the deterministic composer is required when:

- composition is disabled
- the config is missing
- the LLM call fails
- the LLM call times out
- output validation fails
- output contains prohibited content
- output requests tool execution
- output appears to invent unsupported facts
- output contains raw transport-like or stack-trace-like content

Fallback behavior must not expose raw errors or stack traces to the user.

## Feature flag / config boundary

The concrete implementation may use feature flags or config values such as:

- `ENABLE_LLM_RESPONSE_COMPOSER`
- `LLM_RESPONSE_COMPOSER_MODE`
- `LLM_RESPONSE_COMPOSER_TIMEOUT_MS`
- `LLM_RESPONSE_COMPOSER_MAX_INPUT_CHARS`
- `LLM_RESPONSE_COMPOSER_MAX_OUTPUT_CHARS`

This issue does not require introducing a real LLM client.

## Validation strategy

This boundary is contract-first.

Validation should focus on:

- deterministic fallback behavior
- input boundary enforcement
- output boundary enforcement
- no-network default posture
- selected_mark_explanation regression protection
- legacy free-form chat fallback protection

## Safety boundaries

- deterministic ResponseComposer remains baseline
- LLM composition is optional / guarded
- evidence-bounded response material only
- fallback to deterministic composer on failure / timeout / unsafe output
- no LLM-generated Tableau MCP query execution
- no raw MCP or raw transport output in the composer boundary
- no arbitrary query execution
- no underlying data access
- no field values or row data retrieval
- no write-capable tools
- no tokens, secrets, or stack traces in user-facing output or traces
- default tests remain no-network
- `selected_mark_explanation` remains protected
- legacy free-form chat fallback remains available
- Exploration Session remains deferred
- AgentCore remains deferred

- real LLM integration
- LLM-generated Tableau MCP query execution
- arbitrary query generation
- arbitrary query execution

- field values or row data retrieval
- write-capable tools
- Exploration Session implementation
- AgentCore implementation
- frontend redesign
- ChatService full replacement
- runLightweightAgentLoop retirement
