# Hosted Metadata Execution Trace Coverage

## 1. Purpose

This document defines the trace coverage boundary for v0.8.0 Hosted metadata execution.

Its purpose is to:

- make Hosted metadata execution observable without exposing raw payloads
- keep trace payloads summary-level and JSON-safe
- preserve `requestId`, `correlationId`, and `agentRunId` across the Hosted path
- distinguish `fake`, `stdio`, and `hosted` execution paths
- surface fallback information when Hosted execution falls back
- align trace payloads with normalized metadata result status and safe error summaries

## 2. Background

v0.7.0 established metadata normalization and the initial trace-event model.

v0.8.0 extends that model to Hosted execution for `tableau.metadata.describeDatasource`, while keeping:

- fake / no-network tests
- stdio fallback behavior
- safe Hosted error normalization
- auth and transport boundaries that do not leak tokens or secrets

Hosted execution must remain observable, but trace payloads must not expose raw MCP payloads, raw transport results, tokens, secrets, or stack traces.

## 3. Trace coverage boundary

Trace coverage in this milestone includes:

- `tableau.metadata.describeDatasource`
- Hosted transport execution
- fake / no-network fallback execution
- stdio fallback execution, if configured
- normalized Hosted errors
- normalized metadata result status

It does not include:

- UI trace panel work
- trace persistence
- raw MCP payload exposure
- broad Hosted tool exposure

## 4. Event names

The existing metadata trace event names remain the canonical set:

- `tableau_metadata_tool.started`
- `tableau_metadata_tool.completed`
- `tableau_metadata_tool.failed`

Hosted execution uses the same event names as fake and stdio execution.
The difference is in the safe payload fields.

## 5. Hosted trace payload shape

Hosted trace payloads should remain summary-level and JSON-safe.

Recommended fields:

- `toolName`
- `transportKind`
- `transportStatus`
- `requestedTransportKind`
- `selectedTransportKind`
- `fallbackUsed`
- `fallbackFrom`
- `fallbackTo`
- `status`
- `requestId`
- `correlationId`
- `agentRunId`
- `startedAt`
- `completedAt`
- `durationMs`
- `warningCount`
- `errorCode`
- `fakeNoNetwork`
- `hostedFeatureEnabled`
- `noNetworkRequested`
- `metadata`

The `metadata` field should only contain safe summary data and sanitized trace metadata.

## 6. Correlation fields

The following correlation fields must survive the Hosted path:

- `requestId`
- `correlationId`
- `agentRunId`

They should be available in:

- transport trace metadata
- normalized result trace summary
- trace events emitted by the metadata execution boundary

Missing values are allowed, but the fields must never be repurposed to carry raw payload data.

## 7. Duration and timing fields

Duration fields should be:

- numeric
- non-negative
- derived from safe timestamps or injected clocks
- present for successful Hosted execution and safe failure paths where timing is known

Recommended fields:

- `startedAt`
- `completedAt`
- `durationMs`

If a fallback path is used, duration should reflect the actual path that executed, not the attempted path that was skipped.

## 8. Error and warning fields

Hosted trace payloads should expose only safe summary fields:

- `errorCode`
- `warningCount`

The `errorCode` value must align with the normalized Hosted metadata error summary.
The `warningCount` value must align with the normalized warnings count.

Raw error objects, raw stack traces, and raw transport responses must not be attached to trace payloads.

## 9. fake / stdio / hosted distinction

Trace payloads should make it clear whether the execution path was:

- `fake`
- `stdio`
- `hosted`

Fallback cases should also be visible through:

- `fallbackUsed`
- `fallbackFrom`
- `fallbackTo`

This allows the team to distinguish:

- direct fake/no-network execution
- stdio fallback execution
- Hosted execution that succeeded
- Hosted execution that fell back safely

## 10. Fallback trace behavior

If Hosted execution is attempted and then falls back, the trace should still be safe and descriptive.

Recommended behavior:

- mark `fallbackUsed: true`
- record `fallbackFrom: "hosted"`
- record the selected fallback transport in `fallbackTo`
- keep the normalized status aligned with the actual executed path
- do not include raw transport details

If Hosted execution is not configured, trace should still report a safe `not_configured` or fallback summary rather than exposing the raw configuration problem.

## 11. Alignment with normalized result contract

Trace payloads must remain aligned with normalized metadata results.

In practice, that means:

- trace `status` should match normalized result status
- trace `errorCode` should match normalized error summary code
- trace `warningCount` should match normalized warning count
- trace `transportKind` should match the actual executed transport
- trace should not duplicate raw payload fields from the transport result

Trace is a safe observability surface, not a transport payload mirror.

## 12. Token, secret, stack trace, and raw payload suppression

Trace payloads must suppress:

- access tokens
- refresh tokens
- authorization headers
- token secrets
- raw MCP requests
- raw MCP responses
- raw transport results
- raw payloads
- stack traces
- stdout / stderr

If token references are present, they should be masked or omitted in trace output.

## 13. Relationship with HostedTableauMcpTransport

`HostedTableauMcpTransport` provides the transport-level timing and safe trace metadata.

This trace coverage boundary adds execution-level visibility by carrying that information through:

- the metadata execution boundary
- output normalization
- trace summary generation

The transport itself should continue to avoid real network behavior by default unless explicitly injected and enabled.

## 14. Relationship with describeDatasource hosted execution

`tableau.metadata.describeDatasource` is the first Hosted execution target in v0.8.0.

Its Hosted trace coverage should show:

- Hosted selection when the feature flag is on
- safe fallback when Hosted execution is off or unavailable
- normalized output status
- Hosted transport kind and timing

`tableau.metadata.listFields` remains outside the first Hosted execution path until a separate readiness decision is made.

## 15. Relationship with hosted error normalization

Hosted error normalization and trace coverage should stay in sync.

That means:

- normalized Hosted error codes should appear in trace `errorCode`
- user-facing messages should remain summary-level
- raw stack traces and raw transport errors should not be copied into trace payloads

## 16. Non-goals

This issue does not implement:

- UI trace panel
- trace persistence
- raw MCP payload exposure
- broad Tableau MCP tool exposure
- arbitrary query execution
- underlying data access
- write-capable tools
- free-form chat migration

## 17. Recommendations for #164 and #166

For `#164`:

- preserve the fake/no-network regression suite
- preserve stdio fallback trace coverage
- ensure fallback trace payloads remain JSON-safe
- keep raw payload suppression tests close to the execution boundary

For `#166`:

- summarize the final v0.8.0 Hosted execution behavior using the same trace vocabulary
- document how Hosted, stdio, and fake paths are distinguished in the final result
- include the safe trace contract as part of the milestone handoff

## 18. Open questions

- Should stdio fallback always emit its own explicit trace distinction, or should it be collapsed into the selected transport kind?
- Should fallback metadata include a reason code in addition to `fallbackUsed` / `fallbackFrom` / `fallbackTo`?
- Should trace summaries surface hosted feature-flag state in every path, or only when Hosted is selected?
- Are there any additional safe transport trace identifiers that should be standardized before v0.9.0?
