# v0.6.0 AgentCore Spike Plan

## 1. Purpose

v0.6.0 is a planning phase for evaluating whether AgentCore should become a runner option for selected structured orchestration paths.
The main goal is not to improve answer quality first. The main goal is to improve operations, observability, session handling readiness, and implementation boundaries while keeping the current selected_mark_explanation path comparable.

## 2. Original roadmap alignment

The original roadmap placed v0.6.0 in "Phase 5: AgentCore Spike".

That means:

- introduce an `AgentRunner` abstraction
- compare a current Lambda-based runner with a future AgentCore-based runner
- keep selected_mark_explanation as the first comparable path
- compare observability, latency, failure handling, and integration complexity before any broad migration

v0.6.0 is therefore a comparison and boundary-definition phase, not a migration phase.

## 3. Current state

Current runtime responsibilities are already split across several layers:

- `ChatService` and `runLightweightAgentLoop` still own the legacy free-form chat path
- v0.4 structured orchestration owns intent resolution, plan selection, execution, tracing, and placeholder response routing for selected_mark_explanation
- v0.5 Tool Layer owns tool definitions, registry lookup, preconditions, tool execution wrapping, trace events, context-derived pseudo tools, and deterministic response composition
- the explicit selected_mark_explanation action path is now routed through `LambdaAgentRunner` as the thin comparison wrapper for the structured path
- the existing `AgentRunner` abstraction in `backend/src/agent/runner.ts` is still a thin wrapper around `ChatService.generateAnswer()`

The selected_mark_explanation path is now the narrowest full structured path in the system.

## 5. Current lightweight agent loop

The legacy loop still exists for free-form chat.

Current responsibilities include:

- receiving user messages
- assembling conversation and context inputs
- deciding whether a lightweight planning loop should run
- calling Bedrock / LLM planning logic when needed
- optionally calling Tableau MCP for additional context
- retrying or falling back when the loop stalls
- normalizing errors and assembling the final chat response

This overlaps with orchestration concerns that are now also expressed in structured orchestration and the Tool Layer.

## 6. Current structured orchestration and Tool Layer

Structured orchestration and the Tool Layer now own:

- deterministic intent resolution for selected_mark_explanation
- fixed plan selection
- routing decisions via ToolRouter
- registry lookup and tool policy filtering
- precondition modeling
- tool execution wrapping
- trace-first orchestration and tool traces
- deterministic response composition

This makes the selected_mark_explanation path a strong candidate for runner abstraction because the flow is explicit and already isolated.

## 7. Responsibility overlap

The main overlap areas are:

- request routing and orchestration entry selection
- planning vs tool policy decisions
- error normalization and fallback handling
- trace and observability emission
- response assembly

v0.6.0 should not remove these concerns immediately. It should wrap them behind `AgentRunner` so Lambda and AgentCore can be compared consistently.

## 8. AgentRunner abstraction proposal

Proposed abstraction:

```ts
interface AgentRunner {
  run(input: AgentRunInput): Promise<AgentRunResult>;
}
```

The abstraction should let the runtime switch between:

- `LambdaAgentRunner`
- `AgentCoreRunner`

Proposed `AgentRunInput` shape should be small but expressive:

- `agentRunId`
- `runMode`
- `actionId`
- `requestedIntent`
- `userMessage`
- `contextSummary`
- `contextReference`
- `planId`
- `toolPolicy`
- `modelPolicy`
- `budget`
- `traceOptions`
- `locale`
- `metadata`

Proposed `AgentRunResult` should carry:

- `status`
- `runMode`
- intent result
- plan selection
- execution result
- response composer result
- final message
- trace events
- budget usage
- warnings
- errors
- fallback reason
- runner metadata
- observability summary

## 9. LambdaAgentRunner candidate scope

LambdaAgentRunner should wrap the existing backend runtime without broad behavioral change.

Candidate responsibilities:

- run the existing selected_mark_explanation structured orchestration path
- keep the wrapper thin so the selected_mark_explanation orchestration result can be compared against a future AgentCoreRunner result
- call the current IntentResolver, plan selection, ExecutionEngine, Tool Layer, and ResponseComposer skeleton
- preserve agentRunId and trace output
- return a normalized result suitable for API responses and comparison logging
- surface timeout, fallback, and budget info consistently

LambdaAgentRunner should not own:

- ToolRegistry implementation details
- ToolExecutionWrapper internals
- Tableau MCP-specific execution semantics
- frontend-specific behavior
- future AgentCore runtime behavior

## 10. AgentCoreRunner candidate scope

AgentCoreRunner is not implemented in v0.6.0 planning, but its scope should be defined so it can be compared later.

Candidate responsibilities:

- pass `AgentRunInput` into an AgentCore runtime
- bind agentRunId to AgentCore session / trace / observability concepts
- run selected_mark_explanation as the first comparable path
- normalize AgentCore output back into `AgentRunResult`
- compare timeout, failure, and observability behavior with LambdaAgentRunner

Open design decisions:

- how much custom orchestration should remain outside AgentCore
- whether AgentCore should own planning and tool execution or only runner orchestration
- how AgentCore observability should coexist with current trace-first architecture
- how to handle Lambda/API Gateway timeout constraints during comparison

## 11. Why selected_mark_explanation is the first comparison target

selected_mark_explanation is the best first comparison target because:

- it is explicit user action, not free-form chat
- it already has a fixed plan
- it is mostly context-derived and deterministic
- it has a clear required/optional context split
- it has fewer external tool dependencies than broader chat paths
- it is easy to trace and compare across runtimes
- it is a good fit for measuring observability and fallback behavior

## 12. AgentCore adoption purpose

AgentCore should be evaluated primarily for operational benefits:

- better observability
- clearer execution trace visibility
- easier inspection of intermediate outputs
- improved failure debugging
- session and memory readiness
- managed runtime benefits

It should not be adopted simply to make the answer smarter.

## 13. AgentCore comparison criteria

Compare LambdaAgentRunner and AgentCoreRunner on:

- latency
- timeout rate
- failure rate
- trace completeness
- observability usefulness
- implementation complexity
- local testability
- cost estimate
- debugging effort
- selected_mark_explanation output consistency

Optional comparison metrics:

- session continuity
- memory readiness
- tool execution visibility
- deploy complexity
- fallback behavior
- CloudWatch / AgentCore observability correlation

The current fit assessment for `selected_mark_explanation` is documented separately in `docs/v0.6-agentcore-fit-selected-mark-explanation.md`. The conclusion is intentionally conditional: AgentCore looks more valuable for observability and runtime management than for improving this deterministic path's answer quality.

## 14. Non-AgentCore path

If AgentCore is not adopted immediately, the safe path is:

- introduce `AgentRunner` only
- make LambdaAgentRunner the default runner
- keep free-form chat on the legacy path
- keep selected_mark_explanation on LambdaAgentRunner first
- continue tool-layer improvements independently
- revisit AgentCore later with the same comparison harness

## 15. Tableau MCP tool execution strategy

Tableau MCP migration should stay separate from AgentCore evaluation.

Guidance:

- first identify the smallest Tableau tool units that can become ToolDefinitions
- keep selected_mark_explanation mostly context-derived for now
- evaluate metadata / summary / query style tools separately
- do not generate and execute arbitrary queries without policy boundaries
- keep datasource identifier, workbook, view, and permission preconditions explicit
- decide whether Tableau MCP should be migrated before or after AgentCore comparison only after the runner boundary is clear

## 16. LLM ResponseComposer strategy

The current ResponseComposer is deterministic and placeholder-only.

Future comparison questions:

- should deterministic and LLM composers be separate strategies
- should selected_mark_explanation even need an LLM composer early
- how should responseMaterial be converted into prompts without leaking raw data
- how should hallucination and overclaiming be prevented
- how should citations and evidence be represented
- can both runners share the same composer contract

## 17. Trace and observability strategy

The current architecture is already trace-first.

AgentCore comparison should preserve:

- agentRunId correlation
- intent / plan / tool / execution trace continuity
- trace payloads that stay summary-only
- observable fallback reasons
- execution duration and budget usage

Comparisons should be possible without introducing raw context payloads into traces.

## 18. Timeout / budget / cost considerations

Evaluation should include:

- Lambda/API Gateway timeout alignment
- tool-call and model-call budget visibility
- failure mode when a runner times out
- cost differences between managed runtime and current backend runtime

Budget and timeout should stay part of the runner contract, not hidden inside tool execution details.

## 19. Minimal v0.6.0 scope

The smallest useful v0.6.0 scope is:

- define `AgentRunner`
- define `AgentRunInput` and `AgentRunResult`
- document LambdaAgentRunner as the current default wrapper
- document AgentCoreRunner as the future comparison target
- keep selected_mark_explanation as the first comparable path
- define the comparison metrics and handoff boundary
- widen the runner contract only enough to compare selected_mark_explanation across runtimes without changing ChatService or runLightweightAgentLoop yet

## 20. Out of scope

- AgentCore implementation
- AgentCoreRunner implementation
- replacing ChatService
- replacing runLightweightAgentLoop
- free-form chat migration
- broad Tableau MCP migration
- LLM-based ResponseComposer
- frontend UI changes
- backend API changes
- tests implementation
- auth / infrastructure changes

## 21. Risks and constraints

- AgentRunner can become too thin if it only renames the existing path without clarifying boundaries
- AgentCore can overlap with current orchestration if the runner boundary is not respected
- Trace and observability can become inconsistent if comparison metrics are not standardized
- free-form chat and selected_mark_explanation must remain separately understood
- Tableau MCP and LLM migration should not be hidden inside the runner abstraction
