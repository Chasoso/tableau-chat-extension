# Guarded LLM ResponseComposer Prototype

## Overview

Issue #212 adds a minimal guarded prototype for the LLM ResponseComposer branch selected in v0.10.0.

The prototype keeps the deterministic ResponseComposer as the baseline and makes LLM composition optional, gated, and safe-by-default.

## Deterministic fallback

The deterministic composer remains the default path.

- If LLM composition is disabled, the deterministic composer is used.
- If LLM composition is not configured, the deterministic composer is used.
- If LLM composition fails, times out, or produces unsafe output, the deterministic composer is used.
- No default test requires a hosted or real LLM dependency.

## Feature flag and config gate

The prototype is guarded behind config-style gating.

Expected controls include:

- enabled / disabled mode
- timeout limit
- input size limit
- output size limit

The default posture is disabled or otherwise non-active unless explicitly enabled in a safe test or integration context.

## Fake composer behavior

The prototype includes fake no-network composer behavior for tests.

Supported fake behaviors include:

- success
- timeout
- error
- unsafe output
- invalid output

Fake behavior must remain deterministic and must not require secrets, network access, or hosted dependencies.

## Response material usage

The prototype only consumes the safe response material defined in `docs/v0.10-llm-response-material-contract.md`.

It must not receive:

- raw MCP output
- raw transport output
- raw Tableau responses
- tokens or secrets
- stack traces
- underlying data
- field values or row data
- arbitrary query text
- write payloads

## Output validation

Composer output must be validated before it is surfaced to users.

At minimum, validation should reject:

- invalid shapes
- non-string outputs
- outputs that exceed length limits
- outputs that omit required limitation notes
- outputs that request or imply tool execution
- outputs that contain unsafe or raw data-like content

Validation failures must fall back to the deterministic composer.

## Fallback reasons

Safe fallback reasons should be preserved for tracing and debugging.

Examples include:

- `composer_disabled`
- `composer_not_configured`
- `composer_timeout`
- `composer_error`
- `composer_invalid_output`
- `composer_unsafe_output`
- `composer_missing_required_limitations`
- `composer_output_too_long`

Fallback reasons must not expose raw exception details or stack traces.

## Selected mark protection

`selected_mark_explanation` remains protected.

The prototype does not change the deterministic selected-mark path or force it through LLM composition.

## Legacy fallback protection

Legacy free-form chat fallback remains available.

This prototype does not replace `ChatService`, retire `runLightweightAgentLoop`, or rewrite broad chat routing.

## Relation to later issues

- #213 should add fallback and regression tests around the guarded prototype.
- #214 should document the prototype result after implementation is complete.

## Out of scope

This issue does not add:

- real LLM integration
- arbitrary query generation or execution
- raw MCP exposure
- underlying data access
- field values or row data retrieval
- write-capable tools
- Exploration Session
- AgentCore
- frontend redesign

## Safety boundaries

The prototype preserves the v0.9.0 and v0.10.0 safety posture:

- no raw MCP tool exposure
- no raw MCP output exposure
- no raw transport output exposure
- no arbitrary query execution
- no underlying data access
- no field values / row data retrieval
- no write-capable tools
- no tokens / secrets / stack traces in user-facing output or traces
- default tests remain no-network
- hosted or real LLM integration remains optional or gated
